# 营小助配置中心：设计思想、架构现状与 AI Agent 推进规划

> 更新时间：2026-03-13  
> 适用范围：`config-center-admin-web` 当前仓库（前端原型）  
> 基线来源：`prd-analysis.md`、`plan.md`、`sentinel-prd.md`、`job-prd.md`、`decision-ledger.md`、`architecture.md` 以及 `src` 代码实现

## 1. 项目设计思想（Design Philosophy）

### 1.1 核心定位

配置中心不是单点“提示工具”，而是“提示 + 作业 + 平台控制”一体化平台，目标是把业务知识、规则约束、执行动作和平台控制能力收敛到同一平台中。

### 1.2 设计原则

1. 业务角色优先：主用户和主要发布/风险责任承担者是业务角色，技术角色以平台支撑为主。  
2. 平台控制先于扩展：P0 优先发布前强校验、版本替换、审计留痕，再扩展复杂能力。  
3. 对象解耦协同：规则负责“何时触发”，作业负责“触发后做什么”，分开建模、分开发布控制。  
4. 公共底座复用：页面资源、API注册、预处理器作为统一能力底座，避免私有配置扩散。  
5. 异常可降级：运行异常不阻断业务主流程，必须可回退手工路径。  
6. 决策可追溯：产品决策统一落 `decision-ledger.md`，计划与实现均需引用已确认口径。

### 1.3 平台边界

1. 不替代业务系统原生审批流。  
2. 不以内置全链路联调作为 P0 承诺。  
3. 不依赖运行时业务冲突仲裁作为主防线，冲突应在发布前消解。  
4. 不追求高自由度脚本平台化，脚本仅作为受控增强能力。

## 2. 当前架构现状（As-Is）

### 2.1 目标架构（文档层）

基于 `architecture.md`，目标形态是三端分离与三平面解耦：

1. 三端：Admin Web、Backend、JSSDK。  
2. 三平面：控制面、运行面、管理与审计面。  
3. 三能力层：公共支撑层、智能提示层、智能作业层。

### 2.2 当前代码架构（仓库层）

当前仓库是 **Admin Web 原型前端**，以 Mock 服务模拟控制面+管理与审计面核心行为。

1. 应用壳与路由：`src/App.tsx`、`src/components/AppShell.tsx`。  
2. 页面层：`src/pages/*`（总览、页面资源、API注册、预处理器、智能提示、智能作业、发布、审计指标、角色管理）。  
3. 领域模型层：`src/types.ts`（统一对象类型与状态、执行模型、日志模型）。  
4. 服务层：  
`src/services/configCenterService.ts`（CRUD、状态流转、发布校验、状态变更日志、指标聚合）  
`src/services/workflowService.ts`（规则条件求值、作业节点执行模拟、执行日志）  
5. 数据种子层：`src/mock/seeds.ts`（页面/API/规则/作业/发布/日志样例数据）。  
6. 校验与工具：`src/validation/formRules.ts`、`src/utils.ts`。

### 2.3 关键对象与链路映射

1. 公共支撑对象：`PageResource`、`PageElement`、`InterfaceDefinition`、`PreprocessorDefinition`。  
2. 提示对象：`RuleDefinition`、`RuleConditionGroup`、`RuleCondition`。  
3. 作业对象：`JobSceneDefinition`、`JobNodeDefinition`、`JobExecutionSummary`。  
4. 发布对象：`PublishPendingItem`、`ValidationReport`、`PublishAuditLog`。  
5. 观测对象：`TriggerLogItem`、`ExecutionLogItem`、`FailureReasonMetric`。

端到端主链路已在前端原型中打通：

1. 配置对象 -> 进入待处理。  
2. 发布页发起校验/发布/延期/回滚。  
3. 规则命中与作业执行写入日志。  
4. 审计指标页聚合查看。

### 2.4 当前成熟度判断

已具备：

1. 平台信息架构和对象模型主干完整。  
2. API注册、条件编辑、场景编排、发布与审计主界面可用。  
3. 以决策台账驱动的文档体系已形成（总纲+子文档+计划+台账）。  
4. 已建立跨模块防漏机制（`docs/dev-checklist.md` + PR 模板联动检查）。

待补齐（下一阶段重点）：

1. 后端真实服务与 JSSDK 运行面对接（当前以 Mock 为主）。  
2. 自动化测试与回归门禁深度不足（当前以人工验证为主）。  
3. 文档到实现的“自动一致性检查”仍可进一步产品化。

## 3. 面向 AI Agent 的目标定位（To-Be）

AI Agent 的目标不是替代产品/研发决策，而是作为“计划-实现-校验-留痕”的加速器与守门员：

1. 提高跨文档一致性。  
2. 降低跨模块漏改风险。  
3. 缩短从需求到可验证实现的周期。  
4. 强化变更可追溯性与可审计性。

## 4. AI Agent 体系设计（建议）

### 4.1 Agent 分工

1. `Product-Consistency Agent`：对齐 PRD、子文档、台账与计划，输出变更影响面。  
2. `Schema-Guard Agent`：检查 `src/types.ts`、服务 payload、页面表单字段的一致性。  
3. `Implementation Agent`：按已确认决策实现页面与服务联动代码。  
4. `Validation Agent`：自动执行 build、关键检索、冒烟用例并生成验收结论。  
5. `Release-Checklist Agent`：按 `docs/dev-checklist.md` 与 PR 模板输出发布前检查报告。

### 4.2 标准输入与输出

每次需求迭代统一输入：

1. 需求描述（或会话结论）。  
2. 决策来源（`decision-ledger.md` 条目）。  
3. 目标文档和目标模块范围。

每次迭代统一输出：

1. 文档变更（总纲/子文档/计划或说明无需变更）。  
2. 代码变更（页面、类型、服务、mock）。  
3. 校验报告（`rg` 检索结果、`npm run build`、关键手工场景结论）。  
4. 风险与回滚说明。

### 4.3 联动约束（强制）

当新增或修改“来源/取值/映射”能力时，Agent 必须执行并记录以下五项：

1. 对象模型是否补齐。  
2. 配置 UI 是否全模块覆盖。  
3. 保存结构是否可回显。  
4. 运行解析是否可识别并有异常阻断。  
5. 验收用例是否覆盖可配置、可保存、可回显、可执行、异常阻断。

## 5. 推进路线图（AI Agent 视角）

### Phase A：规范化接入（先守门）

目标：让 Agent 先稳定做“文档对齐 + 联动检查 + 构建校验”。

交付：

1. 固化 Agent 执行模板（输入/输出/检查项）。  
2. 每次改动自动生成“联动影响面清单”。  
3. 变更后自动运行 `npm run build` 和关键字段 `rg` 检索。

### Phase B：半自动实现（再提效）

目标：在可控范围内让 Agent 生成和修改代码。

交付：

1. 支持页面表单字段与 `types.ts` 同步更新。  
2. 支持服务层 payload 结构和 mock 解析同步更新。  
3. 自动生成最小验收用例清单。

### Phase C：端到端闭环（可运营）

目标：形成“需求 -> 决策 -> 实现 -> 验收 -> 留痕”的标准流水线。

交付：

1. 需求变更自动映射到受影响文档和模块。  
2. 生成发布前风险报告与回滚建议。  
3. 形成可查询的 AI 变更审计记录。

## 6. 建议的近期任务清单（可直接启动）

1. 在仓库新增 AI Agent 执行规范文档（建议：`docs/ai-agent-workflow.md`）。  
2. 将 `docs/dev-checklist.md` 映射为可执行的命令脚本或 CI 任务。  
3. 增加“类型-页面-服务”一致性检查脚本（优先覆盖 API注册、智能提示条件编辑、智能作业节点配置）。  
4. 建立最小自动化回归：`build + 核心链路 smoke`。  
5. 在 PR 模板中增加“Agent 运行摘要”固定区块。  
6. 把 `decision-ledger.md` 的已确认项和代码模块建立映射索引，减少遗漏。

## 7. 风险与控制

1. 风险：Agent 只改 UI 不改模型。  
控制：强制执行五项联动检查。  
2. 风险：文档与实现口径漂移。  
控制：所有改动必须绑定决策台账条目。  
3. 风险：自动化修改引入隐性回归。  
控制：构建门禁 + 核心用例冒烟 + 人工抽检。  
4. 风险：脚本能力被过度放开。  
控制：脚本类改动必须经过受控发布与审计路径。

## 8. 结论

当前项目已具备较清晰的产品总纲、模块分层与原型代码骨架。下一步引入 AI Agent 的最优策略是“先守门、再提效、后闭环”：先把一致性和联动检查做硬，再逐步扩大到代码自动实现和发布控制辅助，确保效率提升不以可控性为代价。
