# 营小助配置中心 角色与权限模型专题

> 关联文档：`spec/config-center-product/prd-analysis.md`、`spec/config-center-product/spec.md`  
> 文档目标：沉淀角色模型、权限边界、授权方式、角色管理模块设计，并形成最终确认的角色权限矩阵  
> 更新日期：2026-03-12

## 1. 文档定位

本文是营小助配置中心的权限专题文档，用于承接主产品分析文档中的角色与权限相关内容。

主产品文档解决“平台是什么、解决什么问题、如何形成业务闭环”；本文解决“谁能做什么、权限如何分配、角色如何治理”。

## 2. 权限设计原则

角色与权限模型遵循以下原则：

1. 平台主用户与主管理者均为业务角色。
2. 技术角色只承担少量运维配置、平台支持和异常排查工作。
3. 权限模型采用“组织范围控制资源边界，操作类型控制权限边界”的简化思路。
4. 不以单条规则、单个页面、单个作业场景的细粒度授权作为常规模式。
5. 权限变更、人员绑定、角色调整都必须留痕，可审计、可追溯。

## 3. 标准角色体系

当前确认的标准角色包括：

1. 业务操作角色
2. 业务配置角色
3. 业务管理角色
4. 业务审计角色
5. 业务超管角色
6. 平台支持角色

这些角色按职责抽象定义，不按具体岗位零散拆分。除平台支持角色外，其余核心角色均属于业务侧角色。

## 4. 角色职责定义

### 4.1 业务操作角色

负责在业务页面中接收提示、使用智能作业、完成办理或审核过程中的实际操作，不承担配置和治理职责。

### 4.2 业务配置角色

负责页面、规则、作业场景的创建、编辑、校验和联调，是主要配置者，但不默认拥有最终发布权。

### 4.3 业务管理角色

负责发布、停用、延期、回滚、生效范围控制以及自动执行类场景的业务风险确认，是主要治理责任承担者。

### 4.4 业务审计角色

负责查看配置版本、状态变更、触发日志、执行日志和审计记录，不参与配置修改和发布动作。

### 4.5 业务超管角色

由业务部门管理者担任，负责本组织范围内的角色管理、人员分配、权限授权和高风险治理动作。业务超管默认只管理本部门及授权范围内资源，不天然拥有跨部门管理权。

### 4.6 平台支持角色

负责平台级运维、基础能力维护、异常排查和系统支持，不参与日常业务发布决策和业务风险承担。

## 5. 操作类型清单

建议的平台标准操作类型如下：

1. 查看
2. 配置
3. 校验
4. 发布
5. 停用
6. 延期
7. 回滚
8. 审计查看
9. 风险确认
10. 角色授权管理

其中，角色授权管理用于管理角色、分配人员、配置操作权限和组织范围。

## 6. 资源授权模型

### 6.1 组织范围

资源边界主要按组织范围控制，包括业务部门、分支机构或授权组织范围。

页面、规则、作业场景默认继承所属组织范围。

### 6.2 操作类型

权限边界主要按操作类型控制。

同一角色在其组织范围内拥有哪些操作能力，由业务部门配置决定。

### 6.3 特殊跨组织场景

特殊跨组织使用场景优先通过共享或复制机制解决，而不是通过复杂 ACL 做日常维持。

## 7. 角色管理模块

角色管理模块是正式产品能力，由业务侧主导使用和维护。

应至少支持以下能力：

1. 角色与人员绑定管理
2. 角色组织范围配置
3. 角色操作类型授权
4. 角色复制
5. 角色停用
6. 角色恢复
7. 人员批量分配
8. 授权变更审计

### 7.1 角色复制

1. 支持复制既有角色配置，用于快速复用。
2. 复制后形成独立角色，不影响原角色。
3. 复制行为本身需要记录审计日志。

### 7.2 角色停用与恢复

1. 角色支持停用和恢复。
2. 已使用过的角色不做物理删除。
3. 停用后不允许新增人员绑定。
4. 已绑定人员同步失去该角色权限。
5. 停用和恢复动作必须留痕。

### 7.3 人员批量分配

1. 支持批量分配人员到角色。
2. 支持按组织范围做批量授权。
3. 批量分配必须生成审计记录。

## 8. 自动执行风险确认权限

自动执行类场景的业务风险确认，由业务管理角色在发布前完成。

需要明确以下口径：

1. 只有自动执行类场景需要强制风险确认。
2. 风险确认记录至少包含确认人、确认时间、关联场景、执行方式和适用组织范围。
3. 执行方式从非自动改为自动时，必须重新确认。
4. 修改执行链路、写入字段、触发方式等关键执行内容时，必须重新确认。

## 9. 当前已确认权限口径

以下内容已在产品层面确认：

1. 平台增加角色管理模块，由业务侧管理角色、人员绑定和权限分配。
2. 权限模型采用“组织范围控制资源边界，操作类型控制权限边界”的简化思路。
3. 自动执行类场景的业务风险确认由业务管理角色在发布前完成。
4. 角色支持复制、停用、恢复和批量分配。
5. 已使用角色不做物理删除。
6. 业务管理角色不默认拥有配置权限。
7. 平台支持角色拥有查看、校验和审计查看权限，用于技术支持场景。

## 10. 权限矩阵最终稿

当前角色与操作类型映射矩阵已完成确认，可作为权限实现和评审的统一依据。

### 10.1 角色与操作类型映射矩阵（最终确认稿）

以下矩阵为当前最终确认稿。

| 角色 | 查看 | 配置 | 校验 | 发布 | 停用 | 延期 | 回滚 | 审计查看 | 风险确认 | 角色授权管理 |
|------|------|------|------|------|------|------|------|------|------|------|
| 业务操作角色 | 是 | 否 | 否 | 否 | 否 | 否 | 否 | 否 | 否 | 否 |
| 业务配置角色 | 是 | 是 | 是 | 否 | 否 | 否 | 否 | 否 | 否 | 否 |
| 业务管理角色 | 是 | 否 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 否 |
| 业务审计角色 | 是 | 否 | 否 | 否 | 否 | 否 | 否 | 是 | 否 | 否 |
| 平台支持角色 | 是 | 否 | 是 | 否 | 否 | 否 | 否 | 是 | 否 | 否 |
| 业务超管角色 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 是 |

### 10.2 推荐解释

1. 业务操作角色默认只负责使用能力，不承担配置和治理动作。
2. 业务配置角色默认承担“建和改”的职责，但不承担正式发布责任。
3. 业务管理角色默认承担治理和风险责任，不默认拥有配置权限。
4. 业务审计角色默认只读，重点关注审计查看能力。
5. 平台支持角色拥有查看、校验和审计查看权限，用于技术支持、排障和基础能力维护场景，但不参与业务配置、业务发布和业务风险承担。
6. 业务超管角色拥有本组织范围内的完整业务管理能力，并承担角色授权管理职责。
