# 营小助配置中心 PRD 深度分析文档

> 来源文档：`spec/config-center-product/prd-analysis.md`、`spec/config-center-product/spec.md`  
> 分析视角：产品完整性、业务闭环、角色价值、治理可控性  
> 更新日期：2026-03-12  
> 说明：本文从完整产品视角重组需求，强调营小助配置中心作为“提示 + 作业 + 治理”一体化平台的整体形态；`spec.md` 保持原状，不在本文中直接改写其实现边界。角色与权限专题内容另见 `spec/config-center-product/permission-model.md`。  

## 1. 产品结论

营小助配置中心的完整产品形态，不应被理解为一个单纯的提示配置后台，也不应被拆解为彼此割裂的“提示系统”和“作业系统”。从产品完整性看，它本质上是一套面向银行内网页面的智能辅助平台，核心目标是把业务知识、风险约束、操作动作和治理能力统一沉淀到同一个配置中心中。

完整产品应同时解决四类问题：

1. 监管与业务口径变化快，但正式系统改造慢，现场缺少快速触达能力。
2. 一线办理和审核场景中存在大量重复查数、重复录入、重复判断动作。
3. 提示、动作、页面资源、发布治理分散在不同系统时，维护成本高且追溯困难。
4. 规则上线后容易堆积，缺少停用、回滚、审计和效果验证机制。

因此，营小助配置中心的合理产品定义应是：

> 一套面向银行内部业务页面的智能辅助与治理平台。  
> 通过统一配置中心管理页面资源、提示规则、作业场景、接口能力、发布过程和审计追踪，在业务办理现场提供实时提示、自动辅助、可控执行和全链路治理能力。

## 2. 产品定位

### 2.1 产品一句话定义

营小助配置中心是银行内部业务页面的智能辅助平台，通过页面识别、规则判断、提示渲染、作业执行和审计治理，帮助业务人员在办理现场更准确、更高效、更可控地完成业务动作。

### 2.2 产品本质

从产品本质看，这个平台不是“弹窗工具”，而是“页面级业务辅助基础设施”。  
它有三个不可分割的组成部分：

1. **提示能力**：告诉用户当前应该关注什么、注意什么、先做什么。
2. **作业能力**：帮助用户减少查数、切换、复制粘贴和重复录入。
3. **治理能力**：确保规则和动作可发布、可停用、可回滚、可审计、可复盘。

如果只强调其中一个部分，产品都会失衡：

1. 只有提示，没有作业，价值偏“知道”，缺少“做到”。
2. 只有作业，没有提示，价值偏“自动化”，缺少业务语义和风险约束。
3. 只有执行，没有治理，平台会在规模化后失控。

同时，平台的使用者定位也必须明确：**主用户是业务人员，主治理者也是业务管理人员**。  
技术角色不应被设计成平台的日常主操作者，而应主要承担少量运维配置、平台能力维护、异常排查和基础保障工作。  
这意味着整个平台的术语、交互、配置路径和权限设计，都必须默认站在业务角色视角，而不是技术视角。

### 2.3 产品边界

营小助配置中心需要保持清晰边界：

1. 不替代业务系统原生审批流和最终确认动作。
2. 不承担银行核心系统后端改造职责。
3. 不以自动化覆盖所有业务判断，复杂场景保留人工决策权。
4. 不以脚本自由度为目标，而以业务可理解、平台可治理为目标。
5. 出现异常时必须支持降级到人工路径，不得阻断业务继续办理。

## 3. 业务问题与机会

### 3.1 当前业务痛点

从银行内网页面场景出发，当前存在的核心痛点包括：

1. 监管临时要求往往通过邮件、通知、会议或线下传达，难以在操作现场即时落地。
2. 风险提示往往停留在制度和培训中，用户在具体页面、具体字段、具体步骤上缺少上下文提醒。
3. 新员工对操作顺序、注意事项、字段来源不熟悉，容易出现漏项和误填。
4. 一线人员频繁在多个系统之间切换，查数、复制、录入流程冗长且易出错。
5. 页面结构变化后，页面元素配置和提示规则容易失效，但问题定位困难。
6. 分行、总行、审计等角色对规则的创建、启停、发布、复盘缺少统一治理入口。

### 3.2 平台的机会点

营小助配置中心的机会不只是“把提醒弹出来”，而是利用页面上下文把业务规则变成现场可执行能力：

1. 在正确的页面、正确的时机、针对正确的人触发正确的提示。
2. 在提示出现后，顺势把所需数据查询、字段填充、预览确认串成动作闭环。
3. 将页面资源、规则逻辑、接口能力和日志审计统一建模，避免重复建设。
4. 让业务人员成为主要配置者、让业务管理人员成为主要治理者，降低对技术人员持续参与的依赖。

## 4. 目标用户与角色价值

| 角色 | 主要目标 | 关注点 | 平台提供的核心价值 |
|------|------|------|------|
| 一线运营人员 | 顺利完成业务办理 | 准确、快捷、少出错 | 提示及时、自动查数、减少录入 |
| 审核人员 | 提升审核效率与准确性 | 风险可见、数据可比 | 风险提示、字段获取、复核辅助 |
| 业务配置人员 | 快速配置并调整规则 | 易学、易配、少依赖研发 | 可视化配置、模板复用、校验提示 |
| 页面配置维护人员 | 维护页面资源稳定可用 | 页面识别准确、元素统一管理 | 页面建档、元素映射、版本管理 |
| 分行管理人员 | 本地化治理与执行落地 | 可启停、可延期、可审计 | 生效范围、状态治理、日志查看 |
| 总行管理人员 | 统一下发关键规则与能力 | 全行一致、可追踪 | 强控规则、统一配置、全局视图 |
| 审计/合规人员 | 追溯谁做了什么 | 证据完整、口径清晰 | 状态变更记录、触发日志、执行日志 |
| 技术/运维支持人员 | 保障平台稳定运行 | 可用性、异常排查、少量基础配置 | 平台级运维、排障支持、通用能力维护 |

产品设计必须同时照顾三类体验：

1. 使用者体验：提示清楚、动作可靠、不要打扰过度。
2. 配置者体验：流程顺、术语清晰、出错可定位。
3. 管理者体验：能看见、能收口、能追责、能复盘。

这里的配置者和管理者，默认都应理解为业务角色。  
技术角色不是产品信息架构的默认参照对象，不应要求业务配置流程建立在技术人员深度参与的前提上。

## 5. 核心产品场景

### 5.1 监管临时要求承接

当监管或上级单位下发临时性要求，而正式系统尚未完成改造时，平台需要在目标页面和关键节点展示规则化提示，并视场景需要联动后续作业动作，帮助一线人员快速执行临时口径。

### 5.2 事中风险控制

当办理过程命中风险条件时，平台不仅要展示风险原因和处理建议，还应支持根据风险等级决定是否只提醒、是否触发辅助查数、是否要求用户预览确认后再继续。

### 5.3 新员工作业指导

平台需要将静态 SOP 转化为页面上下文中的动态指导，在关键字段、关键步骤、关键节点向新员工提供结构化指引，并在必要时通过作业能力直接辅助完成部分动作。

### 5.4 高重复录入与审核辅助

对于需要跨系统查数、复制字段、重复录入的业务页面，平台应支持通过配置化作业能力完成数据获取、映射、预览和注入，减少低价值重复劳动。

### 5.5 异常中断与手工兜底

当接口失败、页面结构变化、元素缺失或权限异常时，系统必须给出可理解的失败信息，并允许用户回退到手工路径，不因平台异常阻断业务。

## 6. 完整产品能力模型

完整产品应被设计为三层能力协同：

| 能力层 | 核心内容 | 作用 |
|------|------|------|
| 公共支撑层 | 页面资源、接口定义、预处理器、资源共享、权限、版本、审计 | 提供统一基础设施 |
| 智能提示层 | 页面识别、规则匹配、提示渲染、触发控制、多提示治理 | 提供业务语义判断与提醒 |
| 智能作业层 | 场景编排、节点执行、字段映射、预览确认、页面注入、失败降级 | 提供动作辅助与效率提升 |

这三层不是松散组合，而是强依赖关系：

1. 公共支撑层提供唯一页面资源和通用能力底座。
2. 智能提示层负责把页面上下文转成“是否应该提醒/触发”的判断结果。
3. 智能作业层负责把判断结果继续转成可执行动作。

## 7. 公共支撑层分析

### 7.1 页面资源管理

页面资源管理是整个平台的起点。没有稳定的页面资源模型，提示和作业都会不稳定。

页面资源管理至少应包括：

1. 监控网站配置。
2. 监控网页配置。
3. 页面编码与 URL 识别规则。
4. 本页元素配置。
5. 元素逻辑名与选择器映射。
6. 页面、元素、识别规则的版本管理。

产品上的关键要求有三点：

1. 页面元素应“一处维护，多处复用”，避免规则和作业分别维护选择器。
2. 页面识别应优先使用更稳定的业务标识，再用 URL 做保底。
3. 页面配置流程要尽量贴近业务系统的组织方式，例如专区、菜单、页面的逐级定位。

### 7.2 接口定义中心

接口定义中心不是附属模块，而是提示和作业都依赖的重要基础能力。

它至少要承担：

1. 定义接口入参、出参和取值路径。
2. 支持页面取值与固定值作为参数来源。
3. 统一超时、重试、敏感信息保护与后端代调。
4. 为规则判断和作业场景提供复用能力。

从产品角度看，接口中心的价值在于“统一能力治理”，否则每条规则、每个场景都可能私有化定义接口，造成不可控扩张。

### 7.3 预处理器中心

预处理器中心承载的是“数据进入判断或执行前的加工能力”。  
它既服务规则判断，也服务作业执行。

典型用途包括：

1. 字段值格式清洗。
2. 日期、金额、证件号等标准化处理。
3. 接口结果转换。
4. 页面值和接口值的统一比对口径。

产品上应优先提供常见内置预处理器，自定义脚本能力作为增强项而不是默认路径。

### 7.4 资源共享与权限治理（原则级）

平台面向的是多部门、多分行、多角色协同，因此资源共享和权限治理必须是基础能力。

需要重点解决：

1. 哪些页面、规则、接口、预处理器对哪些部门可见。
2. 哪些角色可以查看、编辑、发布、停用、回滚。
3. 跨部门资源是只读复用，还是允许复制后本地维护。
4. 被引用资源如何防误删、防失效。

在权限模型上，建议采用简化授权思路：

1. 角色按职责统一定义，不按具体岗位零散拆分。
2. 资源边界主要按组织范围控制。
3. 权限边界主要按操作类型控制。
4. 页面、规则、作业场景默认继承所属组织范围。
5. 不将单页面、单规则、单场景逐条细粒度授权作为常规模式。
6. 特殊跨组织使用场景优先通过共享或复制机制解决，而不是通过复杂 ACL 维持。

同时，平台应明确建设“角色管理模块”，并将其纳入业务治理体系。角色定义、授权边界、角色管理能力和待确认矩阵，建议独立沉淀到权限专题文档中。`spec/config-center-product/permission-model.md`。`prd-analysis.md` 仅保留以下原则级结论：

1. 角色管理模块由业务侧主导使用和维护。
2. 业务部门可在平台预置角色体系基础上管理角色与人员绑定关系。
3. 业务部门可配置角色的组织范围和可执行操作类型。
4. 角色管理、人员分配、权限调整都必须留痕，可审计、可追溯。
5. 平台支持角色只提供系统级支持，不主导日常角色治理。
6. 角色支持复制，用于快速复用既有角色配置；复制后形成独立角色，不影响原角色。
7. 角色支持停用和恢复，不支持直接删除已使用过的角色。
8. 停用角色后，不允许再新增人员绑定，已绑定人员同步失去该角色权限。
9. 人员绑定支持批量分配，用于业务部门快速完成角色授权。

## 8. 智能提示分析

### 8.1 智能提示的产品职责

智能提示层的职责不只是把文案展示出来，而是完成从“页面状态”到“业务提醒”的转换。

它应包括：

1. 页面识别。
2. 元素取值。
3. 条件判断。
4. 提示内容生成。
5. 提示展示控制。
6. 触发日志留痕。

### 8.2A 规则条件配置体验

规则条件配置应优先服务业务人员理解，而不是暴露底层技术表达式。

建议的产品口径如下：

1. 规则条件配置采用结构化表单式，不将自由脚本表达式作为默认配置方式。
2. 条件组织采用单层条件列表，所有条件共享一个全局 AND/OR 关系，不开放多层嵌套。
3. 条件左值来源统一收敛为页面字段、接口返回字段、上下文变量和固定值。
4. 操作符应按字段类型自动收敛，避免业务人员面对过多无效选项。
5. 每条条件都应展示字段含义和取值来源，而不只展示技术字段名。
6. 当前阶段不提供独立规则预览面板，编辑时通过字段校验与保存前校验保障配置质量。
7. 复杂预处理能力可以保留，但应作为受控能力存在，而不是主配置路径。
8. 不支持两层或以上条件组嵌套，复杂逻辑通过拆分规则处理。
9. 平台应在配置阶段对过度复杂的规则给出提示，引导业务人员通过拆分规则改善可读性和可治理性。
10. 每条规则都应具备可配置的数字型优先级属性，用于多规则同时命中时的展示顺序控制。优先级取值范围统一为 1-999，默认按数值越大优先级越高理解。

### 8.2B 条件校验边界

当前阶段不提供独立规则预览面板，条件配置以“行内编辑 + 保存前校验”为主。

建议明确以下边界：

1. 配置阶段提供必填校验、值类型校验和引用有效性校验。
2. 不自动带入历史命中数据或线上真实样本作为测试输入。
3. 配置页校验目标是保障可配置性与可发布性，不替代真实运行验证。
4. 真实链路验证、跨系统联调和最终结果确认，仍应在外部系统或实际业务环境中完成。

### 8.2 规则模型分析

完整规则模型至少需要以下维度：

1. 规则名称。
2. 规则绑定页面。
3. 条件集合与逻辑关系。
4. 左值和右值来源。
5. 预处理链。
6. 优先级。
7. 生效范围。
8. 有效期。
9. 状态。
10. 提示配置。
11. 是否联动作业。

从产品角度，规则对象是平台中最核心的业务对象，因为它同时连接页面、提示、作业和治理动作。

在规则标识设计上，当前建议仅保留规则名称，并对名称做常规校验，例如非空、长度、全局唯一和明显非法字符校验；不单独建设规则编码字段。

### 8.3 提示类型与展示策略

提示展示不应只考虑“能不能显示”，更要考虑“怎么显示更合理”。

建议的提示策略包括：

1. 当前阶段仅支持静默提示和浮窗提示两种展示方式。
2. 静默提示：写日志、不打断用户。
3. 浮窗提示：页面内展示提醒内容。
4. 提示关闭行为作为固定配置项，支持计时自动关闭、默认手工关闭、计时后手工关闭三种模式。
5. 提示关闭后，当前规则在本次页面停留中的当前命中上下文内不再继续执行任何后续行为。
6. 多提示命中时按规则数字优先级进入串行展示，不并发堆叠；优先级取值范围统一为 1-999，默认按数值越大优先级越高。
7. 同一规则在同一页面稳定状态下默认不重复打扰。

这背后的产品目标，是在“提示有效”与“避免疲劳”之间取得平衡。

### 8.4 提示与业务语义

产品上不应把提示只当成通用文案容器，而应让它承载业务语义。

但当前产品口径上，提示本身不区分“监管临时要求”和“风险提醒”等类型，不按提示类别做独立建模。

建议明确以下原则：

1. 提示统一作为业务规则命中后的信息承载方式。
2. 不同场景之间的差异主要体现在规则条件、触发时机、文案内容和适用范围上，而不是体现在提示类型拆分上。
3. 前端展示层不按“监管类提示”“风险类提示”等类别做差异化产品建模。
4. 业务语义主要通过具体文案和命中上下文表达，而不是通过提示类型标签表达。

### 8.5 提示与作业入口关系

提示可以按场景承载作业引导入口，但提示本身不应演变为多作业分发中心。

建议明确以下口径：

1. 提示的主职责仍然是告知和引导，而不是承担复杂操作编排。
2. 当规则仅关联一个明确作业，且该作业需要用户感知后再触发时，提示中可以承载单一作业入口。
3. 当前阶段提示中的作业入口统一收敛为“确认按钮”，不扩展更多附加操作。
4. 按钮文案统一固定为“确认”，当前不开放业务自定义。
5. 用户点击提示中的确认按钮后，即触发当前关联作业。
6. 当存在多个可执行作业时，提示中不应堆叠多个入口，复杂场景统一由悬浮按钮或独立作业入口承接。
7. 对无提示自动执行和静默提示场景，提示中不需要承载作业入口。
8. 对预览确认后执行、手动触发等需要用户参与的场景，提示中承载作业入口价值更高。
9. 提示入口用于命中当下的即时引导，悬浮按钮用于二次触发和脱离当前提示后的再次进入，两者分工应保持清晰。
10. 用户若直接关闭提示而未点击确认，则不触发关联作业，当前规则在本次页面停留中不再继续执行后续行为。

只有这样，平台的提示能力才不是简单的“配置弹窗”，而是业务知识在页面现场的表达方式。

### 8.6 提示文案配置体验

结合当前场景，提示文案配置应优先满足业务人员直接编写和快速调整的需求。

建议明确以下口径：

1. 提示文案由业务人员按场景自定义编写。
2. 当前阶段不以提示文案复用为目标，不强制建设模板化复用能力。
3. 当前阶段不强制要求标题、摘要、建议动作等结构化字段，业务可直接维护核心提示内容。
4. 提示配置应重点控制展示方式、展示时机和触发条件，而不是过早引入复杂文案模型。
5. 当后续场景数量和治理复杂度明显提升后，再评估是否演进为结构化文案模型。

## 9. 智能作业分析

### 9.1 智能作业的产品职责

智能作业层的核心职责，是把“知道应该做什么”进一步转化为“帮助用户完成动作”。

它的核心业务目标应明确指向提效。对于部分审批时间极短、人工判断窗口只有几秒的场景，平台需要允许业务侧直接配置智能作业的执行方式，以换取更强的现场效率收益；相关业务风险由对应业务管理人员承担，平台负责提供配置能力、执行控制、失败兜底和完整留痕。

它至少应覆盖：

1. 页面取值。
2. 接口查数。
3. 中间结果加工。
4. 字段映射。
5. 页面写入。
6. 预览确认。
7. 异常处理。
8. 执行日志。

### 9.2 为什么智能作业是完整产品的一部分

从产品完整性看，智能作业不能被视为外围能力，原因在于：

1. 提示解决的是认知问题，作业解决的是动作问题。
2. 业务方感知最强的效率收益，通常来自查数、预填、减少手工重复动作。
3. 页面识别、规则触发、接口定义、预处理器等能力天然可向执行层延伸。
4. 如果提示和作业分离设计，配置模型、权限、日志和治理链路会重复建设。

### 9.3 智能作业的核心场景

完整产品中的智能作业，适合承接以下场景：

1. 根据页面已有字段查外部数据并预填当前页面。
2. 在审核过程中提取字段并生成辅助比对信息。
3. 对高频重复录入页面提供半自动填写。
4. 对需要二次执行的动作，通过悬浮按钮提供复用入口。
5. 在失败后允许重新执行，且不复用脏上下文。

在对象关系上，规则和作业场景应保持分开建模、分开维护：

1. 规则负责定义“什么时候触发”。
2. 作业场景负责定义“触发后做什么”。
3. 两者分别版本化、分别治理，不应强行合并成同一个配置对象。
4. 规则可关联已有作业场景，作业场景也可被多个规则按业务语义引用，但绑定关系必须清晰可追溯。

在作业场景标识设计上，建议仅保留场景名称，并对名称做常规校验，例如非空、长度、全局唯一和明显非法字符校验；不单独建设场景编码字段。

### 9.4 场景编排模型

从产品视角，场景编排要解决的是“让业务人员能配置动作流程”，而不是做一个通用低代码引擎。

独立作业模块的页面结构，建议采用五段式组织：

1. 场景基础信息。
2. 触发关联信息。
3. 节点编排区。
4. 执行方式与预览配置。
5. 基础校验与发布区。

其中，节点编排区建议采用“流程图式主视图 + 结构化表单编辑区”的组合交互：

1. 主区域以流程图方式展示节点关系和执行顺序。
2. 节点选中后，在侧边或下方以结构化表单方式编辑配置内容。
3. 不建议采用纯表格式编排，也不建议采用完全自由拖拽、缺少约束的画布式交互。
4. 这样既保留流程可视化的直观性，又能保证业务配置过程可控、可校验、可理解。

当前平台对“测试”能力的定位应保持克制：平台仅承诺提供基础校验能力，不将完整联调测试能力作为正式产品能力承诺。涉及真实链路验证、外部接口联调、页面注入结果确认等测试动作，仍以外部系统或实际业务环境验证为主。

建议的编排原则：

1. 一个场景代表一条完整业务动作链。
2. 节点以顺序执行为主，降低理解门槛。
3. 组件要足够少但足够覆盖核心动作。
4. 配置界面应以业务语义组织，而不是暴露过多脚本细节。
5. 每个节点都应可见输入、输出、失败状态和耗时。

这里的配置主体应默认是业务人员，而不是技术人员。技术角色只在平台运维、基础能力维护和异常排查层面参与，不应成为日常场景编排和规则配置的前置条件。

### 9.5 基础组件模型

智能作业中最具产品价值的基础组件包括：

1. `page_get`：从页面读取已有字段。
2. `api_call`：请求外部或内部接口获取数据。
3. `js_script`：处理复杂转换逻辑。
4. `page_set`：将结果写入页面字段。

它们组成了一个最小而完整的“查数 -> 加工 -> 回填”动作链。

### 9.6 执行模式分析

智能作业不应只有一种执行模式，而应开放给业务侧按场景配置，并由业务管理人员承担对应业务风险。建议将执行方式明确为以下几类：

1. 无提示自动执行：适合审批窗口极短、动作标准化程度高、业务管理人明确接受风险的场景。
2. 提示后自动执行：先向用户传达业务语义，再自动进入执行流程。
3. 预览确认后执行：系统先生成待写入结果，由用户确认后再注入页面。
4. 悬浮按钮手动触发：适合用户需要自主决定执行时机的高频动作场景。

其中，前三类属于执行方式，最后一类属于补充触发入口。平台不替业务侧判断哪类场景应该承担怎样的业务风险，但必须要求每个自动化场景都绑定责任人、适用范围、有效期和发布留痕。

在采用预览确认后执行的场景中，预览确认页建议采用结构化字段对照方式呈现：

1. 以字段对照表作为主视图。
2. 展示字段名称、原值、拟写入值和数据来源。
3. 支持按字段勾选是否写入。
4. 对失败字段、异常字段或缺失字段单独标红并说明原因。
5. 顶部展示本次作业摘要、执行结果概览和必要风险提示。

### 9.7 悬浮按钮与二次触发

悬浮按钮不只是一个 UI 入口，它本质上是动作复用和用户控制权的体现。

其产品价值包括：

1. 用户在理解当前上下文后主动触发动作。
2. 对同一页面高频动作提供低成本复用入口。
3. 在自动触发失败或取消后提供重新执行路径。
4. 每次触发都形成新的执行实例，保证上下文隔离。

悬浮按钮应被视为规则联动之外的第二触发入口。也就是说，平台支持“规则联动触发”和“悬浮按钮二次触发”两类主入口；二次触发时必须新建执行实例，不复用旧实例上下文。

在产品规则上，悬浮按钮建议遵循以下原则：

1. 仅在当前页面存在可执行作业场景时展示。
2. 默认不在所有页面常驻展示，避免形成无关入口干扰。
3. 同一页面默认只保留一个主悬浮入口。
4. 点击后可展示当前页面下的可执行场景列表，或直接执行默认场景。
5. 即使自动执行场景已经执行完成，仍允许用户通过悬浮按钮再次手动触发。

### 9.8 智能作业的关键设计原则

1. 允许业务侧根据效率诉求选择自动化强度，但所有自动化动作都必须具备责任归属和审计留痕。
2. 默认失败即停止，不静默吞错。
3. 默认可以回到手工路径，不锁死在自动化流程中。
4. 默认实例化执行过程，保证事后可追溯。
5. 默认作业能力从高频、稳定、低歧义动作切入，而不是从复杂全自动场景切入。

## 10. 端到端产品闭环

从完整产品角度看，平台需要形成以下闭环：

1. 配置人员选择目标业务页面并完成页面建档。
2. 配置规则条件、提示内容和生效范围。
3. 若需要动作辅助，则配置联动作业或独立作业场景。
4. 发布后，插件进入目标页面并识别页面上下文。
5. 规则引擎执行判断并触发提示。
6. 若配置了作业动作，则进入查数、加工、预览、注入流程。
7. 触发、执行、失败、取消、关闭等全过程留痕。
8. 管理人员可查看效果、停用规则、调整范围、回滚版本。

这条闭环成立，平台才真正具备“从配置到执行再到治理”的完整产品能力。

## 11. 配置流程与交互要求

### 11.1 面向业务角色的配置流程

配置流程应尽量贴近业务人员的理解方式，而不是系统底层模型。

在流程设计上，应采用“分模块建设、主流程串联”的思路：

1. 规则和作业场景是两个独立模块，分别承担触发判断和动作编排职责。
2. 复杂作业场景允许在独立模块中单独开发、编排、调试和版本管理。
3. 规则创建流程中应支持直接选择已有作业场景，或跳转新建作业场景后自动回填绑定关系。
4. 规则创建页只承载轻量作业关联配置，不承载复杂节点编排、字段映射和脚本细节。
5. 规则创建页中的作业相关字段，建议仅保留：是否关联作业场景、已关联作业场景、执行方式、是否需要提示联动。
6. 节点编排细节、字段映射细节、脚本配置细节、预览结果模板等内容，统一放在独立作业模块中维护。
7. 这样既能保证业务主流程顺畅，又不会把复杂编排硬塞进规则页面。

对开阳等业务系统，建议采用向导式主流程，并围绕以下路径组织主要步骤：

1. 选择业务范围。
2. 选择专区。
3. 选择业务菜单。
4. 选择或补录页面。
5. 配置规则。
6. 配置提示。
7. 关联作业场景。
8. 执行基础校验。
9. 保存草稿或提交发布。

### 11.2 配置体验的关键要求

1. 上下文信息持续可见，不因步骤切换丢失。
2. 页面不存在时应能在当前流程中补录，而不是跳出流程。
3. 规则和作业的配置关系应清晰，避免双向重复配置。
4. 错误提示要结构化、可定位、可执行。
5. 发布前应进行完整性校验，避免线上才暴露问题。
6. 平台当前只提供基础校验，不承诺内置完整测试能力；涉及真实联调和结果验证，需在外部系统或实际业务环境中完成。

### 11.3 可视化与低门槛原则

产品交互应尽量遵循以下原则：

1. 多用表单、向导和结构化配置，少暴露脚本和技术术语。
2. 高频能力提供模板、默认值和示例。
3. 将复杂概念拆成业务可理解步骤，例如“页面选择”“规则判断”“作业动作”“结果确认”。
4. 将技术人员参与范围收敛到平台级运维、少量基础配置和异常支持，不把日常规则与作业配置建立在技术角色参与前提上。

## 12. 生命周期治理

### 12.1 为什么治理是核心能力

在银行场景中，平台能不能上线只是第一步，能不能长期被安全使用、被合理收口、被有效审计更重要。

治理能力至少包括：

1. 草稿、发布、停用、过期、回滚等状态管理。
2. 高风险操作审计。
3. 配置变更版本记录。
4. 生效范围控制。
5. 状态变更原因记录。

### 12.2 临时规则治理

监管或过渡性提示有天然生命周期，平台必须支持：

1. 设置责任人。
2. 设置有效期。
3. 设置停用动作。
4. 记录延期或替代原因。

否则平台会快速积累大量失效规则，影响使用体验和治理成本。

### 12.3 发布与回滚

发布治理不应只停留在“保存即生效”，而应具备完整发布行为：

1. 发布前校验依赖资源完整性。
2. 发布时记录版本与影响范围。
3. 回滚时回到确定版本，而不是手工恢复。
4. 规则、页面、作业场景的生效必须基于一致版本视图。

### 12.4 状态模型与流转规则

规则和作业场景建议统一使用以下主状态：

1. 待发布。
2. 生效中。
3. 已停用。
4. 已失效。

建议的状态流转口径如下：

1. 新建对象默认进入待发布。
2. 待发布对象在完成发布后进入生效中。
3. 生效中对象可被业务手工停用，进入已停用。
4. 生效中对象到达失效时间，或因替代、资源异常等原因退出生效状态时，进入已失效。
5. 已停用对象如需再次启用，应重新进入发布流程，而不是直接绕过发布恢复线上状态。
6. 已失效对象不直接恢复，如需继续使用，应基于当前内容或历史版本重新生成新的待发布版本。
7. 所有生效中对象只要发生内容修改，都不直接改动线上版本，而是生成新的待发布版本；新版本发布成功后，再替代当前生效版本。

### 12.5 冲突前置校验与运行时防护

同一页面、同一字段的写入冲突，原则上必须在发布前完成校验和消解，而不是留到执行时再做业务仲裁。

建议明确以下治理口径：

1. 发布前必须检查同页同字段是否被多个规则或作业场景重复写入。
2. 冲突配置默认不允许通过发布，需由业务配置人员调整后重新提交。
3. 业务管理人员如确需保留特殊冲突关系，必须在发布前完成明确确认并留痕。
4. 运行时不再做业务优先级仲裁，只保留最小化技术防护。
5. 若因缓存、脏数据、灰度残留等异常导致重复写入尝试，系统应拒绝第二次写入、记录冲突日志，并按失败处理。

### 12.6 发布前强校验清单

平台应将以下内容作为发布前的强校验项，未通过时不允许正式生效：

1. 缺少责任人，不允许发布。
2. 缺少适用范围，不允许发布。
3. 缺少生效时间或失效时间，不允许发布。
4. 同页同字段存在写入冲突，不允许发布。
5. 规则引用失效资源，不允许发布。
6. 作业场景缺少人工作业时长，不允许发布。
7. 自动执行类场景缺少业务风险责任确认，不允许发布。

### 12.7 自动执行风险确认

自动执行类场景的业务风险确认，应由业务管理角色在发布前完成。

建议明确以下口径：

1. 只有自动执行类场景需要强制风险确认。
2. 风险确认记录至少应包含确认人、确认时间、关联场景、执行方式和适用组织范围。
3. 场景执行方式从非自动改为自动时，必须重新确认。
4. 仅修改文案、说明等非执行关键字段时，不强制重新确认。
5. 修改执行链路、写入字段、触发方式等关键执行内容时，必须重新确认。

## 13. 审计与可观测性

### 13.1 需要记录什么

平台至少要记录三类日志：

1. **治理日志**：谁发布、停用、延期、回滚了什么配置。
2. **触发日志**：某条规则何时、在哪个页面、对谁命中。
3. **执行日志**：某个作业场景的每个节点输入、输出、耗时和结果。

### 13.2 为什么日志不能只是“留痕”

日志不仅用于事后审计，也用于日常运营与问题定位：

1. 页面识别失败时，能知道是页面编码问题还是 URL 未命中。
2. 规则未命中时，能知道是字段缺失、预处理失败还是接口异常。
3. 作业失败时，能知道失败节点、失败原因和影响范围。
4. 提示过多时，能分析是否存在规则冲突或触发控制失效。

### 13.3 看板与运营指标

平台应优先建设以下核心指标：

1. 作业执行成功率。
2. 平均节省时长。
3. 失败原因分布。
4. 已失效规则/场景数量。
5. 即将到期规则/场景数量。

其中，“平均节省时长”建议采用统一口径：

1. 创建作业场景时必须填写人工作业时长，作为基准值。
2. 人工作业时长建议以秒为基础单位存储。
3. 作业关键流程发生变化时，必须同步更新人工作业时长。
4. 平均节省时长 = 人工作业时长 - 实际成功完成时长。
5. 若计算结果小于 0，则按 0 处理，不记负收益。
6. 仅统计成功完成的作业执行，失败和取消执行不计入节省时长。

### 13.4 治理视图与筛选设计

面向业务管理角色的治理页面，不应默认展示“全部配置”，而应默认进入“待处理”视图，帮助业务管理人员先聚焦需要处理的对象。

建议的治理视图口径如下：

1. 首页默认进入待处理视图，而不是全部视图。
2. 主状态筛选统一包括：待发布、生效中、已停用、已失效。
3. 高频快捷筛选建议包括：待处理、即将到期、校验不通过、存在冲突、待风险确认。
4. 基础筛选维度优先保留组织范围、责任人、状态、生效时间、失效时间、执行方式、是否自动执行。
5. 不优先暴露过多技术型筛选条件，避免业务用户被低频复杂维度干扰。
6. 列表字段建议至少包含名称、类型、所属组织、责任人、执行方式、当前状态、生效时间、失效时间和最近变更时间。

## 14. 安全、合规与稳定性分析

### 14.1 安全边界

平台需要处理的安全问题主要包括：

1. 页面字段和接口数据可能涉及敏感信息。
2. 规则与作业能力可能改变用户操作行为。
3. 自定义脚本、接口代调和页面注入都具有潜在风险。

因此，产品设计上必须强调：

1. 敏感信息加密存储、脱敏展示。
2. 页面侧不暴露敏感配置和密钥。
3. 脚本能力受控使用并保留审计。
4. 自动动作可确认、可回退、可追溯。

### 14.2 稳定性要求

在实际业务环境中，平台常见的不稳定因素包括：

1. 页面结构变化。
2. 网络波动。
3. 接口超时。
4. 配置错误。
5. 浏览器兼容差异。

平台应以“持续可用、优先降级”为原则：

1. 能用缓存时继续受控运行。
2. 不能稳定执行时切回人工模式。
3. 错误信息要让用户知道下一步能做什么。

## 15. 关键产品风险

从完整产品角度，当前最值得关注的风险包括：

1. **产品边界被切碎**  
   如果提示、作业、治理分拆讨论，最终容易得到多个半成品而不是完整平台。
2. **页面资源维护成本高**  
   页面元素不稳会同时影响提示与作业，成为全局瓶颈。
3. **自动化收益与安全风险拉扯**  
   动作越自动，越要重视确认机制和审计。
4. **业务配置门槛仍然偏高**  
   如果配置流程仍像技术后台，业务角色难以真正独立使用。
5. **规则和场景长期失控**  
   没有责任人、有效期和停用机制，平台会演变成规则堆积系统。
6. **指标只看触发，不看结果**  
   如果只统计提示次数，不看执行成功、失败结构和真实效率收益，就无法评估平台价值。

## 16. 产品建议

### 16.1 已确认产品决策

基于当前产品讨论，以下口径已明确，可作为后续设计与落地的统一依据：

1. 智能作业的核心目标是提效，尤其要服务审批窗口极短、人工反应时间只有几秒的业务场景。
2. 智能作业开放给业务侧配置，业务管理人员对场景执行方式与业务风险承担管理责任。
3. 执行方式统一包括：无提示自动执行、提示后自动执行、预览确认后执行；悬浮按钮手动触发作为补充入口保留。
4. 平台支持“规则联动触发”和“悬浮按钮二次触发”两类主入口；二次触发必须新建实例，不复用旧上下文。
5. 规则和作业场景都必须绑定责任人、适用范围、有效期、发布记录和停用记录，缺少治理信息的配置不应正式生效。
6. 平台主用户与主管理者均为业务角色，技术人员只承担少量运维配置、平台支持和异常排查工作。
7. 平台增加角色管理模块，由业务侧管理角色、人员绑定和权限分配。
8. 权限模型采用“组织范围控制资源边界，操作类型控制权限边界”的简化思路，不以单条规则或单个页面细粒度授权作为常规模式。
9. 自动执行类场景的业务风险确认由业务管理角色在发布前完成；执行方式从非自动改为自动时，必须重新确认。
10. 同页同字段写入冲突必须在发布前完成校验和消解；运行时不做业务仲裁，只保留最小化技术防护与异常留痕。
11. 平台效果评估应优先关注作业执行成功率、平均节省时长、失败原因分布、已失效规则/场景数量和即将到期规则/场景数量。
12. 创建作业场景时必须填写人工作业时长，作为平均节省时长的计算基准。
13. 发布前必须执行强校验清单，责任人、范围、生效失效时间、冲突校验、人工作业时长和自动执行风险确认缺一不可。
14. 角色支持复制、停用、恢复和批量分配；已使用角色不做物理删除。
15. 面向业务管理角色的治理页面默认进入“待处理”视图，而不是“全部”视图。
16. 规则和作业场景统一采用“待发布、生效中、已停用、已失效”四类主状态；所有生效中对象的内容修改都必须生成新的待发布版本，不能直接改线上版本。
17. 平台只承诺基础校验能力，不承诺内置完整测试能力；涉及真实联调和结果验证，应在外部系统或实际业务环境中完成。
18. 提示展示当前仅支持静默提示和浮窗提示两种方式；关闭行为支持计时自动关闭、默认手工关闭、计时后手工关闭三种模式；提示关闭后当前规则在本次页面停留中不再继续执行后续行为。
19. 提示可按场景承载单一作业引导入口，当前入口形态统一为提示确认按钮，按钮文案固定为“确认”；用户点击确认后即触发关联作业，若关闭提示则不触发作业；不扩展多按钮或复杂附加功能，复杂或多作业场景由悬浮按钮或独立作业入口承接。
20. 规则增加数字型优先级属性，并作为多规则同时命中时的提示展示顺序依据；优先级取值范围统一为 1-999，默认按数值越大优先级越高。
21. 规则仅保留名称字段并做常规校验，名称在平台内需全局唯一，不单独建设规则编码。
22. 作业场景仅保留名称字段并做常规校验，名称在平台内需全局唯一，不单独建设场景编码。
23. 业务管理角色不默认拥有配置权限；平台支持角色拥有查看、校验和审计查看权限，用于技术支持场景。
24. 规则条件编辑当前统一为单层模型，不开放多层条件组嵌套；所有条件共享一个全局 AND/OR 逻辑。
25. 条件编辑顺序固定为：左值类型、选择值、左预处理器、逻辑运算、右值类型、选择值、右预处理器；预处理器交互采用下拉多选而非弹窗。
26. 作业场景中心不展示执行实例模块，不保留手工预览区；编排主界面采用左（节点库）-中（画布）-右（属性）布局。
27. 作业编排画布支持节点选中后右侧属性编辑与一键自动排版，以提升大场景可视性与编辑效率。

### 16.2 产品层建议

基于完整性分析，建议在产品文档层面坚持以下原则：

1. 将营小助配置中心定义为“提示 + 作业 + 治理”的统一产品，而不是单模块产品。
2. 将页面资源、规则对象、作业场景和审计体系放在同一产品框架内建模。
3. 明确智能作业是平台核心能力之一，并在产品文档中做完整分析，而不是只做架构预留。
4. 明确平台主用户与主管理者均为业务角色，技术人员只承担少量运维配置与平台支持工作。
5. 将预览确认、失败降级、状态治理和日志追踪视为平台主能力，而不是补充功能。

### 16.3 当前状态

当前主产品文档中的关键产品口径已完成确认，暂不保留新的待业务确认事项。角色与权限矩阵已单独沉淀并确认于 `spec/config-center-product/permission-model.md`。

## 16.4 产品文档拆分建议（智能提示 vs 智能作业）

结论：**有必要拆分**，并保留一份平台总文档作为总纲。

建议文档结构：
1. 总纲文档（平台级）：产品定位、统一术语、角色权限、共享模型、发布治理、审计口径。
2. 智能提示产品文档（子文档）：规则模型、触发链路、提示交互、关闭策略、提示治理。
3. 智能作业产品文档（子文档）：场景编排、节点模型、执行方式、预览确认、失败降级。

拆分收益：
1. 降低跨模块变更耦合，提升评审效率。
2. 避免提示与作业互相覆盖需求细节，减少口径冲突。
3. 支持不同节奏迭代（提示先行、作业增强）且不影响平台总口径稳定。

拆分原则：
1. 跨域规则只在总纲维护，子文档只引用不重复定义。
2. 共享对象（页面资源、接口、预处理器、角色）由总纲主责。
3. 子文档变更需回写总纲的“影响边界”与“版本记录”。
## 17. 最终结论

从产品完整性、业务闭环和治理可控性三方面综合判断，营小助配置中心的合理产品形态应是一个统一的智能辅助平台：

1. 通过页面资源管理建立稳定的页面基础模型。
2. 通过智能提示把业务规则、风险约束和操作指引带到现场。
3. 通过智能作业把查数、预填、复核、注入等高频动作配置化落地。
4. 通过发布、停用、回滚、日志和审计形成完整治理闭环。

也就是说，这个产品的完整价值不在于“能不能提示”，也不在于“能不能自动执行”，而在于它是否能把**业务知识、页面上下文、动作能力和治理机制**收敛到同一个平台中，并真正服务银行内网业务场景的准确性、效率和可控性。
