# 营小助配置中心 PRD 需求分析文档（基于 Spec v2026-03-11）

> 来源文档：`spec/config-center-product/spec.md`  
> 分析目标：将已确认 Spec 重组为可执行 PRD 视图，并识别落地风险与需确认项  
> 分析日期：2026-03-11

## 1. 产品定义

营小助配置中心是银行内网业务页面的智能辅助平台，通过预注入 JSSDK 提供两类核心能力：
1. 智能提示：监管要求、风险提醒、作业指引。
2. 智能作业：基于提示触发的数据查询、场景编排与页面注入。

定位边界：不重构业务系统流程，不改变审批流，仅做“提示 + 辅助填充 + 审计追踪”。
平台定位补充：配置平台主要由业务人员使用，产品交互需满足低门槛、易上手。

## 2. 业务目标与成功标准

### 2.1 业务目标
1. 监管与风险要求触达时效提升到分钟级。
2. 人工录入/查数步骤减少 30%+。
3. 因遗漏提示导致的差错率下降 20%+。
4. 业务人员无需代码即可完成常见规则配置与发布。

### 2.2 上线成功指标（建议固化为 SLO）
1. 提示触发链路 P95 <= 800ms。
2. 作业注入链路 P95 <= 1500ms（不含上游超时重试）。
3. 关键页面注入成功率 >= 98%。
4. 高风险提示漏报率 <= 1%。
5. 审计日志完整率 = 100%。

## 3. 用户与场景

### 3.1 目标用户
1. 一线运营人员。
2. 审核人员。
3. 规则运营人员。
4. 管理/合规人员。

### 3.2 核心场景
1. 页面命中监管临时规则后立即弹窗提示。
2. 录入前从内部系统取数，预览后一次性注入。
3. 同轮命中多提示时合窗展示，并按触发顺序执行联动作业。
4. 对重复动作通过悬浮按钮二次触发，且每次新建编排实例。

## 4. 端到端业务流程（PRD 主链路）

1. 业务系统预注入 JSSDK，外部 SDK 监听 DOM 变化并通知智能提示 SDK。
2. 智能提示引擎取值并执行条件表达式（固定值/页面 XPath/接口值，支持 OR/AND 与预处理器链）。
3. 规则命中后统一提示渲染（同轮合窗）。
4. 若规则配置联动作业，则在提示渲染后触发智能作业。
5. 作业执行场景编排（`page_get -> api_call/js_script -> page_set` 等串行节点）。
6. 若开启预览模式，在编排完成后统一展示待填充快照，用户确认后一次性写入。
7. 注入/失败/取消全量记录，日志查询按敏感级别脱敏。

## 5. 功能需求分解（MVP/P0）

### 5.1 智能提示模块
1. 规则模型：条件表达式（左值/运算符/右值）+ 条件关系（OR/AND）。
2. 取值来源：固定值、页面 XPath、接口返回值。
3. 预处理器：左右独立链路、顺序执行、失败按不满足处理。
4. 展示策略：风险分级、同轮合窗、高风险不可静默消失。
5. 触发治理：DOM 防抖、重复弹窗抑制、联动触发顺序可追溯。

### 5.2 智能作业与编排模块
1. 触发边界：仅允许提示命中触发，不提供独立入口。
2. 编排模型：单作业绑定单编排，节点顺序串行执行。
3. 首批节点：`page_get`、`page_set`、`api_call`、`js_script`。
4. 失败策略：固定 `STOP`，失败留痕但不阻断业务主流程。
5. 预览模式：编排完成后统一预览、支持编辑、确认后一次性写入。
6. 悬浮按钮：仅提示命中后可见，点击触发新实例，不复用上下文。

### 5.3 接口配置子项模块
1. 生命周期：`DRAFT -> PUBLISHED -> OFFLINE`。
2. 引用约束：仅 `PUBLISHED` 可被规则引用；被引用接口禁止下线。
3. 权限控制：仅 `owner` 与管理员可发布/下线。
4. 参数映射：固定值 + 页面取值，支持 `test_value`，默认 `string`。
5. 平台兜底：`timeout_ms`、`retry_count` 平台统一控制，前端不暴露。

### 5.4 审计与运营模块
1. 事件日志：触发、命中、注入、取消、失败原因、触发来源。
2. 条件日志：左右值取值与预处理结果、判定明细。
3. 编排日志：节点输入输出快照、失败节点、实例状态。
4. 指标看板：命中率、采纳率、注入成功率、平均节省时长。

### 5.5 配置平台易用性模块
1. 常见场景采用可视化表单与向导流程，避免依赖脚本或代码。
2. 提供高频模板、默认值和示例，减少从零配置成本。
3. 表单校验支持字段级错误定位与修复建议，降低误配率。
4. 规则创建、联动作业绑定、发布与生效范围在同一主流程完成。

## 6. 非功能与合规要求

1. 全链路 TLS，敏感字段加密存储，查询/展示脱敏。
2. 审计日志保留 >= 180 天（按行内规范可调整）。
3. 浏览器兼容：内网 Chrome/Edge 主流版本。
4. 网络降级：提示可用缓存规则，注入降级为人工确认。
5. 易用性要求：业务人员可在向导流程中独立完成“规则创建 -> 联动作业绑定 -> 发布并设置生效范围”。

## 7. 版本范围与里程碑建议

1. V1（P0）：聚焦开阳系统菜单级闭环，完成“提示->联动作业->编排->预览->注入->审计”。
2. V1.1（P1）：增强提示疲劳治理、低置信度人工确认、A/B 效果对比。
3. 上线策略：菜单级灰度 -> 机构扩展 -> 全量上线。

## 8. 原始 Spec 优化空间（按优先级）

### P0（上线前必须补齐）
1. **脚本安全口径冲突**：`JS 预处理器`要求沙箱隔离且禁止访问页面对象；但`js_script 节点`允许直接读写页面对象。需明确两类 JS 的权限边界、审计级别与发布门禁。
2. **注入冲突仲裁规则不完整**：文档有“同字段同一时刻只执行一条注入”约束，但缺少多规则命中/多作业串行时的确定性仲裁算法（优先级、时间戳、规则版本）。
3. **频控参数缺失**：已定义“防抖与频控”，但未给出数值口径（时间窗、阈值、按用户/页面/字段维度），会导致实现与验收不一致。
4. **指标口径未数学化**：命中率、采纳率、节省时长缺少分母定义与统计窗口，运营看板无法统一。
5. **权限模型不完整**：除 owner/admin 外，缺少租户、机构、角色到资源（规则/接口/编排/日志）的细粒度授权矩阵。

### P1（建议在试点期补齐）
1. **XPath 单一定位风险高**：复杂 SPA 或动态 DOM 下稳定性偏弱，建议增加 selector 备选策略与定位健康检查。
2. **日志全量落库成本与策略缺失**：仅说明全量存储，缺少分层存储、冷热分级、归档与检索 SLA。
3. **跨 iframe 场景约束不清**：兼容性写了 iframe，但未定义跨域 iframe 的可观测/可注入边界。

## 9. 确认结果（2026-03-11）

1. 同意将“预处理器 JS”和“编排脚本 JS”拆分为两套安全等级与发布流程。
2. 同意同字段冲突采用“风险等级优先 > 规则优先级 > 触发时间”的固定仲裁顺序。
3. 同意防抖频控默认值：2 秒防抖、30 秒内同规则最多触发 1 次（后续可配置化）。
4. 同意运营指标按“自然日 + 菜单维度”统一统计。
5. 同意 V1 增加 RBAC 资源矩阵（规则/接口/编排/日志四类）。
6. 同意在 XPath 基础上允许 CSS Selector 兜底定位。
7. 同意日志分层策略：近 30 天热存 + 180 天冷存。
8. 确认“发布与配置生效范围合并管理”：发布时必须一体化提交生效范围（机构/系统/角色/业务菜单）。

## 10. 结论

当前 `spec.md` 业务方向和功能闭环完整，已具备进入开发的主干条件。关键决策已完成确认并回填到 Spec，当前主要关注点转为“跨 iframe 边界细化与试点期稳定性验证”。
