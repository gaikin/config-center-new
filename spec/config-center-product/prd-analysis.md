# 营小助配置中心 PRD 深度分析文档

> 主产品文档：`spec/config-center-product/prd-analysis.md`  
> 分析视角：产品完整性、业务闭环、角色价值、平台控制与审计可控性  
> 更新日期：2026-03-12  
> 说明：本文从完整产品视角重组需求，强调营小助配置中心作为“提示 + 作业 + 平台控制”一体化平台的整体形态，并作为当前唯一产品总纲；角色与权限专题内容另见 `spec/config-center-product/permission-model.md`。  

## 1. 产品结论

营小助配置中心的完整产品形态，不应被理解为一个单纯的提示配置后台，也不应被拆解为彼此割裂的“提示系统”和“作业系统”。从产品完整性看，它本质上是一套面向银行内网页面的智能辅助平台，核心目标是把业务知识、风险约束、操作动作和平台控制能力统一沉淀到同一个配置中心中。

完整产品应同时解决四类问题：

1. 监管与业务口径变化快，但正式系统改造慢，现场缺少快速触达能力。
2. 一线办理和审核场景中存在大量重复查数、重复录入、重复判断动作。
3. 提示、动作、页面资源、发布控制分散在不同系统时，维护成本高且追溯困难。
4. 规则上线后容易堆积，缺少停用、回滚、审计和效果验证机制。

因此，营小助配置中心的合理产品定义应是：

> 一套面向银行内部业务页面的智能辅助平台。  
> 通过统一配置中心管理页面资源、提示规则、作业场景、接口能力、发布控制和审计追踪，在业务办理现场提供实时提示、自动辅助、可控执行和可追溯上线能力。

## 2. 产品定位

### 2.1 产品一句话定义

营小助配置中心是银行内部业务页面的智能辅助平台，通过页面识别、规则判断、提示渲染、作业执行、发布控制和审计追踪，帮助业务人员在办理现场更准确、更高效、更可控地完成业务动作。

### 2.2 产品本质

从产品本质看，这个平台不是“弹窗工具”，而是“页面级业务辅助基础设施”。  
它有三个不可分割的组成部分：

1. **提示能力**：告诉用户当前应该关注什么、注意什么、先做什么。
2. **作业能力**：帮助用户减少查数、切换、复制粘贴和重复录入。
3. **平台控制能力**：确保规则和动作可发布、可停用、可回滚、可审计、可复盘。

如果只强调其中一个部分，产品都会失衡：

1. 只有提示，没有作业，价值偏“知道”，缺少“做到”。
2. 只有作业，没有提示，价值偏“自动化”，缺少业务语义和风险约束。
3. 只有执行，没有平台控制，平台会在规模化后失控。

同时，平台的使用者定位也必须明确：**主用户是业务人员，主要发布与风险确认责任人也是业务管理人员**。  
技术角色不应被设计成平台的日常主操作者，而应主要承担少量运维配置、平台能力维护、异常排查和基础保障工作。  
这意味着整个平台的术语、交互、配置路径和权限设计，都必须默认站在业务角色视角，而不是技术视角。

### 2.3 产品边界

营小助配置中心需要保持清晰边界：

1. 不替代业务系统原生审批流和最终确认动作。
2. 不承担银行核心系统后端改造职责。
3. 不以自动化覆盖所有业务判断，复杂场景保留人工决策权。
4. 不以脚本自由度为目标，而以业务可理解、平台可控制为目标。
5. 出现异常时必须支持降级到人工路径，不得阻断业务继续办理。

## 3. 业务问题与机会

### 3.1 当前业务痛点

从银行内网页面场景出发，当前存在的核心痛点包括：

1. 监管临时要求往往通过邮件、通知、会议或线下传达，难以在操作现场即时落地。
2. 风险提示往往停留在制度和培训中，用户在具体页面、具体字段、具体步骤上缺少上下文提醒。
3. 新员工对操作顺序、注意事项、字段来源不熟悉，容易出现漏项和误填。
4. 一线人员频繁在多个系统之间切换，查数、复制、录入流程冗长且易出错。
5. 页面结构变化后，页面元素配置和提示规则容易失效，但问题定位困难。
6. 分行、总行、审计等角色对规则的创建、启停、发布、复盘缺少统一发布与审计入口。

### 3.2 平台的机会点

营小助配置中心的机会不只是“把提醒弹出来”，而是利用页面上下文把业务规则变成现场可执行能力：

1. 在正确的页面、正确的时机、针对正确的人触发正确的提示。
2. 在提示出现后，顺势把所需数据查询、字段填充、预览确认串成动作闭环。
3. 将页面资源、规则逻辑、接口能力和日志审计统一建模，避免重复建设。
4. 让业务人员成为主要配置者、让业务管理人员成为主要发布与风险责任承担者，降低对技术人员持续参与的依赖。

## 4. 目标用户与角色价值

| 角色 | 主要目标 | 关注点 | 平台提供的核心价值 |
|------|------|------|------|
| 一线运营人员 | 顺利完成业务办理 | 准确、快捷、少出错 | 提示及时、自动查数、减少录入 |
| 审核人员 | 提升审核效率与准确性 | 风险可见、数据可比 | 风险提示、字段获取、复核辅助 |
| 业务配置人员 | 快速配置并调整规则 | 易学、易配、少依赖研发 | 可视化配置、模板复用、校验提示 |
| 页面配置维护人员 | 维护页面资源稳定可用 | 页面识别准确、元素统一管理 | 页面建档、元素映射、版本管理 |
| 分行管理人员 | 本地化发布管理与执行落地 | 可启停、可延期、可审计 | 生效范围、状态控制、日志查看 |
| 总行管理人员 | 统一下发关键规则与能力 | 全行一致、可追踪 | 强控规则、统一配置、全局视图 |
| 审计/合规人员 | 追溯谁做了什么 | 证据完整、口径清晰 | 状态变更记录、触发日志、执行日志 |
| 技术/运维支持人员 | 保障平台稳定运行 | 可用性、异常排查、少量基础配置 | 平台级运维、排障支持、通用能力维护 |

产品设计必须同时照顾三类体验：

1. 使用者体验：提示清楚、动作可靠、不要打扰过度。
2. 配置者体验：流程顺、术语清晰、出错可定位。
3. 管理者体验：能看见、能收口、能追责、能复盘。

这里的配置者和管理者，默认都应理解为业务角色。  
技术角色不是产品信息架构的默认参照对象，不应要求业务配置流程建立在技术人员深度参与的前提上。

## 5. 核心产品场景

### 5.1 监管临时要求承接

当监管或上级单位下发临时性要求，而正式系统尚未完成改造时，平台需要在目标页面和关键节点展示规则化提示，并视场景需要联动后续作业动作，帮助一线人员快速执行临时口径。

### 5.2 事中风险控制

当办理过程命中风险条件时，平台不仅要展示风险原因和处理建议，还应支持根据风险等级决定是否只提醒、是否触发辅助查数、是否要求用户预览确认后再继续。

### 5.3 新员工作业指导

平台需要将静态 SOP 转化为页面上下文中的动态指导，在关键字段、关键步骤、关键节点向新员工提供结构化指引，并在必要时通过作业能力直接辅助完成部分动作。

### 5.4 高重复录入与审核辅助

对于需要跨系统查数、复制字段、重复录入的业务页面，平台应支持通过配置化作业能力完成数据获取、映射、预览和注入，减少低价值重复劳动。

### 5.5 异常中断与手工兜底

当接口失败、页面结构变化、元素缺失或权限异常时，系统必须给出可理解的失败信息，并允许用户回退到手工路径，不因平台异常阻断业务。

## 6. 完整产品能力模型

完整产品应被设计为三层能力协同：

| 能力层 | 核心内容 | 作用 |
|------|------|------|
| 公共支撑层 | 页面资源、API注册、预处理器、资源共享、权限、版本、审计 | 提供统一基础设施 |
| 智能提示层 | 页面识别、规则匹配、提示渲染、触发控制、多提示控制 | 提供业务语义判断与提醒 |
| 智能作业层 | 场景编排、节点执行、字段映射、预览确认、页面注入、失败降级 | 提供动作辅助与效率提升 |

这三层不是松散组合，而是强依赖关系：

1. 公共支撑层提供唯一页面资源和通用能力底座。
2. 智能提示层负责把页面上下文转成“是否应该提醒/触发”的判断结果。
3. 智能作业层负责把判断结果继续转成可执行动作。

## 7. 公共支撑层分析

### 7.1 页面资源管理

页面资源管理是整个平台的起点。没有稳定的页面资源模型，提示和作业都会不稳定。

页面资源管理至少应包括：

1. 监控网站配置。
2. 监控网页配置。
3. 页面编码与 URL 识别规则。
4. 本页元素配置。
5. 元素逻辑名与选择器映射。
6. 页面、元素、识别规则的版本管理。
7. 页面字段定义与公共字段复用关系。
8. 页面字段元素类型、定位器与读写属性配置。

产品上的关键要求有三点：

1. 页面元素应“一处维护，多处复用”，避免规则和作业分别维护选择器。
2. 页面识别应优先使用更稳定的业务标识，再用 URL 做保底。
3. 页面配置流程要尽量贴近业务系统的组织方式，例如专区、菜单、页面的逐级定位。

页面字段建模建议补充以下口径：

1. 条件来源统一收敛为“页面字段、API字段、名单字段、上下文变量、固定值”五类；不新增“DOM 文本”额外来源类型。
2. 页面字段内部再区分“公共字段”和“页面特有字段”，用于表达复用范围，而不是来源类型。
3. 公共字段主要解决条件可迁移；若业务希望复用整条规则，则需在智能提示层引入“规则模板”能力，而不是把规则直接做成无页面上下文约束的普通规则。
4. 规则模板仅允许引用公共字段；模板复用到页面后生成独立页面规则，再按需补充页面特有字段。
5. 业务人员的主交互应是“新建页面规则时快速套用模板”，而不是先进入独立模板维护流程再回到规则配置。
6. 模板应作为创建页面规则时的快捷来源，自动带入条件树、提示方式与关闭策略，页面规则保存后仍保持独立发布与版本控制。
7. 可单独建设“规则模板中心”供专家角色沉淀和维护高复用模板，但不应替代规则中心中新建页面规则的主路径。
8. 页面资源录入字段时必须选择元素类型，至少区分 `INPUT`、`SELECT`、`TEXTAREA`、`BUTTON`、`READONLY_TEXT`。
9. 页面资源保存统一 `locator` 模型；V1 可优先使用 `XPath`，但平台模型不锁死为 `XPath-only`。
10. 规则和作业只引用页面字段，不直接保存选择器或 `XPath`。

### 7.2 API注册

API注册不是附属模块，而是提示和作业都依赖的重要基础能力。

它至少要承担：

1. 定义 API 的入参、出参和取值路径。
2. 支持页面取值与固定值作为参数来源。
3. 统一超时、重试、敏感信息保护与后端代调。
4. 为规则判断和作业场景提供复用能力。

从产品角度看，API注册模块的价值在于“统一能力管理”，否则每条规则、每个场景都可能私有化定义接口，造成不可控扩张。

### 7.2A API注册对象模型

API注册建议统一为一个标准对象，至少包含以下内容：

1. 基础信息：API名称、唯一标识、所属组织、责任人、用途说明。
2. 调用定义：请求方法、目标地址、鉴权方式、请求头模板。
3. 参数清单：参数名、类型、是否必填、来源类型、来源值、默认值、测试值。
4. 响应映射：响应字段路径、目标业务字段、字段类型、是否必填、脱敏策略。
5. 运行策略：统一超时和重试策略由平台托管，不作为业务侧自由配置项。
6. 引用关系：当前被哪些规则与作业场景引用，以及引用状态。
7. 状态与版本：与平台主状态口径一致，采用待发布、生效中、已停用、已失效。

### 7.2B 参数来源与映射规则

API参数来源建议收敛为四类：

1. 固定值。
2. 页面字段。
3. 运行时上下文变量。
4. 上游节点或规则输出值。

映射规则建议明确如下：

1. 必填参数缺失时，不发起 API 调用，按失败处理并留痕。
2. 映射配置优先采用结构化表单，不以自由脚本作为默认路径。
3. 参数类型默认按字符串处理，显式声明后才按数字、布尔或日期等类型转换。
4. 响应映射需支持路径级配置，避免业务侧编写解析脚本。
5. 敏感字段在配置与日志展示中必须脱敏。

### 7.2C 版本与生命周期控制

API注册的版本控制建议遵循平台统一原则：

1. 生效中版本不直接修改，任何内容变更都生成新的待发布版本。
2. 新版本发布成功后替换线上生效版本，不通过新建 API 对象承载变更。
3. API兼容策略统一为“版本升级”：无论变更是否破坏兼容，均在原 API 对象上生成新待发布版本。
4. 已被生效中规则或场景引用的 API 不允许直接停用或失效。
5. 跨组织复用优先采用“共享只读”或“复制后维护”，避免直接跨组织改写。
6. API 的停用、恢复、延期、回滚都必须记录操作人、原因和时间。

### 7.2D 发布前强校验（API维度）

API注册发布前至少需要以下强校验：

1. 责任人、所属组织和用途说明完整。
2. 请求方法、地址、鉴权配置完整。
3. 必填参数存在且来源有效。
4. 响应映射路径合法，关键映射字段不缺失。
5. 敏感字段脱敏策略已配置。
6. 被下游规则或作业依赖的关键字段若发生破坏性变更，必须完成影响评估与引用方确认，未确认不允许发布。
7. 引用冲突与依赖风险已提示并可追溯。

### 7.2E 运行时执行与失败兜底

API注册在运行时建议遵循以下行为边界：

1. API调用通过平台后端代调，不在前端暴露敏感鉴权信息。
2. 调用失败、超时、参数缺失时，返回可理解错误原因并写入执行日志。
3. API失败不应阻断业务主流程，允许回退手工路径。
4. 多次失败应有可观测指标，支持按 API、页面、场景维度排查。
5. 执行日志需至少记录 trace、入参来源、调用结果、耗时和失败原因。

### 7.2F 配置体验与边界

面向业务角色的 API注册配置体验建议如下：

1. 采用向导式配置流程，先定义 API，再定义参数映射与响应映射。
2. 提供常用模板，减少从零配置成本。
3. 提供基础校验与测试值验证，帮助提前发现明显配置错误。
4. 平台只承诺配置有效性校验，不承诺内置真实环境全链路联调能力。
5. 复杂场景下的联调与结果验证，仍应在外部系统或真实业务环境完成。

### 7.2G API注册新建页面设计

API注册新建页面采用双栏布局：左侧维护基础信息，右侧维护输入与输出参数配置。

左侧基础信息区字段如下：

1. 名称。
2. 描述。
3. 方法。
4. 测试环境路径。
5. 生产环境路径。

右侧参数配置区分为“输入参数”和“输出参数”两部分。

输入参数建议采用 4 个 Tab：

1. Path 参数。
2. Query 参数。
3. Header 参数。
4. Body 参数。

输入参数配置规则建议如下：

1. 每个 Tab 支持结构化表格配置参数名、类型、是否必填、描述、默认值、示例值。
2. Body 参数支持 JSON 解析：可粘贴 JSON 自动生成参数结构。
3. Body 在 JSON 解析后仍可手工微调字段。
4. JSON 解析失败时应给出可定位的错误提示，不允许静默失败。

4 个输入 Tab 的字段规范建议如下：

1. Path Tab：参数名必填且全局唯一；默认必填；不支持空默认值覆盖路径参数。
2. Query Tab：支持必填与可选；可配置默认值；支持重复键开关（例如数组参数）。
3. Header Tab：支持是否必填、是否敏感、是否透传；敏感 Header 在日志中必须脱敏。
4. Body Tab：支持树形结构编辑；字段支持名称、类型、是否必填、描述、默认值；支持对象与数组嵌套。

输入参数校验规则建议如下：

1. 同一 Tab 内参数名不可重复；跨 Tab 同名时必须显式提示来源差异。
2. Path 中声明的参数必须在测试/生产路径模板中存在占位符并一一对应。
3. Body 解析 JSON 后，字段类型冲突时需给出冲突明细并阻断保存。
4. 必填参数若缺少类型或名称，阻断保存并定位到具体行。
5. 参数类型变更若影响已配置映射，需给出影响提示。

输出参数配置规则建议如下：

1. 支持手工添加出参。
2. 每个出参至少支持配置：名称、描述、类型。
3. 当类型为对象时，支持继续细化属性结构。
4. 当类型为数组且元素为对象时，支持细化对象属性结构。
5. 输出参数支持 JSON 直接解析并自动生成结构树。
6. JSON 解析生成后，允许手工补充描述与类型修正。

输出参数树形细化规则建议如下：

1. 支持基础类型：string、number、integer、boolean、object、array。
2. object 类型必须可新增子属性，并支持属性级描述与必填标识。
3. array 类型必须声明元素类型；元素为 object 时可继续配置属性树。
4. 顶层与同级属性名称不可重复。
5. 删除对象节点时，应联动删除其全部子属性并二次确认。

交互与可用性要求建议如下：

1. 输入输出参数区应支持折叠与展开，降低复杂接口编辑负担。
2. 当字段较多时，参数区应支持搜索与快速定位。
3. 新建页保存前应校验：基础信息必填、路径格式、参数结构合法性。
4. 若测试环境路径与生产环境路径不一致，应保留差异提示，避免误发布。
5. 本页面以“可配置性与可发布性校验”为目标，不替代真实环境联调验证。
### 7.2H API注册新建页面字段字典与错误码

#### 7.2H.1 左侧基础信息字段字典

| 字段 | 类型 | 必填 | 默认值 | 规则说明 |
|------|------|------|--------|----------|
| 名称 | string | 是 | 无 | 2-64 字符；同组织内唯一；仅允许中英文、数字、`-`、`_`、空格。 |
| 描述 | string | 否 | 空 | 最长 500 字符。 |
| 方法 | enum | 是 | `POST` | 允许值：`GET` / `POST` / `PUT` / `DELETE` / `PATCH`。 |
| 测试环境路径 | string(url/path) | 是 | 无 | 必须为合法 HTTP/HTTPS 地址或规范路径模板。 |
| 生产环境路径 | string(url/path) | 是 | 无 | 必须为合法 HTTP/HTTPS 地址或规范路径模板。 |

#### 7.2H.2 右侧参数通用字段字典

| 字段 | 类型 | 必填 | 默认值 | 规则说明 |
|------|------|------|--------|----------|
| 参数名 | string | 是 | 无 | 1-64 字符；同级唯一；推荐小写下划线。 |
| 类型 | enum | 是 | `string` | 允许值：`string` / `number` / `integer` / `boolean` / `object` / `array`。 |
| 是否必填 | boolean | 是 | `false` | Path 参数固定为 `true`。 |
| 描述 | string | 否 | 空 | 最长 500 字符。 |
| 默认值 | any | 否 | 空 | 类型需与“类型”一致。 |
| 示例值 | any | 否 | 空 | 用于测试和联调，不参与正式运行。 |
| 是否敏感（Header） | boolean | 否 | `false` | 仅 Header Tab 展示；为 `true` 时日志脱敏。 |
| 是否透传（Header） | boolean | 否 | `true` | 仅 Header Tab 展示；控制运行时是否透传。 |
| 是否允许重复键（Query） | boolean | 否 | `false` | 仅 Query Tab 展示；用于数组型查询参数。 |

#### 7.2H.3 Body 与出参树形字段规则

1. `object` 类型：必须支持 `properties` 子节点，子节点可继续嵌套。
2. `array` 类型：必须声明 `items` 类型；`items=object` 时需支持属性树。
3. 树形节点支持拖拽排序，但保存时按树结构持久化。
4. 删除父节点时联动删除全部子节点并弹出二次确认。
5. 字段路径（如 `data.customer.name`）在同一作用域内必须唯一。

#### 7.2H.4 错误码与提示规范（新建页）

| 错误码 | 触发场景 | 阻断级别 | 提示要求 |
|--------|----------|----------|----------|
| `API_NAME_REQUIRED` | 名称为空 | 阻断保存 | 高亮名称字段并提示“请填写 API 名称”。 |
| `API_NAME_DUPLICATED` | 名称重复 | 阻断保存 | 提示冲突对象并引导改名。 |
| `METHOD_INVALID` | 方法不在枚举内 | 阻断保存 | 提示可选方法列表。 |
| `TEST_URL_INVALID` | 测试环境路径不合法 | 阻断保存 | 指明非法片段位置。 |
| `PROD_URL_INVALID` | 生产环境路径不合法 | 阻断保存 | 指明非法片段位置。 |
| `PATH_PARAM_MISMATCH` | Path 参数与路径占位符不一致 | 阻断保存 | 标明缺失或多余参数。 |
| `PARAM_NAME_DUPLICATED` | 参数名重复 | 阻断保存 | 定位到重复行。 |
| `PARAM_TYPE_CONFLICT` | 默认值/示例值与类型不匹配 | 阻断保存 | 展示预期类型与实际值。 |
| `BODY_JSON_PARSE_FAILED` | Body JSON 解析失败 | 阻断保存 | 展示行号、列号、错误原因。 |
| `OUTPUT_JSON_PARSE_FAILED` | 出参 JSON 解析失败 | 阻断保存 | 展示行号、列号、错误原因。 |
| `OBJECT_CHILD_MISSING` | object 未配置任何子属性 | 阻断发布 | 引导补充 properties。 |
| `ARRAY_ITEM_MISSING` | array 未配置 items 类型 | 阻断发布 | 引导配置 items。 |
| `SENSITIVE_HEADER_NOT_MASKED` | 敏感 Header 未启用脱敏策略 | 阻断发布 | 强制设置脱敏开关。 |

#### 7.2H.5 保存与发布校验分层

1. 保存校验：字段完整性、格式合法性、结构可解析性。
2. 发布校验：敏感字段策略、路径参数一致性、对象/数组结构完整性。
3. 运行前校验：仅做最小化防护，不替代真实环境联调与压测。
### 7.2I API注册列表页调试功能

在 API 完成保存后，列表页需提供“调试”入口，支持业务配置人员快速验证接口配置有效性。

#### 7.2I.1 入口与可见性

1. 仅当接口已保存成功（至少存在一个可读取版本）时，列表项展示“调试”按钮。
2. 列表页可按版本维度触发调试，默认使用当前选定版本。
3. 无调试权限时隐藏按钮，不展示置灰不可点状态。

#### 7.2I.2 调试面板字段

调试面板建议分为三块：

1. 基础区：接口名称、方法、当前版本、目标环境（默认测试环境）。
2. 参数区：Path / Query / Header / Body 四类调试参数输入。
3. 结果区：响应概览与映射结果。

参数区规则：

1. Path / Query / Header 参数支持键值对编辑。
2. Body 支持 JSON 文本输入、格式化和清空。
3. 参数可一键回填示例值，便于快速调试。
4. 调试参数仅用于本次请求，不回写接口正式配置。

#### 7.2I.3 请求组装与执行流程

1. 点击“开始调试”后，系统基于接口版本快照组装请求。
2. 组装顺序：路径参数替换 -> Query 拼接 -> Header 注入 -> Body 组装。
3. 路径参数缺失时阻断调试并提示缺失字段。
4. 生产环境调试仅在具备权限时可选；默认始终落在测试环境。
5. 调试请求超时后直接结束本次执行，并保留当前输入参数。

#### 7.2I.4 结果展示规范

调试结果至少展示以下内容：

1. 响应状态码。
2. 响应耗时。
3. 响应头。
4. 响应体。
5. 出参映射结果。

展示规则：

1. 出参映射结果与右侧出参结构树联动展示，定位映射成功/失败字段。
2. 响应体支持 JSON 树与原始文本双视图切换。
3. 映射失败字段需标记原因，例如“路径不存在”“类型不匹配”。
4. 调试结果只做即时展示，不作为发布门禁依据。

#### 7.2I.5 失败提示规范

1. 失败提示需直接指出失败阶段：参数校验失败、请求超时、响应解析失败、映射失败。
2. 失败提示应包含可执行建议，例如“补齐 path 参数 customer_id”。
3. 不展示技术堆栈原文，避免对业务用户造成噪音。

### 7.3 预处理器中心

预处理器中心承载的是“数据进入判断或执行前的加工能力”。  
它既服务规则判断，也服务作业执行。

典型用途包括：

1. 字段值格式清洗。
2. 日期、金额、证件号等标准化处理。
3. 接口结果转换。
4. 页面值和接口值的统一比对口径。

产品上应优先提供常见内置预处理器，自定义脚本能力作为增强项而不是默认路径。

### 7.4 资源共享与权限管理（原则级）

平台面向的是多部门、多分行、多角色协同，因此资源共享和权限管理必须是基础能力。

需要重点解决：

1. 哪些页面、规则、接口、预处理器对哪些部门可见。
2. 哪些角色可以查看、编辑、发布、停用、回滚。
3. 跨部门资源是只读复用，还是允许复制后本地维护。
4. 被引用资源如何防误删、防失效。

在权限模型上，建议采用简化授权思路：

1. 角色按职责统一定义，不按具体岗位零散拆分。
2. 资源边界主要按组织范围控制。
3. 权限边界主要按操作类型控制。
4. 页面、规则、作业场景默认继承所属组织范围。
5. 不将单页面、单规则、单场景逐条细粒度授权作为常规模式。
6. 特殊跨组织使用场景优先通过共享或复制机制解决，而不是通过复杂 ACL 维持。

同时，平台应明确建设“角色管理模块”，并将其纳入业务管理体系。角色定义、授权边界、角色管理能力和待确认矩阵，建议独立沉淀到权限专题文档中。`spec/config-center-product/permission-model.md`。`prd-analysis.md` 仅保留以下原则级结论：

1. 角色管理模块由业务侧主导使用和维护。
2. 业务部门可在平台预置角色体系基础上管理角色与人员绑定关系。
3. 业务部门可配置角色的组织范围和可执行操作类型。
4. 角色管理、人员分配、权限调整都必须留痕，可审计、可追溯。
5. 平台支持角色只提供系统级支持，不主导日常角色管理。
6. 角色支持复制，用于快速复用既有角色配置；复制后形成独立角色，不影响原角色。
7. 角色支持停用和恢复，不支持直接删除已使用过的角色。
8. 停用角色后，不允许再新增人员绑定，已绑定人员同步失去该角色权限。
9. 人员绑定支持批量分配，用于业务部门快速完成角色授权。

## 8. 智能提示分析

### 8.1 智能提示的产品职责

智能提示层在总纲中的定位，是把“页面状态”转化为“业务提醒”，并与发布控制、审计体系闭环。

总体职责包括：

1. 页面识别与上下文理解。
2. 规则条件判断与命中控制。
3. 提示展示与关闭管理。
4. 触发行为留痕与审计追溯。

### 8.2A 规则条件配置体验

在总纲层面，规则条件配置只保留以下原则：

1. 以业务可理解性优先，默认采用结构化配置而非自由脚本。
2. 条件模型保持单层，统一全局 AND/OR 逻辑。
3. 条件来源收敛为页面字段、API字段、名单字段、上下文变量、固定值五类。
4. 条件编辑采用左值-逻辑运算-右值主流程，复杂能力受控开放。
5. 选择页面字段时，先选择“公共字段”或“页面特有字段”，再引用页面资源中已建档字段，不直接录入选择器。

### 8.2A.1 专题下沉指引（智能提示）

1. 智能提示的交互细节、样式规范、验收标准与设计标注清单，统一维护在 `spec/config-center-product/sentinel-prd.md`。
2. `prd-analysis.md` 仅保留总体原则、跨模块边界和产品级结论。
3. 设计与研发实施请以 `sentinel-prd.md` 为执行基线。

### 8.2B 条件校验边界

总纲层面校验边界为：

1. 平台承诺配置有效性校验与发布前校验。
2. 平台不承诺替代真实业务环境的联调与结果验证。

### 8.2 规则模型分析

总纲口径下，规则模型至少覆盖：

1. 规则基础信息与状态。
2. 条件集合与逻辑关系。
3. 提示配置与触发控制。
4. 生效范围、有效期与发布控制信息。
5. 是否联动作业与关联边界。

### 8.3 提示类型与展示策略

总纲口径下，提示展示策略为：

1. 当前仅支持静默提示与浮窗提示。
2. 多提示命中采用串行执行策略，不并发堆叠。
3. 关闭策略与去打扰策略需保持一致控制口径。

### 8.4 提示与业务语义

总纲层面强调：提示承载业务语义，不按“监管类/风险类”等类型做独立产品建模。

### 8.5 提示与作业入口关系

总纲层面强调：提示可以承载单一作业入口，但不演进为多作业分发中心；复杂场景由悬浮按钮或独立作业入口承接。

### 8.6 提示文案配置体验

总纲层面强调：文案配置以业务侧可快速维护为原则，结构化模板化能力按后续复杂度演进。

## 9. 智能作业分析

### 9.1 智能作业的产品职责

智能作业层在总纲中的定位，是把“知道该做什么”转化为“可执行动作”，并与提示和平台控制形成闭环。

总体职责包括：

1. 动作编排与执行。
2. 多来源数据处理与页面写入。
3. 失败兜底与手工回退。
4. 执行留痕与责任确认。

### 9.2 为什么智能作业是完整产品的一部分

总纲结论：智能作业与智能提示共同构成平台核心价值，二者在模型、平台控制、审计上必须协同设计，不能割裂建设。

### 9.3 智能作业的核心场景

总纲层面仅保留场景框架：查数预填、审核辅助、高频动作复用、失败后重试与二次触发。

### 9.4 场景编排模型

总纲层面强调：编排能力应兼顾可视化与可校验，面向业务可理解，不演进为无约束低代码引擎。

### 9.5 基础组件模型

总纲层面保留最小动作链组件框架：页面取值、接口调用、加工处理、页面写值。

### 9.6 执行模式分析

总纲层面保留执行方式框架：无提示自动执行、提示后自动执行、预览确认后执行、悬浮按钮手动触发。

### 9.7 悬浮按钮与二次触发

总纲层面强调：悬浮按钮是规则联动之外的第二触发入口；二次触发必须创建新执行实例。

### 9.8 智能作业的关键设计原则

1. 自动化强度可配置，但责任归属与审计留痕不可缺失。
2. 失败默认可感知、可回退、可追溯。
3. 能力建设优先覆盖高频、稳定、低歧义场景。

### 9.9 专题下沉指引（智能作业）

1. 智能作业的场景模型、编排交互、节点模型、执行模式、悬浮按钮规则与验收标准，统一维护在 `spec/config-center-product/job-prd.md`。
2. `prd-analysis.md` 仅保留平台级框架、边界与跨模块原则。
3. 设计与研发实施请以 `job-prd.md` 为执行基线。

## 10. 端到端产品闭环

从完整产品角度看，平台需要形成以下闭环：

1. 配置人员选择目标业务页面并完成页面建档。
2. 配置规则条件、提示内容和生效范围。
3. 若需要动作辅助，则配置联动作业或独立作业场景。
4. 发布后，插件进入目标页面并识别页面上下文。
5. 规则引擎基于页面当前基线状态与后续已观测字段变化执行判断，并按配置决定是否展示提示。
6. 若配置了作业动作，则按执行方式进入无提示自动执行、提示后自动执行、预览确认后执行或悬浮按钮手动触发流程。
7. 触发、执行、失败、取消、关闭等全过程留痕。
8. 管理人员可查看效果、停用规则、调整范围、回滚版本。

这条闭环成立，平台才真正具备“从配置到执行再到发布控制与审计”的完整产品能力。

## 11. 配置流程与交互要求

### 11.1 面向业务角色的配置流程

配置流程应尽量贴近业务人员的理解方式，而不是系统底层模型。

在流程设计上，应采用“分模块建设、主流程串联”的思路：

1. 规则和作业场景是两个独立模块，分别承担触发判断和动作编排职责。
2. 复杂作业场景允许在独立模块中单独开发、编排、调试和版本管理。
3. 规则创建流程中应支持直接选择已有作业场景，或跳转新建作业场景后自动回填绑定关系。
4. 规则创建页只承载轻量作业关联配置，不承载复杂节点编排、字段映射和脚本细节。
5. 规则创建页中的作业相关字段，建议仅保留：是否关联作业场景、已关联作业场景、执行方式、是否需要提示联动。
6. 节点编排细节、字段映射细节、脚本配置细节、预览结果模板等内容，统一放在独立作业模块中维护。
7. 这样既能保证业务主流程顺畅，又不会把复杂编排硬塞进规则页面。

对开阳等业务系统，建议采用向导式主流程，并围绕以下路径组织主要步骤：

1. 选择业务范围。
2. 选择专区。
3. 选择业务菜单。
4. 选择或补录页面。
5. 配置规则。
6. 配置提示。
7. 关联作业场景。
8. 执行基础校验。
9. 保存草稿或提交发布。

### 11.2 配置体验的关键要求

1. 上下文信息持续可见，不因步骤切换丢失。
2. 页面不存在时应能在当前流程中补录，而不是跳出流程。
3. 规则和作业的配置关系应清晰，避免双向重复配置。
4. 错误提示要结构化、可定位、可执行。
5. 发布前应进行完整性校验，避免线上才暴露问题。
6. 平台当前只提供基础校验，不承诺内置完整测试能力；涉及真实联调和结果验证，需在外部系统或实际业务环境中完成。

### 11.3 可视化与低门槛原则

产品交互应尽量遵循以下原则：

1. 多用表单、向导和结构化配置，少暴露脚本和技术术语。
2. 高频能力提供模板、默认值和示例。
3. 将复杂概念拆成业务可理解步骤，例如“页面选择”“规则判断”“作业动作”“结果确认”。
4. 将技术人员参与范围收敛到平台级运维、少量基础配置和异常支持，不把日常规则与作业配置建立在技术角色参与前提上。

## 12. 生命周期与发布控制

### 12.1 为什么平台控制是核心能力

在银行场景中，平台能不能上线只是第一步，能不能长期被安全使用、被合理收口、被有效审计更重要。

平台控制能力至少包括：

1. 草稿、发布、停用、过期、回滚等状态管理。
2. 高风险操作审计。
3. 配置变更版本记录。
4. 生效范围控制。
5. 状态变更原因记录。

### 12.2 临时规则控制

监管或过渡性提示有天然生命周期，平台必须支持：

1. 设置责任人。
2. 设置有效期。
3. 设置停用动作。
4. 记录延期或替代原因。

否则平台会快速积累大量失效规则，影响使用体验和管理成本。

### 12.3 发布与回滚

正式发布不应只停留在“保存即生效”，而应具备完整发布行为：

1. 发布前校验依赖资源完整性。
2. 发布时记录版本与影响范围。
3. 回滚时回到确定版本，而不是手工恢复。
4. 规则、页面、作业场景的生效必须基于一致版本视图。

### 12.4 状态模型与流转规则

规则和作业场景建议统一使用以下主状态：

1. 待发布。
2. 生效中。
3. 已停用。
4. 已失效。

建议的状态流转口径如下：

1. 新建对象默认进入待发布。
2. 待发布对象在完成发布后进入生效中。
3. 生效中对象可被业务手工停用，进入已停用。
4. 生效中对象到达失效时间，或因替代、资源异常等原因退出生效状态时，进入已失效。
5. 已停用对象如需再次启用，应重新进入发布流程，而不是直接绕过发布恢复线上状态。
6. 已失效对象不直接恢复，如需继续使用，应基于当前内容或历史版本重新生成新的待发布版本。
7. 所有生效中对象只要发生内容修改，都不直接改动线上版本，而是生成新的待发布版本；新版本发布成功后，再替代当前生效版本。

### 12.5 冲突前置校验与运行时防护

同一页面、同一字段的写入冲突，当前阶段采用运行时覆盖策略处理：后者覆盖前者；发布前仍需完成冲突识别与提示。

建议明确以下平台控制口径：

1. 发布前必须检查同页同字段是否被多个规则或作业场景重复写入，并输出冲突提示。
2. 冲突配置不再默认阻断发布，但必须完成覆盖策略确认并留痕。
3. 运行时同字段存在多个写入候选时，按实际执行顺序处理，后执行覆盖先执行。
4. 覆盖行为必须记录覆盖前值、覆盖后值、来源规则/场景与时间戳，确保可审计追溯。
5. 若因缓存、脏数据、灰度残留导致重复写入，仍按“后者覆盖前者”执行，同时记录冲突日志与防护日志。

### 12.6 发布前强校验清单

平台应将以下内容作为发布前的强校验项，未通过时不允许正式生效：

1. 缺少责任人，不允许发布。
2. 缺少适用范围，不允许发布。
3. 缺少生效时间或失效时间，不允许发布。
4. 同页同字段存在写入冲突时，必须完成覆盖策略确认与日志留痕配置。
5. 规则引用失效资源，不允许发布。
6. 作业场景缺少人工作业时长，不允许发布。
7. 自动执行类场景缺少业务风险责任确认，不允许发布。

### 12.7 自动执行风险确认

自动执行类场景的业务风险确认，应由业务管理角色在发布前完成。

建议明确以下口径：

1. 只有自动执行类场景需要强制风险确认。
2. 风险确认记录至少应包含确认人、确认时间、关联场景、执行方式和适用组织范围。
3. 场景执行方式从非自动改为自动时，必须重新确认。
4. 仅修改文案、说明等非执行关键字段时，不强制重新确认。
5. 修改执行链路、写入字段、触发方式等关键执行内容时，必须重新确认。

## 13. 审计与可观测性

### 13.1 需要记录什么

平台至少要记录三类日志：

1. **状态变更日志**：谁发布、停用、延期、回滚了什么配置。
2. **触发日志**：某条规则何时、在哪个页面、对谁命中。
3. **执行日志**：某个作业场景的每个节点输入、输出、耗时和结果。

### 13.2 为什么日志不能只是“留痕”

日志不仅用于事后审计，也用于日常运营与问题定位：

1. 页面识别失败时，能知道是页面编码问题还是 URL 未命中。
2. 规则未命中时，能知道是字段缺失、预处理失败还是接口异常。
3. 作业失败时，能知道失败节点、失败原因和影响范围。
4. 提示过多时，能分析是否存在规则冲突或触发控制失效。

### 13.3 看板与运营指标

平台应优先建设以下核心指标：

1. 作业执行成功率。
2. 平均节省时长。
3. 失败原因分布。
4. 已失效规则/场景数量。
5. 即将到期规则/场景数量。

其中，“平均节省时长”建议采用统一口径：

1. 创建作业场景时必须填写人工作业时长，作为基准值。
2. 人工作业时长建议以秒为基础单位存储。
3. 作业关键流程发生变化时，必须同步更新人工作业时长。
4. 平均节省时长 = 人工作业时长 - 实际成功完成时长。
5. 若计算结果小于 0，则按 0 处理，不记负收益。
6. 仅统计成功完成的作业执行，失败和取消执行不计入节省时长。

### 13.4 发布页默认视图与筛选设计

面向业务管理角色的发布页面，不应默认展示“全部配置”，而应默认进入“待处理”视图，帮助业务管理人员先聚焦需要处理的对象。

建议的发布页默认视图口径如下：

1. 首页默认进入待处理视图，而不是全部视图。
2. 主状态筛选统一包括：待发布、生效中、已停用、已失效。
3. 高频快捷筛选建议包括：待处理、即将到期、校验不通过、存在冲突、待风险确认。
4. 基础筛选维度优先保留组织范围、责任人、状态、生效时间、失效时间、执行方式、是否自动执行。
5. 不优先暴露过多技术型筛选条件，避免业务用户被低频复杂维度干扰。
6. 列表字段建议至少包含名称、类型、所属组织、责任人、执行方式、当前状态、生效时间、失效时间和最近变更时间。

## 14. 安全、合规与稳定性分析

### 14.1 安全边界

平台需要处理的安全问题主要包括：

1. 页面字段和接口数据可能涉及敏感信息。
2. 规则与作业能力可能改变用户操作行为。
3. 自定义脚本、接口代调和页面注入都具有潜在风险。

因此，产品设计上必须强调：

1. 敏感信息加密存储、脱敏展示。
2. 页面侧不暴露敏感配置和密钥。
3. 脚本能力受控使用并保留审计。
4. 自动动作可确认、可回退、可追溯。

### 14.2 稳定性要求

在实际业务环境中，平台常见的不稳定因素包括：

1. 页面结构变化。
2. 网络波动。
3. 接口超时。
4. 配置错误。
5. 浏览器兼容差异。

平台应以“持续可用、优先降级”为原则：

1. 能用缓存时继续受控运行。
2. 不能稳定执行时切回人工模式。
3. 错误信息要让用户知道下一步能做什么。

## 15. 关键产品风险

从完整产品角度，当前最值得关注的风险包括：

1. **产品边界被切碎**  
   如果提示、作业、平台控制分拆讨论，最终容易得到多个半成品而不是完整平台。
2. **页面资源维护成本高**  
   页面元素不稳会同时影响提示与作业，成为全局瓶颈。
3. **自动化收益与安全风险拉扯**  
   动作越自动，越要重视确认机制和审计。
4. **业务配置门槛仍然偏高**  
   如果配置流程仍像技术后台，业务角色难以真正独立使用。
5. **规则和场景长期失控**  
   没有责任人、有效期和停用机制，平台会演变成规则堆积系统。
6. **指标只看触发，不看结果**  
   如果只统计提示次数，不看执行成功、失败结构和真实效率收益，就无法评估平台价值。

## 16. 产品建议

### 16.1 已确认产品决策

基于当前产品讨论，以下口径已明确，可作为后续设计与落地的统一依据：

1. 智能作业的核心目标是提效，尤其要服务审批窗口极短、人工反应时间只有几秒的业务场景。
2. 智能作业开放给业务侧配置，业务管理人员对场景执行方式与业务风险承担管理责任。
3. 执行方式统一包括：无提示自动执行、提示后自动执行、预览确认后执行；悬浮按钮手动触发作为补充入口保留。
4. 平台支持“规则联动触发”和“悬浮按钮二次触发”两类主入口；二次触发必须新建实例，不复用旧上下文。
5. 规则和作业场景都必须绑定责任人、适用范围、有效期、发布记录和停用记录，缺少发布控制信息的配置不应正式生效。
6. 平台主用户与主管理者均为业务角色，技术人员只承担少量运维配置、平台支持和异常排查工作。
7. 平台增加角色管理模块，由业务侧管理角色、人员绑定和权限分配。
8. 权限模型采用“组织范围控制资源边界，操作类型控制权限边界”的简化思路，不以单条规则或单个页面细粒度授权作为常规模式。
9. 自动执行类场景的业务风险确认由业务管理角色在发布前完成；执行方式从非自动改为自动时，必须重新确认。
10. 同页同字段写入冲突采用运行时覆盖策略处理：后者覆盖前者；发布前必须完成冲突识别、覆盖策略确认与留痕配置。
11. 平台效果评估应优先关注作业执行成功率、平均节省时长、失败原因分布、已失效规则/场景数量和即将到期规则/场景数量。
12. 创建作业场景时必须填写人工作业时长，作为平均节省时长的计算基准。
13. 发布前必须执行强校验清单，责任人、范围、生效失效时间、冲突校验、人工作业时长和自动执行风险确认缺一不可。
14. 角色支持复制、停用、恢复和批量分配；已使用角色不做物理删除。
15. 面向业务管理角色的发布页面默认进入“待处理”视图，而不是“全部”视图。
16. 规则和作业场景统一采用“待发布、生效中、已停用、已失效”四类主状态；所有生效中对象的内容修改都必须生成新的待发布版本，不能直接改线上版本。
17. 平台只承诺基础校验能力，不承诺内置完整测试能力；涉及真实联调和结果验证，应在外部系统或实际业务环境中完成。
18. 提示展示当前仅支持静默提示和浮窗提示两种方式；关闭行为支持计时自动关闭、默认手工关闭、计时后手工关闭三种模式；提示关闭后当前规则在本次页面停留中不再继续执行后续行为。
19. 提示可按场景承载单一作业引导入口，但是否进入作业、何时进入作业以及是否需要预览确认，统一由作业场景执行方式配置决定；当执行方式需要提示承接时，入口形态统一为提示确认按钮，按钮文案固定为“确认”；复杂或多作业场景由悬浮按钮或独立作业入口承接。
20. 规则增加数字型优先级属性，并作为多规则同时命中时的提示展示顺序依据；优先级取值范围统一为 1-999，默认按数值越大优先级越高。
21. 规则仅保留名称字段并做常规校验，名称在平台内需全局唯一，不单独建设规则编码。
22. 作业场景仅保留名称字段并做常规校验，名称在平台内需全局唯一，不单独建设场景编码。
23. 业务管理角色不默认拥有配置权限；平台支持角色拥有查看、校验和审计查看权限，用于技术支持场景。
24. 规则条件编辑当前统一为单层模型，不开放多层条件组嵌套；所有条件共享一个全局 AND/OR 逻辑。
25. 条件组编辑采用左右布局：左侧按“左值-逻辑运算-右值”固定顺序编辑；右侧仅保留一个属性面板，选中左值或右值时切换显示其属性，不设置固定页签；预处理器采用“点击添加条目 + 参数输入”方式配置。
26. 作业场景中心不展示执行实例模块，不保留手工预览区；编排主界面采用左（节点库）-中（画布）-右（属性）布局。
27. 作业编排画布支持节点选中后右侧属性编辑与一键自动排版，以提升大场景可视性与编辑效率。
28. API注册兼容策略统一为“版本升级”：无论变更是否破坏兼容，均在原 API 对象上生成新待发布版本，不通过新建 API 对象承载变更。
29. API注册新建页面采用双栏布局：左侧基础信息（名称、描述、方法、测试环境路径、生产环境路径）；右侧参数配置（输入 4 Tab：Path/Query/Header/Body，输出支持手工添加与 JSON 解析，且对象类型可继续细化属性）。
30. API注册新建页面中 4 个输入 Tab 的字段与校验规则统一采用结构化配置：Path/Query/Header/Body 均支持参数级校验，Body 与出参均支持 JSON 解析与树形细化。
31. API注册新建页面补充字段字典与错误码规范：左侧基础信息、右侧参数字段、Body/出参树形规则、保存与发布分层校验均使用统一口径。
32. API注册列表页在接口保存后提供调试功能：支持参数输入、测试环境调试、结果与映射联动展示。
33. API注册列表页调试功能采用分区与分阶段流程规范：入口可见性、参数输入、请求组装、结果展示和失败提示均按统一交互口径实现。
34. 条件组编辑右侧仅保留单属性面板：选中左值或右值时切换显示其属性，不设置固定页签。
35. 条件组编辑左值/右值必须提供来源类型可视化样式；API来源统一展示为“API名称.输出值路径”，提升业务查看与排查效率。
36. 条件值来源样式需形成统一规范：固定值/页面元素/API/上下文变量采用一致标签与颜色语义，编辑态与摘要态均可读；API缺失输出值路径时阻断保存并告警提示。
37. 条件值来源样式需形成设计与前端共用字典（标签、图标、颜色 token、状态样式、验收标准），并下沉至 `sentinel-prd.md` 专项维护。
38. 条件值来源样式设计稿需提供可交付标注清单（字号、间距、圆角、图标尺寸、状态色值、交互时长、token 命名与状态矩阵），具体执行以 `sentinel-prd.md` 口径为准。
39. 条件来源统一收敛为页面字段、API字段、名单字段、上下文变量、固定值五类；页面字段内部再区分公共字段与页面特有字段。
40. 页面资源录入字段时必须选择元素类型，至少区分 `INPUT`、`SELECT`、`TEXTAREA`、`BUTTON`、`READONLY_TEXT`；字段定位统一采用 `locator` 模型，V1 可优先使用 `XPath`。
41. 页面字段取值结果至少区分 `ABSENT`、`EMPTY`、`VALUE` 三态；其中 `ABSENT` 表示元素不存在，条件结果记为 `INVALID`，规则层默认视为不命中。
42. JSSDK 运行时采用“基线采样 + 已观测终态变化”模型：SDK ready 后先采样当前终态，后续只对已观测到的字段终态变化触发规则检查，不补偿未观测到的历史变化。

### 16.2 产品层建议

基于完整性分析，建议在产品文档层面坚持以下原则：

1. 将营小助配置中心定义为“提示 + 作业 + 平台控制”的统一产品，而不是单模块产品。
2. 将页面资源、规则对象、作业场景和审计体系放在同一产品框架内建模。
3. 明确智能作业是平台核心能力之一，并在产品文档中做完整分析，而不是只做架构预留。
4. 明确平台主用户与主管理者均为业务角色，技术人员只承担少量运维配置与平台支持工作。
5. 将预览确认、失败降级、状态控制和日志追踪视为平台主能力，而不是补充功能。

## 16.4 产品文档拆分建议（智能提示 vs 智能作业）

结论：**有必要拆分**，并保留一份平台总文档作为总纲。

建议文档结构：
1. 总纲文档（平台级）：产品定位、统一术语、角色权限、共享模型、发布控制、审计口径。
2. 智能提示产品文档（子文档）：规则模型、触发链路、提示交互、关闭策略、提示控制。
3. 智能作业产品文档（子文档）：场景编排、节点模型、执行方式、预览确认、失败降级。

拆分收益：
1. 降低跨模块变更耦合，提升评审效率。
2. 避免提示与作业互相覆盖需求细节，减少口径冲突。
3. 支持不同节奏迭代（提示先行、作业增强）且不影响平台总口径稳定。

拆分原则：
1. 跨域规则只在总纲维护，子文档只引用不重复定义。
2. 共享对象（页面资源、接口、预处理器、角色）由总纲主责。
3. 子文档变更需回写总纲的“影响边界”与“版本记录”。
## 17. 最终结论

从产品完整性、业务闭环和平台控制与审计可控性三方面综合判断，营小助配置中心的合理产品形态应是一个统一的智能辅助平台：

1. 通过页面资源管理建立稳定的页面基础模型。
2. 通过智能提示把业务规则、风险约束和操作指引带到现场。
3. 通过智能作业把查数、预填、复核、注入等高频动作配置化落地。
4. 通过发布、停用、回滚、日志和审计形成完整控制与审计闭环。

也就是说，这个产品的完整价值不在于“能不能提示”，也不在于“能不能自动执行”，而在于它是否能把**业务知识、页面上下文、动作能力和平台控制机制**收敛到同一个平台中，并真正服务银行内网业务场景的准确性、效率和可控性。
