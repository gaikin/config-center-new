## 1. 文档目标

明确“分行上传一批 Excel 数据，并在规则或作业中做命中判断”这一能力在 MVP 阶段的产品定义、用户路径、复用边界、运行方式与平台口径。

本次重点回答七个问题：

1. 这个能力到底应该叫什么。
2. 用户从哪里进入，主路径怎么保持连续。
3. 规则里怎么用，且不破坏现有条件模型。
4. 作业编排里怎么复用，而不是另起一套。
5. 运行时如何兼顾本地规则判定、敏感信息保护和性能。
6. 名单维护、保存、发布、引用关系如何管理与发布控制。
7. MVP 阶段先做什么，不做什么。

## 2. 结论摘要

### 2.1 产品命名

MVP 阶段统一命名为：

- `名单数据`
- `名单检索`
- `名单检索节点`

不建议对业务用户主打“Excel 检索”“Excel 名单”或“Excel 数据源”。

原因：

1. 用户真正要完成的是“拿一批名单做命中判断”，不是“管理 Excel 技术细节”。
2. Excel 只是当前导入方式，不应成为主入口主文案。
3. 后续即使数据来源扩展，产品语义仍然稳定。

### 2.2 用户主路径结论

名单检索必须服从 `mvp-implementation-guide.md` 中的主路径原则：

1. 高频入口从 `菜单管理 -> 菜单详情 -> 新增提示规则 / 新增作业配置` 发起。
2. 规则侧把“名单检索”放进“改内容”阶段的 `高级条件`，不新增一级导航。
3. 作业侧通过新增 `名单检索` 节点承接，不新建“Excel 管理节点”。
4. 名单维护入口后置到 `高级配置 -> 名单数据`。
5. 规则 / 作业配置过程中如果缺少可用名单，只提供抽屉式 `新建名单` 快捷入口，不打断当前上下文。
6. 用户点击发布时，系统自动完成最终校验，不要求先走独立校验流程。

### 2.3 复用结论

名单数据必须作为平台统一的可管理、可发布资产建设。

规则与作业若都需要使用，只允许复用同一套：

1. 名单对象模型。
2. 版本与状态模型。
3. Excel 导入解析逻辑。
4. 匹配列配置逻辑。
5. 运行时检索服务。
6. 校验与发布门禁。
7. 引用关系与审计口径。

不允许分别做出两套：

1. 规则用的 Excel 检索。
2. 作业用的 Excel 检索。

### 2.4 运行时结论

结合 `architecture.md` 与 `jssdk-design.md`，MVP 阶段推荐如下运行口径：

1. 规则最终命中判断仍由 JSSDK 本地规则引擎承接。
2. 名单原始数据不直接下发到浏览器，不把完整名单快照暴露给页面侧。
3. 规则中的名单检索通过运行面受控检索接口获取“命中 / 未命中”结果，再由本地规则引擎完成后续条件组合判断。
4. 作业中的 `list_lookup` 节点复用同一运行面检索服务，不再单独访问 Excel 或私有表。
5. JSSDK 与作业执行链路都只缓存最小检索结果，不缓存完整名单内容。

### 2.5 MVP 实施原则

1. 名单检索在规则侧表现为一种受控的 `来源类型`（名单字段），不再额外拆独立条件卡片。
2. 主路径只讲“名单检索”，不讲“Excel 第几列”。
3. 高频配置前置，低频名单维护后置到高级配置。
4. 校验体验遵循三层：编辑态提醒、保存态拦截非法结构、发布态最终门禁。
5. MVP 只支持 `单文件导入 + 多检索字段精确匹配 + 下拉选择输出字段`。

## 3. 产品定义

### 3.1 名单数据对象

名单数据是一个可管理、可发布、可被规则与作业复用的公共对象。

它的本质是：

1. 一批业务键值集合。
2. 当前通过 Excel 单文件导入。
3. 由平台统一维护一组 `导入字段（importColumns）` 和一组 `输出字段（outputFields）`。
4. 可以被规则和作业同时引用，但不在规则或作业内重复存储数据。

“公共资产”的含义是：

1. 统一对象模型和发布控制方式。
2. 统一引用方式和校验口径。
3. 统一运行时检索服务。

它**不等于**：

1. 所有人都可编辑。
2. 所有机构都天然可见。
3. 可以绕开组织范围和发布控制。

权限、可见范围、共享方式仍沿用平台统一控制口径。

### 3.2 名单版本对象

名单数据需要有独立版本语义，和平台其他资源保持一致：

1. 名单主对象承载名称、归属机构、状态、当前生效版本等稳定信息。
2. 名单版本承载本次导入文件、表头、导入字段、输出字段、数据条数、解析结果、有效期等发布内容。
3. 生效中版本不直接修改；任何变更都生成新的待发布版本。
4. 发布成功后替换当前生效版本，不通过新建另一份名单对象承载同一名单的迭代。

### 3.2A 存储分层口径

名单数据采用“元数据入库、明细入 ES”的分层存储：

1. 名单基础信息、版本信息、发布状态、引用关系、解析摘要等元数据保存在 `TDSQL`。
2. 名单导入后的实际检索数据按版本写入 `Elasticsearch`。
3. 运行态检索统一查已发布版本对应的 ES 索引或别名，不查 TDSQL 明细表。
4. 浏览器侧与业务用户界面不感知 ES 索引名，只消费平台统一的名单对象语义。

这样做的原因是：

1. `TDSQL` 适合承载发布控制元数据、版本关系和发布事务。
2. `Elasticsearch` 更适合承载按一个或多个检索字段组合的高频精确检索。
3. 可以避免把大批量名单明细塞进事务型主库。

### 3.2B 导入、建索引与发布关系

名单导入后的最小处理链路建议为：

1. 用户上传 Excel。
2. 控制面完成文件解析、表头识别以及导入字段 / 输出字段校验。
3. 解析通过后，将当前版本的名单明细写入 ES 索引。
4. ES 建索引完成后，版本状态才允许进入可发布。
5. 发布成功后，将运行时可读别名切换到该版本索引。

这意味着：

1. “解析成功”不等于“已经可发布”。
2. 若 ES 建索引失败或未完成，名单版本不能正式发布。
3. 规则和作业永远只读取已发布别名，不直接读取草稿索引。

### 3.3 名单字段来源对象

规则里的名单检索，在当前 MVP 中作为一种新的受控来源类型出现，但仍服从统一操作数模型。

它至少包含以下信息：

1. `名单对象`
2. `检索键`
3. `输出字段`
4. `值类型 / 比较方式`
5. `检索说明摘要`

其中：

1. `检索键` 支持多条，每条由 `匹配字段 + 取值来源` 组成。
2. `取值来源` 继续复用规则统一来源模型：页面字段、API字段、上下文变量、固定值。
3. `名单对象` 选择已发布且当前可用的名单资产。
4. `输出字段` 必须从名单已配置的 `outputFields` 中下拉选择。
5. 当前规则原型中，多条检索键按 AND 语义组合，全部有值才发起检索。

### 3.4 名单数据与规则的关系

规则里使用名单数据时，本质上是在表达：

> 用当前页面字段、API 结果、上下文值或固定值，按一个或多个检索键去查某份名单；再取回指定输出字段，继续参与规则判断。

因此业务用户看到的应该是：

- `名单字段`
- `检索键`
- `输出字段`

而不是：

- `上传 Excel 文件`
- `读取第 2 列`
- `配置本地表`

### 3.4A 规则侧命中语义

规则侧名单字段的最小运行结果建议统一为：

1. `matched`：是否命中。
2. `outputValues`：按请求返回的输出字段值。
3. `listId / listName / listVersionId`：用于解释与审计。
4. `reasonCode`：用于表达未命中、输入缺失、结果歧义或服务异常。

在规则计算时：

1. 若未命中，则名单字段结果视为“空值/无结果”。
2. 若命中，则把所选输出字段值作为当前操作数的实际值。
3. 后续仍由规则引擎完成 `=`、`!=`、`IN` 等比较判断。

### 3.5 名单数据与作业的关系

若作业编排中也需要使用名单数据，本质上是在表达：

> 在执行链路里，根据某个输入值查名单，再决定后续写值、分支或风险标记。

因此作业侧应复用同一份名单资产与同一检索服务，而不是自建独立数据表或独立 Excel 节点。

### 3.6 状态与生命周期

名单数据生命周期与平台统一口径对齐：

1. `DRAFT`：草稿中，可编辑但未正式生效。
2. `ACTIVE`：已发布且当前可用。
3. `DISABLED`：被停用，不允许新引用，也不参与运行时检索。
4. `EXPIRED`：已过有效期，不允许正式生效。

补充约束：

1. 规则与作业只能引用 `ACTIVE` 且未过期的名单版本。
2. 名单若已被引用，发布新版本时必须做影响评估。
3. 生效中名单不允许被直接删除，只允许停用、失效或版本替换。

## 4. 用户路径与入口设计

### 4.1 菜单管理仍是业务主入口

结合 `mvp-implementation-guide.md` 与 `business-user-optimization.md`，业务主路径保持不变：

1. 用户先从 `菜单管理` 找到目标菜单与页面。
2. 在菜单详情里判断当前页面已开通什么能力。
3. 直接发起 `新增提示规则` 或 `新增作业配置`。

名单检索不能另起一条与主路径并行的业务入口。

### 4.2 规则主路径中的位置

规则页必须继续遵循“四步式”：

1. 选页面
2. 改内容
3. 预览
4. 保存

名单检索在这个流程中的落点是：

1. 放在第 2 步“改内容”的 `高级条件` 中。
2. 作为一种可选条件能力出现。
3. 不打断用户先配页面、提示内容、基础条件的主流程。

这样做的原因是：

1. 名单检索属于条件增强，不是规则主路径第一屏信息。
2. 大多数规则不会每次都用名单，低频项不应前置。
3. 这符合“小而美 MVP”里“高频先于低频、连续流程先于孤立页面”的原则。

### 4.3 规则页内快捷入口

当用户选择“名单检索”但当前没有可用名单时，提供快捷入口：

- `新建名单`

交互建议：

1. 使用右侧抽屉或轻弹窗完成最小新建。
2. 不离开当前规则编辑页面。
3. 创建成功后自动回填到当前名单字段配置。

抽屉里只保留最小必填信息：

1. 名单名称
2. 归属机构
3. 有效期
4. 导入文件
5. 导入字段预览
6. 输出字段

复杂控制信息仍留在名单详情页补充。

### 4.4 作业页入口

如果作业编排需要使用名单数据，入口原则如下：

1. 不新建“Excel 管理”入口。
2. 在作业节点库中新增 `名单检索` 节点。
3. 节点名称固定为 `名单检索`，而不是 `Excel 节点` 或 `名单表查询`。
4. 节点配置面板复用和规则侧一致的公共配置块。

### 4.5 维护入口

名单数据维护入口放在：

- `高级配置 -> 名单数据`

定位：

1. 低频维护区。
2. 由有权限的业务配置人员或管理角色维护。
3. 默认不放在业务主路径首屏。

### 4.6 发布与灰度中的承接

名单检索必须服从平台统一发布链路：

1. 用户在 `发布与灰度` 中发起发布。
2. 系统自动校验名单、规则、作业之间的依赖关系。
3. 若失败，直接给出结构化阻断项。
4. 若成功，直接展示受影响页面、规则、作业和机构范围。

不允许要求用户：

1. 先单独进入“名单校验”页面。
2. 再回来发布。

## 5. 规则侧设计

### 5.1 在规则里的呈现方式

名单检索建议做成新的受控来源类型：`名单字段`。

用户在规则页里可配置：

1. 选择名单
2. 配置一个或多个检索键
3. 下拉选择输出字段
4. 参与后续比较运算
5. 查看检索结果摘要

### 5.2 检索值来源口径

为保持和 `sentinel-prd.md` 一致，名单字段内部每个检索键的取值来源继续复用统一来源模型：

1. 页面字段
2. API字段
3. 上下文变量
4. 固定值

不单独发明：

1. Excel字段
2. 临时输入来源

### 5.3 推荐交互字段

规则页中的名单字段配置建议包含：

1. `名单名称`
2. `检索键`
   - 支持多条
   - 每条包含 `匹配字段 + 取值来源`
3. `输出字段`
   - 从名单已配置输出字段中下拉选择
4. `说明`
   - 显示归属机构、数据条数、更新时间、有效期、引用状态
5. `摘要态`
   - 显示当前命中的名单、已配置检索键数量和所选输出字段

### 5.4 推荐展示文案

业务用户最终应理解成：

- “如果按客户号 + 证件号检索到‘高风险客户名单’，且 `risk_level = HIGH`，则触发提示”
- “如果 `身份证白名单.whitelist_flag != Y`，则继续提示人工核验”

不应让用户理解成：

- “如果 Excel 文件第 2 列命中，则触发规则”
- “如果本地名单表查询成功，则命中”

### 5.5 条件摘要与预览

名单检索进入摘要态后，推荐统一展示为：

- `名单字段：高风险客户名单.risk_level（检索键 2 个）`
- `名单字段：身份证白名单.whitelist_flag（检索键 1 个）`

规则第 3 步“预览”里，名单检索至少展示：

1. 每个检索键当前来源及其示例值。
2. 本次用于检索的归一化值。
3. 命中 / 未命中结果。
4. 命中使用的名单名称、版本和所选输出字段值。
5. 若未能完成检索，给出明确原因。

预览页面只返回解释性结果，不展示原始名单内容。

### 5.6 输入值与匹配语义

MVP 阶段名单检索的匹配语义统一收敛为：

1. 只支持 `单值精确命中`。
2. 检索前统一转为字符串。
3. 默认去除首尾空白。
4. 保留中间空格、保留前导零，不做模糊匹配。
5. 不支持 contains、前缀匹配、正则匹配、多值拆分匹配。

输入值运行时语义与现有字段模型保持一致：

1. `ABSENT`：来源字段不存在或取值失败，条件结果记为 `INVALID`，规则默认不命中。
2. `EMPTY`：来源存在但值为空，条件结果按 `未命中` 处理。
3. `VALUE`：有实际值，进入正式检索。

### 5.7 规则侧运行时行为

规则运行时推荐采用以下链路：

1. JSSDK 先解析出检索值来源。
2. 若结果为 `VALUE`，向运行面受控检索接口发起名单检索请求。
3. 运行面返回最小结果：是否命中、名单名称、名单版本、原因码。
4. JSSDK 再把该结果并入本地规则条件计算。

这样做的原因是：

1. 不把原始名单数据下发到浏览器。
2. 仍保持“规则组合判断在本地完成”的总体架构原则。
3. 运行面可直接基于 ES 承接名单检索，同时复用缓存、鉴权、限流和审计能力。

## 6. 作业编排复用设计

### 6.1 总原则

若作业编排需要使用名单数据，优先复用，不重复建设。

复用范围包括：

1. 名单资产本身。
2. 名单选择器。
3. 匹配列展示逻辑。
4. 运行面检索服务。
5. 命中结果结构。
6. 校验与发布门禁。

### 6.2 推荐实现方式

MVP 推荐采用：

1. 规则先支持名单检索。
2. 作业侧同步支持新增 `list_lookup` 节点。
3. 节点配置面板直接复用规则里的“名单选择 + 匹配字段 + 输入来源”公共语义。
4. 节点执行时直接复用同一运行面名单检索服务。

### 6.3 节点配置口径

`list_lookup` 节点最小配置建议包含：

1. `名单名称`
2. `匹配字段`
3. `输入来源`
4. `命中结果变量名`
5. `未命中时继续 / 阻断` 的执行策略

其中：

1. 作业节点当前原型先支持选择一个匹配字段。
2. `命中结果变量名` 用于后续节点引用结构化输出。
3. 执行策略只控制流程承接，不改变命中口径。

### 6.4 作业侧命中结果口径

作业编排中使用名单检索时，MVP 只返回最小结果：

1. `matched`：是否命中
2. `listName`：命中名单名称
3. `resultKey`：供后续节点引用的运行时变量键

内部日志与执行审计可附带：

1. `listId`
2. `listVersionId`
3. `reasonCode`

MVP 阶段先不返回：

1. 整行多列字段对象
2. 多行结果集
3. 二次筛选结果
4. 自定义聚合结果

### 6.5 作业侧运行时行为

运行时建议按以下方式处理：

1. 若检索值为空，节点正常完成并返回 `matched=false`。
2. 若名单运行时不可用或检索服务异常，节点按失败处理，并允许场景回退手工路径。
3. 若命中结果需要参与后续分支或写值，只消费结构化结果，不直接消费 Excel 原始列数据。

### 6.6 作业侧典型场景

典型场景包括：

1. 命中高风险名单后，给页面某字段写入“高风险”。
2. 命中白名单后，跳过某些后续步骤。
3. 未命中特定名单时，追加人工复核标志位。

## 7. 名单数据维护页设计

### 7.1 页面定位

名单数据页是一个低频维护页，不是业务主路径首屏。

它回答四个问题：

1. 这份名单叫什么。
2. 现在是否可用。
3. 当前有哪些导入字段和输出字段。
4. 被哪些规则和作业引用。

### 7.2 列表页建议字段

名单列表建议展示：

1. 名单名称
2. 归属机构
3. 状态
4. 有效期
5. 数据条数
6. 导入字段
7. 输出字段数
8. 引用数
9. 最近更新时间

字段展示要求：

1. 机构显示中文名，不直接显示内部 ID。
2. 技术主键、对象存储路径、解析任务 ID 不进入主列表。
3. 版本号只在详情或发布/审计视图中查看，不作为主列表主文案。

### 7.3 详情 / 编辑页建议结构

建议按四块组织：

1. `基础信息`
   - 名单名称
   - 名单说明
   - 归属机构
   - 适用范围
   - 有效期
2. `导入与解析`
   - 导入文件
   - 表头预览
   - 数据条数
   - 导入字段预览（解析生成、只读）
   - 输出字段（由上传人从解析表头中选择）
   - ES 建索引状态
3. `引用关系`
   - 被哪些规则引用
   - 被哪些作业场景引用
4. `发布信息`
   - 当前状态
   - 当前生效版本
   - 最近发布时间

### 7.4 MVP 页面能力

建议支持：

1. 新建名单
2. 上传 Excel
3. 解析预览表头
4. 查看导入字段预览
5. 从解析表头中选择输出字段
6. 查看数据条数
7. 查看被哪些规则和作业引用
8. 保存草稿
9. 提交发布

MVP 阶段不建议支持：

1. 在线逐行编辑名单内容
2. 在页面里直接修改任意 Excel 单元格
3. 复杂导入日志工作台

### 7.5 快捷新建与完整维护的边界

规则 / 作业中的快捷新建只承接最小闭环：

1. 先建出一个可引用的名单草稿。
2. 保存成功后回填当前配置。

而完整维护仍放在名单详情页处理：

1. 引用关系排查
2. 历史版本查看
3. 生效状态与发布时间查看
4. 后续修改与重新发布

### 7.6 不建议暴露给业务用户的内容

主路径中不建议暴露：

1. Excel 解析规则细节
2. 内部数据表编码
3. 文件存储路径
4. 技术主键
5. ES 索引名 / 别名
6. 原始运行调用参数

## 8. 校验与发布控制

### 8.1 三层校验模型

名单检索必须遵循 `mvp-implementation-guide.md` 的三层校验模型：

1. 编辑态：尽量前移低成本问题。
2. 保存态：阻断非法结构。
3. 发布态：自动完成最终门禁。

### 8.2 编辑态提醒

当用户配置名单检索时，应即时展示：

1. 当前名单是否已发布可用。
2. 当前名单是否已过期或即将过期。
3. 当前选择的匹配字段是否存在。
4. 当前检索键来源是否完整。
5. 当前输出字段是否已正确选择。
6. 当前名单是否存在下游引用风险提示。

提醒要求：

1. 就地提示。
2. 尽量定位到字段或条件块。
3. 文案必须可执行。

### 8.3 名单保存校验

名单数据保存时至少校验：

1. 名单名称非空。
2. 归属机构完整。
3. 已上传文件。
4. 文件可解析。
5. 存在可选表头。
6. 至少完成一次表头解析并生成导入字段。
7. 至少配置一个输出字段。
8. 数据条数大于 0。

允许保存草稿，但不允许写入非法结构。

### 8.4 名单发布校验

名单数据发布时至少校验：

1. 当前名单已完成解析。
2. 已解析出的导入字段存在，且输出字段均从解析表头中选择。
3. 导入字段按当前索引规则可稳定建立检索能力。
4. 有效期完整。
5. 归属机构和适用范围完整。
6. ES 索引已构建完成且可读。
7. 若已被规则或作业引用，变更不会破坏现有匹配字段 / 输出字段语义。

### 8.5 规则侧保存 / 发布校验

规则引用名单数据时至少校验：

1. 已选择名单。
2. 名单已发布可用。
3. 至少存在一条完整检索键。
4. 输出字段已选择且属于该名单输出字段集合。
5. 若存在预览结果异常，需给出明确原因但不强行代替真实运行验证。

规则发布时额外校验：

1. 引用名单状态仍为 `ACTIVE`。
2. 名单有效期覆盖规则生效期。
3. 目标页面、字段、API 来源与名单引用关系未失效。

### 8.6 作业侧保存 / 发布校验

作业节点引用名单数据时至少校验：

1. 已选择名单。
2. 名单已发布可用。
3. 匹配字段存在。
4. 输入值来源完整。
5. 若节点依赖命中结果分支，后续分支配置完整。

### 8.7 发布任务中的影响摘要

当名单、规则或作业进入发布流程时，发布摘要至少展示：

1. 受影响页面数。
2. 受影响规则数。
3. 受影响作业场景数。
4. 受影响机构范围。
5. 即将到期名单数。

这样用户才能理解：

1. 这次改了什么。
2. 会影响哪里。
3. 什么时候正式生效。

## 9. 运行面与接口边界

### 9.1 控制面职责

控制面负责：

1. 名单对象与版本管理。
2. Excel 导入与解析预览。
3. 导入字段与输出字段维护。
4. ES 建索引任务编排与发布别名切换。
5. 引用关系查询。
6. 发布前强校验。

### 9.2 运行面职责

运行面负责：

1. 接受规则侧和作业侧的名单检索请求。
2. 基于已发布名单版本对应的 ES 索引 / 别名完成精确命中判断。
3. 返回最小命中结果，不返回原始名单内容。
4. 记录检索审计与失败原因。
5. 提供受控缓存与限流能力。

### 9.3 推荐接口边界

结合 `api-design.md`，推荐至少补齐以下接口：

控制面：

1. `POST /api/control/list-datas`
2. `GET /api/control/list-datas`
3. `GET /api/control/list-datas/{listId}`
4. `POST /api/control/list-datas/{listId}/versions`
5. `PUT /api/control/list-datas/{listId}/versions/{versionId}`
6. `POST /api/control/list-datas/{listId}/versions/{versionId}/parse-preview`
7. `POST /api/control/list-datas/{listId}/versions/{versionId}/build-index`
8. `GET /api/control/list-datas/{listId}/versions/{versionId}/index-status`
9. `GET /api/control/list-datas/{listId}/references`

运行面：

1. `POST /api/runtime/list-lookups/query`

### 9.4 运行面检索响应建议

运行面名单检索接口建议返回：

1. `matched`
2. `outputValues`
3. `listId`
4. `listName`
5. `listVersionId`
6. `reasonCode`
7. `ttlSeconds`

`reasonCode` 建议至少覆盖：

1. `MATCHED`
2. `NOT_MATCHED`
3. `INPUT_EMPTY`
4. `INPUT_ABSENT`
5. `LIST_UNAVAILABLE`
6. `LIST_VERSION_MISMATCH`
7. `LOOKUP_RESULT_AMBIGUOUS`
8. `LOOKUP_SERVICE_ERROR`

### 9.5 缓存与审计建议

为控制性能与留痕，建议：

1. JSSDK 按 `listVersionId + lookupValue` 做页面会话级短缓存。
2. 作业执行链路按执行实例缓存同一输入的重复检索结果。
3. 审计日志记录名单版本、输入来源、结果、耗时和失败原因。
4. 普通日志默认不落原始敏感值，必要时脱敏。

### 9.6 ES 索引模型建议

为保证版本隔离、发布可回退和运行时查询稳定，ES 建议采用“每个名单版本一个物理索引”的模型。

推荐命名方式：

1. 物理索引：`cc_list_data_{listId}_{versionId}`
2. 生效别名：`cc_list_data_{listId}_active`
3. 历史版本可读别名：`cc_list_data_{listId}_{versionId}_read`

其中：

1. 运行面默认只查 `active` 别名。
2. 控制面校验、回滚、对账可按版本别名或物理索引查看。
3. 不允许多个版本共写同一个物理索引。
4. 运行面收到 `listVersionId` 后，需校验 `active` 别名当前指向版本与请求版本一致；不一致时返回版本不匹配错误，而不是静默查旧索引。

MVP 阶段索引文档建议至少包含：

1. `listId`
2. `listVersionId`
3. `ownerOrgId`
4. `matchKeyPayload`
5. `rowHash`
6. `rowPayload`
7. `importBatchNo`

映射建议：

1. 每个 `importColumns` 中可参与检索的字段都建立 `keyword` 精确检索列，并保留 `raw` / `norm` 两种值。
2. `rowPayload` 可保留原始行 JSON，但运行面默认不返回。
3. `rowHash` 用于幂等导入、抽样校验和重建对账。

### 9.7 建索引、发布与别名切换

控制面一次完整的名单版本发布建议按以下步骤执行：

1. 解析 Excel 并完成表头、导入字段、输出字段和必要的数据质量校验。
2. 生成当前版本对应的物理索引名。
3. 将数据批量写入 ES。
4. 完成索引刷新与抽样校验。
5. 将版本 `indexBuildStatus` 标记为 `READY`。
6. 用户发起发布时，发布服务原子切换 `active` 别名到目标版本索引。
7. 别名切换成功后，更新 TDSQL 中的 `current_version_id`、状态和审计记录。

建议的建索引状态至少包括：

1. `PENDING`
2. `BUILDING`
3. `READY`
4. `FAILED`

补充原则：

1. 建索引失败不影响当前生效版本。
2. 新版本未发布前，即使索引已 `READY`，运行面也不能通过 `active` 别名读到。
3. 别名切换必须使用 ES 原子别名更新能力，不手工先删后加。

### 9.8 重建、回滚与补偿

名单版本的重建与回滚建议采用“新建索引、旧索引不覆写”的原则。

重建规则：

1. 同一版本如需重建，优先新建新的物理索引，不覆盖旧索引。
2. 重建完成后更新版本记录中的当前索引名。
3. 仅当新索引校验通过后，才允许进入发布或回滚链路。

回滚规则：

1. 回滚本质上是把 `active` 别名切回某个历史可用版本的索引。
2. 回滚前必须确认目标历史版本索引仍存在且可读。
3. 若历史索引已被清理，则先重建历史版本索引，再允许回滚。
4. 回滚动作必须生成新的状态变更记录与审计记录，不静默修改。

补偿规则：

1. 若别名切换成功但 TDSQL 状态更新失败，发布任务应进入补偿态。
2. 补偿优先恢复到上一生效别名，并记录异常。
3. 未完成补偿前，不允许继续发起下一次名单发布。

### 9.9 索引保留与清理策略

为兼顾回滚能力与存储成本，建议：

1. 当前 `active` 版本索引永久保留。
2. 最近一个历史生效版本索引保留，便于快速回滚。
3. 仍被规则或作业引用的历史版本索引不得清理。
4. 草稿失败索引、未引用且超过保留期的历史索引可异步清理。

建议保留策略至少记录：

1. `lastIndexedAt`
2. `lastPublishedAt`
3. `lastReferencedAt`
4. `gcEligibleAt`

清理前校验：

1. 不是当前 `active` 别名指向索引。
2. 不是最近回滚保护版本。
3. 当前不存在规则、作业或发布任务引用。
4. 对应原始导入文件或可重建数据源仍可追溯。

## 10. MVP 范围

### 10.1 In Scope

1. 名单数据对象。
2. Excel 单文件导入。
3. 多检索字段精确匹配。
4. 规则页以“名单字段”来源引用名单，并支持多检索键与输出字段下拉选择。
5. 规则侧预览展示命中解释结果。
6. 名单数据发布控制。
7. 引用关系展示。
8. 作业编排新增 `名单检索` 节点并复用同一名单资产。
9. 运行面受控名单检索接口。

### 10.2 Out of Scope

1. 多表关联。
2. SQL 式筛选。
3. 模糊匹配。
4. 范围匹配与排序检索。
5. 返回整行对象给规则自由使用。
6. 大规模实时同步。
7. 复杂版本比对工具。
8. 多来源混合名单。
9. 页面内逐行编辑 Excel 数据。

## 11. 最终实施原则

这项能力的核心不是“支持 Excel”，而是：

> 让业务用户能把一批名单数据稳定地接入规则与作业判断中，并且只理解一次。

因此后续实现必须坚持以下原则：

1. 名单数据是公共资产，不是规则私有配置。
2. 规则页展示“名单检索”，不展示“Excel 技术细节”。
3. 作业若要使用，优先复用同一份名单能力，不另起一套。
4. 主路径不被低频维护动作打断。
5. 所有正式生效的名单对象都经过发布门禁。
6. 运行时不把原始名单直接暴露到浏览器。
