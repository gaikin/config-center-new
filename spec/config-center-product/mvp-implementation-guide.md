## 0. 执行摘要

这份文档只回答一个问题：

> 从用户视角看，当前前端原型的 MVP 应该怎么做，才能做到“小而美”。

先明确一个产品背景判断：

1. 当前同类产品竞争激烈。
2. 前端界面就是用户直接感知产品价值的地方。
3. 用户没有太多试错成本。
4. 一旦前端不好理解、不好操作、不好完成任务，用户通常不会耐心学习，而是直接放弃。

这里的“小而美”，不是功能少，而是：

1. 用户一进来知道先做什么。
2. 用户在主路径里少看到技术术语、内部编码和低频复杂项。
3. 用户能用一条连续路径完成“配置 -> 保存 -> 发布 -> 看结果”。

因此，这份规范**只约束前端原型表现层**：

1. 页面怎么组织。
2. 字段怎么展示。
3. 用户流程怎么连续。
4. 校验怎么前移、怎么提示。
5. 哪些技术信息要藏起来，哪些要放详情。

这份规范**不讨论**：

1. 后端真实接口如何实现。
2. 真实目录中心如何接入。
3. 真实生产联调怎么做。
4. 运行时架构怎么落地。

## 2. 复盘结论

从用户视角看，当前问题可以收敛成四类。

### 2.1 主路径不够顺

用户最核心的问题不是“不会点按钮”，而是：

1. 不知道从哪里开始。
2. 同一件事被拆在多个页面里。
3. 配完之后不知道下一步去哪。

结论：

1. 前端原型必须围绕“业务任务”组织，而不是围绕“系统对象”组织。
2. `菜单管理` 必须成为业务主入口。

### 2.2 技术字段暴露过多

用户不关心：

1. 内部编码。
2. 版本号。
3. 规则集编码。
4. 槽位编码。
5. `frameCode`、`pageCode` 这类实现标识。

结论：

1. 原型中这些信息可以存在，但不应该让用户手填。
2. 只有在详情或技术视图里才允许查看。

### 2.3 机构 / 人员展示不统一

用户不应该直接面对一串内部 ID 或编码值。

结论：

1. 前端原型必须统一做到“内部保留稳定值、界面展示中文名”。
2. 机构和人员都要通过公共组件与公共映射层处理。

### 2.4 校验流程割裂

用户会觉得流程割裂，通常因为：

1. 编辑时提示不够。
2. 保存时反馈不清楚。
3. 发布前还要再走一遍独立校验。

结论：

1. 前端原型必须把校验体验做成一条连续流程。
2. 用户可以理解“发布前还有最终门禁”，但不能接受“必须先单独点去校验页面”。

### 2.5 更深层次的问题模型

继续往下看，这些表层问题背后其实不是零散交互缺陷，而是前端原型存在六个更深层的问题。

#### 问题 1：前端在展示“系统结构”，而不是在承接“用户任务”

从多份文档可以看到，当前很多页面仍然优先按照对象拆分：

1. 页面资源
2. 启用策略
3. 规则
4. 作业
5. 发布

这说明前端更像是在把系统内部结构平铺给用户，而不是帮用户完成任务。

更深层结论：

1. 当前前端缺的不是更多页面，而是一个稳定的“任务视角容器”。
2. 如果不先解决这个问题，后面继续加字段、加提示、加按钮，只会让页面更满，不会让任务更顺。

#### 问题 2：前端缺少“展示层翻译”，把领域模型直接端给了用户

很多问题表面上是：

1. 编码难懂
2. 版本不该填
3. 机构显示内部值
4. 人员没有中文名

但本质上是前端没有建立稳定的“展示层翻译规则”。

更深层结论：

1. 前端不能直接把领域模型字段原样暴露给用户。
2. 前端必须承担“把内部模型翻译成业务可理解界面”的职责。
3. 如果没有这层翻译，任何新对象一接进来，都会重复出现 ID、编码、版本、技术词泄漏问题。

#### 问题 3：用户并不是不懂功能，而是不信任自己是否配对了

从 `business-user-optimization.md`、`TODO/frontend.md`、`TODO/sentinel-validation-todo.md` 可以看到，用户反复卡住的关键并不是“不会配置”，而是：

1. 不确定自己是不是配对了
2. 不确定现在保存后是否可用
3. 不确定发布后会影响哪里

更深层结论：

1. 当前前端的核心短板不是“操作能力不够”，而是“结果确定性不够”。
2. 用户一旦不能快速建立信心，就会频繁退回、重复确认，最后放弃。
3. 所以前端原型必须优先强化“可理解反馈”，而不是继续堆配置能力。

#### 问题 4：页面之间缺少连续承接，导致用户的上下文不断丢失

现在很多信息虽然存在，但不连续：

1. 页面选好了，下一步入口不明显
2. 规则配完了，发布承接弱
3. 发布做完了，结果查看没有自然闭环

更深层结论：

1. 当前问题不是单页不好，而是页面之间没有形成链路。
2. 一个好的前端原型，不只是“每一页都能看”，而是“上一页做完的事，下一页能接住”。

#### 问题 5：同一类问题在不同页面反复出现，说明缺的不是修页面，而是修规则

从 `ui-improve.md` 和本轮代码排查可以看到，类似问题反复出现：

1. ID 可编辑
2. 统计值可编辑
3. 内部编码暴露
4. 中文名映射缺失
5. 关闭无脏数据提醒
6. 反馈只靠 toast

更深层结论：

1. 这不是单页实现疏漏，而是前端缺少统一约束。
2. 如果不形成统一规范、共享组件和共用字典，同类问题会不断回流。

#### 问题 6：原型正在承受“产品竞争压力”，不能再按后台思维做

结合你补充的背景，可以明确一点：

1. 用户不是被动接受系统培训后再来用。
2. 用户会直接用界面判断产品是否值得继续使用。

更深层结论：

1. 当前前端原型不能再按“内部后台系统”心态设计。
2. 必须按“用户第一次上手就要成功”的标准设计。
3. 这意味着前端优先级要从“对象完整性”转向“任务完成率”。

### 2.6 Git 演进记录透露出的更深层问题

从 Git 历史继续往回看，还能看到一些仅靠当前页面现状不容易发现的问题。

#### 信号 1：导航结构经历了“技术导向 -> 对象导向 -> 用户任务导向”的连续摆动

从历史提交可以看到，导航大致经历了三轮明显变化：

1. 最早期更像技术能力导航，例如配置向导、模板中心、菜单范围、运行时配置包。
2. 中间阶段变成对象中心导航，例如页面资源中心、规则中心、作业场景中心、接口定义中心、旧发布工作台。
3. 最近才逐步收敛成用户任务导航，例如我的工作台、菜单管理、智能提示、智能作业、API注册、发布与灰度、运行统计。

这说明一开始前端并没有稳定地站在用户任务视角上，而是在不断寻找“系统应该怎么组织”的答案。

更深层结论：

1. 信息架构不是一开始就稳定的。
2. 过去前端更容易受系统实现结构影响，而不是受用户任务影响。
3. 这也是为什么很多页面虽然单看功能齐全，但整体使用仍然不顺。

#### 信号 2：文档对齐提交频繁出现，说明“前端在做，口径也在变”

Git 里连续出现多次文档对齐与计划重写类提交，说明在一段时间内：

1. 业务侧体验口径还在调整。
2. 页面定位和实施边界还在移动。
3. 文档与实现是在互相追赶，而不是单向稳定落地。

更深层结论：

1. 用户看到的前端变化，其实承载了“产品认知仍在收敛”的过程。
2. 一旦用户术语、导航、页面职责没有先冻结，前端就容易反复返工。
3. 未来做 MVP 时，必须先冻结用户视角下的主路径和主文案，再进入大规模页面改造。

#### 信号 3：大提交频繁重构页面结构，说明页面边界最初并不稳定

历史里多次出现大范围页面重构、拆分、迁移和改名。

这反映出：

1. 前端最初更像快速堆原型。
2. 高变化区域缺少稳定的页面边界和共享抽象。
3. 同一批概念在多个页面里反复重写。

更深层结论：

1. 这不是单纯的“代码整理不够”，而是前端早期没有先定义清楚“哪个页面负责哪个用户任务”。
2. 一旦页面职责不稳，后续任何体验优化都会变贵。
3. MVP 阶段必须优先稳定页面职责，而不是继续叠页面。

#### 信号 4：复杂交互往往是先实现、后补规范

从条件编辑区独立 UI 规范、提示/作业细节拆分文档等历史能看出：

1. 很多复杂交互是先在页面中长出来。
2. 真正发现难用之后，才开始补专项规范。

更深层结论：

1. 高复杂度交互区不能只靠页面内边做边调。
2. 像条件编辑、参数映射、作业编排这类区域，必须先有小规格说明，再写页面。
3. 否则前端会不断陷入“局部可用，但整体难维护”的循环。

#### 信号 5：简化类提交反复出现，说明复杂度会自然回流

Git 里不止一次出现“统一术语”“精简提示”“主路径重构”“简化复用交互”这类提交。

这意味着：

1. 前端复杂度不是一次清理就结束。
2. 如果没有明确红线，页面会自然重新长出技术词、低频字段和额外步骤。

更深层结论：

1. “小而美”不是一次性改造结果，而是一种持续约束。
2. MVP 不仅要定义目标页面，还要定义哪些内容以后也不能重新长回来。

#### 信号 6：原型速度很快，但也带来了“可回看性变差”

从一些大批量功能提交和兜底性提交可以看出，原型推进速度很快，但代价是：

1. 单次提交承载的信息过多。
2. 一个提交里同时混入结构调整、文案调整、交互调整和功能补齐。
3. 复盘时不容易看清“哪一次变化真正提升了用户体验”。

更深层结论：

1. 这会削弱团队从历史中学习的能力。
2. 对前端原型来说，后续更适合按“用户价值切片”推进，而不是按“大包提交”推进。
3. 否则产品在快速迭代，但复盘无法沉淀成稳定方法论。

## 3. MVP 目标

前端原型 MVP 只追求以下目标：

### 3.1 先解决“不要让用户放弃”

从竞争视角看，前端原型首先要做到的，不是“展示系统能力很多”，而是：

1. 不让用户在第一屏就困惑。
2. 不让用户在关键路径上频繁卡住。
3. 不让用户因为术语、字段和流程复杂而放弃继续使用。

因此，MVP 的第一优先级不是“功能数量”，而是“首次使用成功率”和“关键路径完成率”。

### 3.2 用户能快速开始

用户进入系统后，能很快知道：

1. 今天要处理什么。
2. 去哪个页面处理。
3. 页面里下一步是什么。

### 3.3 用户能少做无意义输入

用户在原型中不应再手工维护：

1. 系统自动生成的 ID / 编码。
2. 系统自动维护的版本。
3. 业务不关心的技术标识。

### 3.4 用户能完成连续闭环

前端原型至少要支持用户完成以下闭环：

1. 找到页面。
2. 配置提示 / 作业 / API。
3. 保存草稿。
4. 发起发布。
5. 看到发布结果。
6. 查看结果反馈。

## 4. 前端原型实施原则

## 4.1 用户先于对象

原型中所有一级入口和主页面，都优先回答：

1. 用户现在要做什么。
2. 用户下一步要去哪。

而不是优先回答：

1. 这是哪个底层对象。
2. 这是哪个技术模块。

这条原则背后的原因是：

1. 用户不是来学习系统对象模型的。
2. 用户是来完成业务任务的。
3. 当前竞争环境下，只要前几步不顺，用户就可能直接流失。

## 4.2 高频先于低频

前端主导航和主页面只能优先放高频动作：

1. 找页面
2. 配规则
3. 配作业
4. 注册 API
5. 发布
6. 看结果

低频复杂能力必须后置到：

1. 菜单详情
2. 高级区
3. 高级配置

## 4.3 业务语义先于技术语义

前端文案优先展示业务语义：

1. 页面
2. 规则
3. 机构
4. 预热策略
5. 启用范围
6. 运行提醒

而不是优先展示：

1. 编码
2. 版本槽位
3. loader/core
4. frameCode
5. 技术主键

### 4.3.1 业务展示术语白名单

为避免平台层概念泄漏到业务主路径，前端原型必须优先使用以下业务展示术语：

1. `发布与灰度`
2. `权限管理人员`
3. `配置人员`
4. `菜单启用`
5. `正式版本 / 试点版本`
6. `启用范围`
7. `正式发布`
8. `关联配置项`
9. `权限管理`

执行要求：

1. 新增页面、弹窗、卡片、表格标题时，必须优先从本白名单中选词。
2. 如果某个文案无法直接落入业务术语，默认说明该文案仍偏平台视角，需要先改词再落页面。
3. 用户可见文案中不得保留历史旧词或平台内部口径。

## 4.4 连续流程先于孤立页面

每个原型页面都应当成为流程中的一个节点，而不是一个孤岛。

例如：

1. 菜单详情里要能直接进入“新增提示规则”。
2. 保存后要能承接到发布。
3. 发布后要能承接到结果查看。

### 4.4.1 主路径实现三原则（跨页面强制）

结合本轮改进文档复盘与 Git 演进记录，前端原型实现必须额外冻结以下三条跨页面原则：

1. 先定义用户动作，再定义对象模型：
   - 页面先回答“用户现在要做什么”
   - 不能先让用户理解字段、绑定、规则集、编码、节点等内部对象
2. 先展示判断信息，再进入编辑动作：
   - 菜单详情优先承担“判断当前状态、决定下一步”的职责
   - 复杂编辑默认收敛到弹出层、抽屉或次级区域
3. 默认路径优先一步完成，低频治理后置：
   - 高频新增动作默认一屏或一次提交完成
   - 低频治理、公共能力、批量维护统一后置到 `高级配置` 或独立治理页

如果某个页面方案同时违反以上三条中的任意两条，则默认判定该方案不符合“小而美 MVP”。

## 4.5 条件显示优先于固定展示（通用规范）

条件显示是 MVP 全页面通用规范，不只适用于菜单管理。

1. 只展示当前可执行动作：
   - 可执行动作显示可点击按钮
   - 不可执行动作默认隐藏，不用“先展示再教育”
2. 阻断状态必须明确可见：
   - 未开通、审批中、权限不足等阻断状态必须有显式提示
   - 阻断提示必须给出当前唯一可执行动作（如申请开通）或清晰状态（如开通中）
3. 正向提示默认降噪：
   - “已满足”“可配置”等成功提示默认不展示
   - 仅在用户刚完成关键动作时短暂反馈
4. 导航与切换控件按需显示：
   - 仅在存在多个可切换对象时展示切换控件
   - 单对象场景不展示无意义切换器
5. 状态命名必须稳定：
   - 禁止“建议下一步”“待补齐”“缺口”等过渡期命名
   - 统一使用稳定状态词，如“已开通”“待开通”“开通中”
6. 页面级规则必须引用本条：
   - 各页面可补充细则，但不得与本条冲突
   - 发生冲突时，以本条通用规范优先

## 4.6 前端 MVP 红线

基于本轮深层复盘，前端原型必须遵守以下红线。

### 红线 1：不能让用户在主路径里手填系统字段

凡是系统自动维护字段，都不得在主路径里要求用户输入。

### 红线 2：不能把内部实现词当成主信息展示给用户

凡是 `编码`、`版本槽位`、`frameCode`、内部 ID 等技术信息，都不能成为主路径主文案。

### 红线 3：不能让用户跨多个一级入口拼装一件事

一个高频任务如果必须跨多个页面来回跳转才能完成，就说明前端主路径设计失败。

### 红线 4：不能只告诉用户“失败了”，却不告诉用户“为什么”和“改哪里”

错误提示必须可定位、可理解、可执行。

### 红线 5：不能让用户在发布前额外学习一条独立校验流程

最终门禁可以保留，但必须在原有主流程里自动承接。

### 红线 6：不能在不同页面重复发明同一类交互

机构、人员、状态、错误、阻断、确认、详情展示等相似问题，必须统一组件和统一规则。

### 红线 7：不能在主路径未冻结前，先做大面积页面重构

如果用户主路径、主术语、主页面职责还没冻结，就不应先进入大面积实现铺开。

### 红线 8：不能让高复杂度交互区只靠页面内临场发挥

条件编辑、参数映射、作业编排等高复杂度区域，必须先有小规格说明，再进入页面实现。

---

## 5. 页面级实施规范

## 5.1 我的工作台

用户视角下，工作台不是平台总览，而是“今天要做什么”。

前端原型必须展示：

1. 我的待处理
2. 我最近改过
3. 运行提醒
4. 常用入口
5. 业务看板

前端原型不应展示：

1. 大量平台建设状态
2. 技术运行细节
3. 对业务没有直接行动价值的统计

## 5.2 菜单管理

菜单管理是用户主入口。

用户在这里应该能完成三件事：

1. 找到菜单
2. 在菜单下定位页面
3. 直接发起提示或作业配置动作

菜单详情在 MVP 中的职责必须收敛为：

1. 帮用户快速判断“这是不是我要找的页面”
2. 帮用户判断“当前能做什么、不能做什么”
3. 把复杂编辑动作分发到次级入口

菜单详情不是二级配置中心，不承担完整字段治理、完整对象维护、批量技术配置等低频复杂任务。

前端原型必须支持：

1. 按专区 / 菜单名称 / 页面名称 / 机构 / 状态筛选
2. 菜单卡片列表 + 菜单详情（单页展示，可切换）结构
3. 菜单详情中承接当前页面动作：
   - 新增提示规则
   - 新增作业配置
   - 元素映射（次级动作）

菜单详情中承接的当前页面信息必须优先展示：

1. 页面名称
2. 所属专区 / 菜单
3. 页面状态
4. 提示规则数量 / 作业场景数量
5. 最近运行情况

当前页面信息块中的字段信息展示必须收敛：

1. 只展示 `已绑定字段数`
2. 不展示完整字段表、完整绑定表、元素技术细节
3. 复杂维护动作统一进入右侧弹出层

菜单管理文案必须以菜单为业务主语，不得强调“页面复用”。

`API注册` 视为公共资源入口，不作为菜单管理中的绑定叙事。

`发布与灰度` 不作为菜单管理主动作入口，菜单管理只承接“新增提示 / 新增作业”主路径。

### 5.2.1 条件显示与命名规范（菜单管理，MVP强制）

菜单管理必须遵循 `4.5 条件显示优先于固定展示（通用规范）`，并补充以下场景化细则。

1. 菜单列表动作区条件显示：
   - 菜单状态为 `已开通`：显示 `新增提示规则`、`新增智能作业`
   - 菜单状态为 `待开通`：仅显示 `申请菜单开通`
   - 菜单状态为 `开通中`：仅显示 `开通中`（禁用态）
2. 菜单详情中的当前页面动作区条件显示：
   - 页面所在菜单 `已开通`：显示 `新增提示规则`、`新增作业配置`
   - 页面所在菜单 `待开通`：不显示新增动作，仅显示 `申请菜单开通`
   - 页面所在菜单 `开通中`：不显示新增动作，仅显示 `开通中`
3. 信息块条件显示：
   - 阻断提示（待开通/开通中）必须显示
   - 正向成功提示默认不显示（减少噪音）
   - “当前页面已配置规则”仅在规则数 > 0 时显示
   - 页面切换控件仅在菜单下页面数 > 1 时显示
4. MVP 命名禁令：
   - 禁止使用过渡期命名：`建议下一步`、`配置进度`、`可直接配置`、`需申请开通`
   - 禁止使用过渡期措辞：`建议优先`、`待补齐`、`缺口`、`必须完成`
5. 菜单管理标准文案（必须统一）：
   - 状态筛选：`全部状态`、`能力已开通`、`待开通`、`开通中`
   - 左侧列表：`菜单名称`、`所属专区`、`配置概览`
   - 阻断提示：`菜单能力尚未开通`、`菜单能力开通中`
   - 动作文案：`新增提示规则`、`新增智能作业`、`新增作业配置`、`元素映射`、`申请菜单开通`、`开通中`
6. 存量文案清洗清单（本次已落地）：
   - `建议下一步` -> 删除该列
   - `配置进度` -> `配置概览`
   - `可直接配置` -> `能力已开通`
   - `需申请开通` -> `待开通`
   - `申请处理中` -> `开通中`

### 5.2.2 页面字段与元素映射规范（菜单管理，MVP强制）

页面字段相关能力在 MVP 中必须遵循“元素优先、一步完成、治理后置”。

1. 菜单详情中的当前页面主动作名称统一为 `元素映射`，不得再使用 `字段维护`、`生成字段` 等对象导向命名。
2. `元素映射` 打开后，用户先录入：
   - 元素名
   - 选择器类型
   - 元素选择器
3. 系统默认自动使用元素名生成页面字段名，并在提交时自动完成：
   - 创建页面元素
   - 创建页面字段
   - 创建字段绑定
4. 只有当用户勾选 `使用公共字段` 时，才显示公共字段下拉；未勾选时，不显示字段选择控件。
5. 页面字段新增流程不得拆成“先新增字段，再新增绑定”两次主操作；新增默认必须一步完成。
6. 公共字段不进入页面主路径集中治理：
   - 页面内只允许“引用已有公共字段”
   - 公共字段治理页放在 `高级配置`
7. 页面字段映射界面中，必须优先让用户感知“元素映射”动作，不得让用户先理解字段、绑定、元素三个后台对象。
8. 菜单详情中不再单独提供“字段列表维护”入口：
   - 用户在当前页面信息块中只看到 `已绑定字段数`
   - 真正的新增与调整统一通过 `元素映射` 完成
9. 当用户勾选 `使用公共字段` 时，界面只补充一个“公共字段选择”动作：
   - 不额外引入新的配置步骤
   - 不在当前页面打开公共字段治理流程
10. `元素映射` 文案必须始终保持用户任务视角：
   - 优先使用 `元素名`、`元素选择器`、`使用公共字段`
   - 禁止回退为 `字段编码`、`绑定关系`、`页面字段定义` 等后台口径

## 5.3 智能提示

用户关心的是：

1. 这个页面什么时候提示。
2. 提示什么内容。
3. 保存前能不能先看一下效果。

前端原型必须采用“主路径四步式”：

1. 选页面
2. 改内容
3. 预览
4. 保存

模板在原型中的定位应是：

1. 快捷复用来源
2. 不是主学习入口

前端原型必须做到：

1. 高频配置项前置
2. 复杂条件放高级区
3. 保存前有摘要
4. 预览结果清楚

## 5.4 智能作业

用户关心的是：

1. 要不要自动执行
2. 要不要先预览确认
3. 出问题会不会影响业务

前端原型必须：

1. 优先从页面进入作业配置
2. 用场景化方式表达作业能力
3. 把复杂编排放到后置区域

前端原型里，作业页面不应该一上来就让用户面对：

1. 大量节点细节
2. 底层执行参数
3. 风险实现字段

本轮口径同步：

1. 自动执行不再要求独立风险确认动作。
2. 自动执行默认直接执行，预览确认属于特殊行为，按需开启。
3. 列表页默认不展示“自动执行 / 直接执行”文案，只展示特殊行为标签（预览确认、按钮触发、补充按钮）。

## 5.5 API注册

用户对 API 页的核心诉求是：

1. 知道这个 API 是干什么的
2. 能快速填完必要信息
3. 能先测一下再保存

前端原型必须使用五步式：

1. 选用途
2. 填基础信息
3. 填参数示例
4. 在线测试
5. 保存确认

原型必须避免：

1. 首屏堆满技术字段
2. 让用户直接维护系统编号
3. 把“在线测试”误做成“唯一校验手段”

## 5.6 发布与灰度

用户最关心的是：

1. 这次改了什么
2. 会影响哪些页面和机构
3. 什么时候生效

前端原型必须展示：

1. 待发布内容
2. 生效范围
3. 试点范围
4. 发布记录

前端原型中的发布流程必须尽量连续：

1. 用户点击发布
2. 系统自动做最终校验
3. 若失败，直接展示结构化阻断项
4. 若成功，直接展示结果

前端原型不应强迫用户：

1. 先单独去点“校验”
2. 再回来点“发布”

## 5.7 运行统计

从用户视角，运行统计只需要回答两个问题：

1. 有没有在正常触发
2. 有没有明显下降

因此，前端原型只保留两个核心视图：

1. 触发统计
2. 下降提醒

前端原型不应把技术排障面板直接暴露给业务用户。

## 5.8 高级配置

高级配置不是主路径入口，只是“需要时找得到”的收口区。

前端原型中，高级配置要做到：

1. 默认不打扰业务主流程
2. 需要时可以进入
3. 看得懂范围和风险

高级配置的职责边界必须明确：

1. 这里只承接低频治理类能力，不承接高频主路径动作
2. 这里只放“默认用户今天不一定要做”的配置
3. 任何放入 `高级配置` 的能力，都不能反向要求用户先回来这里完成主路径

本轮 MVP 中，以下能力统一归入 `高级配置`：

1. 公共字段治理
2. 低频的公共映射维护
3. 需要全局影响评估的治理项

与菜单管理的边界要求如下：

1. 菜单管理负责“引用”和“一步新增”
2. 高级配置负责“治理”和“全局维护”
3. 菜单管理中如果只为完成当前页面任务，不得把用户强行引流到 `高级配置`
4. `高级配置` 中的治理结果可以被菜单管理引用，但引用动作本身必须保持轻量

---

## 6. 字段展示规范

## 6.1 三类字段处理规则

### 第一类：系统自动维护字段

例如：

1. `id`
2. `currentVersion`
3. 内部编码
4. 系统计数

原型规则：

1. 不让用户手填
2. 不放在主编辑区
3. 需要时只做只读展示

### 第二类：技术稳定标识

例如：

1. `pageCode`
2. `frameCode`
3. `menuCode`
4. `laneCode`

原型规则：

1. 不放业务主路径表单
2. 只放详情或高级区
3. 优先展示中文上下文，再展示技术值

### 第三类：业务可理解字段

例如：

1. 名称
2. 说明
3. 状态
4. 机构范围
5. 页面
6. 规则名称
7. 用途说明

原型规则：

1. 允许编辑
2. 必须放在主路径
3. 必须有清晰文案和校验

## 7. 机构 / 人员规范

从用户视角，前端必须做到一件事：

> 用户看到的是中文名，系统内部保存的是稳定值。

因此，前端原型必须统一满足：

1. 机构选择使用统一机构组件
2. 人员选择使用统一人员组件
3. 列表展示使用统一中文名渲染
4. 详情展示使用统一中文名渲染

前端原型允许：

1. 当前先用 Mock 或静态映射
2. 未来再切换真实目录源

但前端页面本身不允许再散落写：

1. 手工映射逻辑
2. 手工硬编码中文名
3. 直接显示内部 ID 给业务用户

---

## 8. 校验体验规范

## 8.1 用户理解模型

用户视角下，前端原型中的校验应该被理解为三层：

1. 编辑时提醒我
2. 保存时拦非法结构
3. 发布时做最终门禁

这个模型可以保留，但页面交互必须连续。

## 8.2 编辑态

前端原型应尽量前移低成本问题：

1. 必填缺失
2. 基础格式错误
3. 明显范围错误
4. 引用资源不存在
5. 来源切换后的脏数据

提示要求：

1. 就地提示
2. 尽量定位到字段或区域
3. 文案可执行

## 8.3 保存态

保存前端原型应让用户明白：

1. 哪些地方还能保存草稿
2. 哪些错误已经不能保存

保存反馈要求：

1. 不只给 toast
2. 应给结构化摘要
3. 应能帮助用户返回修改

## 8.4 发布态

发布前端原型应让用户理解：

1. 发布会自动做最终检查
2. 未通过不能正式生效
3. 失败后能看到明确原因

发布反馈要求：

1. 自动触发最终校验
2. 阻断项可读
3. 风险项可读
4. 影响范围可读

---

## 9. 原型范围内不做的事

为了保证“小而美”，前端原型本轮明确不做：

1. 大量真实后端联调前置
2. 人员级灰度
3. 复杂平台级实验能力
4. 低频复杂技术参数前置暴露
5. 为少量边缘场景牺牲主路径简洁度

---

## 10. 原型验收标准（用户视角）

当以下条件满足时，可认为前端原型 MVP 达标：

### 10.1 入口达标

1. 用户知道从哪里开始
2. 用户知道下一步去哪

### 10.2 字段达标

1. 用户不再手填系统字段
2. 用户主路径中明显减少技术词和内部编码
3. 技术字段主要出现在详情或高级区

### 10.3 流程达标

1. 用户能完成配置 -> 保存 -> 发布 -> 看结果
2. 用户不需要额外进入独立校验流程

### 10.4 展示达标

1. 机构与人员统一显示中文名
2. 页面信息层次清楚
3. 提示、作业、API、发布之间承接自然

---

## 11. 结论

这份前端原型实施规范最终只保留一句判断标准：

> 如果一个设计让业务用户更少思考“系统内部是怎么实现的”，而更容易完成“我要做的业务动作”，它就更接近“小而美 MVP”。

后续前端原型实现如有冲突，统一按以下优先级取舍：

1. 用户主路径是否更顺
2. 用户是否少看到技术实现细节
3. 用户是否少做无意义输入
4. 用户是否更容易完成连续闭环
5. 技术扩展性

这份文档仅作为前端原型侧的 MVP 实施规范。
