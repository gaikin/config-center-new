﻿# 营小助配置中心实施计划（Plan）

> 来源文档：[prd-analysis.md](/D:/Dev/work/config-center-new/spec/config-center-product/prd-analysis.md)  
> 目标：形成可直接落地执行的实现计划（不绑定具体日期，按阶段推进）  
> 版本：V1.0-P0  
> 更新日期：2026-03-11

## 1. 实施目标

1. 打通 P0 端到端闭环：`提示命中 -> 联动作业 -> 编排执行 -> 统一预览 -> 一次性注入 -> 全量审计`。
2. 确保配置平台由业务人员低门槛使用：可视化、向导化、模板化、可理解校验。
3. 达成上线门禁：性能、成功率、漏报率、日志完整率全部满足。

## 2. 范围与边界

### 2.1 本次范围（P0）
1. 智能提示引擎（OR/AND、运算符、三类值来源、预处理器链）。
2. 智能作业与场景编排（单作业单编排、顺序执行、失败 `STOP`、四类节点）。
3. 接口配置子项（发布流转、引用保护、权限、参数映射）。
4. JSSDK 运行链路（DOM 触发、防抖频控、渲染、注入、事件回传）。
5. 安全审计与指标看板（全量日志、冷热分层、脱敏查询、统一指标口径）。
6. 配置平台易用性（向导、模板、实时校验、主流程收敛）。

### 2.2 不在本次范围（P1+）
1. 提示疲劳治理增强（如策略实验平台）。
2. 低置信度人工确认流引擎扩展。
3. 跨域 iframe 深度能力扩展（仅保留边界定义与降级策略）。

## 3. 落地口径（必须统一）

1. 发布与生效范围一体化：发布时必须带机构/系统/角色/菜单范围。
2. 冲突仲裁顺序固定：`风险等级 > 规则优先级 > 触发时间`。
3. 防抖频控默认值：2 秒防抖；30 秒同规则最多触发 1 次（同用户同页面）。
4. 页面定位策略：默认 XPath，失败时允许 CSS Selector 兜底。
5. JS 分级治理：预处理器 JS与编排 `js_script`暂不治理。
6. 指标统一维度：自然日 + 业务菜单。
7. 日志分层：30 天热存，180 天冷存，查询脱敏。
8. 易用性底线：业务人员可无代码完成“创建规则->绑定作业->发布生效”。

## 4. 阶段计划（按里程碑）

## 4.1 Phase 0：启动与基线冻结

目标：冻结 P0 实现口径与跨团队接口。

交付物：
1. 需求基线包（PRD + 验收清单 + 口径表）。
2. 技术方案评审纪要（架构、数据模型、安全审计）。
3. 测试基线（功能/性能/安全/可用性）。

退出条件：
1. 关键口径无阻塞争议。
2. 各模块 owner、依赖、排期窗口明确。

## 4.2 Phase 1：配置与运行时主链路

目标：打通提示触发到作业联动最小链路。

交付物：
1. JSSDK 初始化、DOM 通知、防抖频控。
2. 规则引擎（表达式、取值、预处理、命中判定）。
3. 提示渲染与同轮合窗。

退出条件：
1. 能在试点页面稳定命中提示并记录日志。
2. 联动作业触发链路可运行。

## 4.3 Phase 2：作业编排与接口治理

目标：完成作业执行核心能力。

交付物：
1. 接口配置子项（`DRAFT/PUBLISHED/OFFLINE`、权限、引用保护）。
2. 参数映射与组装（`test_value`、默认 string、缺参跳过）。
3. 编排执行器与内置节点（`page_get/page_set/api_call/js_script`）。
4. 预览模式与悬浮按钮重复触发新实例。

退出条件：
1. 编排链路跑通，失败 `STOP` 生效。
2. 配置与发布治理规则全部生效。

## 4.4 Phase 3：易用性、安全与观测收口

目标：满足“业务人员易上手 + 合规可审计 + 可运营”。

交付物：
1. 向导化配置主流程（创建->绑定->发布生效）。
2. 模板中心（高频模板、示例、默认值）。
3. 字段级实时校验与修复建议。
4. JS 分级审计、日志冷热分层、脱敏查询。
5. 指标看板（命中率/采纳率/注入成功率/节省时长）。

退出条件：
1. 业务人员独立配置验收通过。
2. 审计日志完整率和数据口径一致性达标。

## 4.5 Phase 4：联调、压测、灰度上线

目标：完成生产前验证并灰度发布。

交付物：
1. 端到端联调报告。
2. 性能压测与瓶颈优化报告。
3. 菜单级灰度方案与回滚预案。
4. Go/No-Go 决策材料。

退出条件：
1. 上线门禁全部通过。
2. 灰度运行稳定，可进入扩面。

## 6. 验收计划

### 6.1 功能验收（必须通过）
1. 提示命中、合窗、联动作业顺序执行。
2. 编排节点串行、失败 `STOP`、预览确认后注入。
3. 接口发布流转、引用保护、参数组装策略。
4. 悬浮按钮重复触发新实例。

### 6.2 非功能验收（必须通过）
1. 提示链路 P95 <= 800ms。
2. 作业链路 P95 <= 1500ms。
3. 注入成功率 >= 98%。
4. 高风险漏报率 <= 1%。
5. 审计日志完整率 = 100%。

### 6.3 易用性验收（必须通过）
1. 业务人员可无代码完成配置发布全流程。
2. 错误提示可理解且可直接指导修复。
3. 模板创建较空白创建显著减少操作步骤。

## 7. 上线门禁（Go/No-Go）

1. P0 阻断缺陷清零。
2. 功能、性能、安全、易用性四类验收全部通过。
3. 灰度方案与回滚预案演练完成。
4. 监控告警与值守机制就绪。

## 8. 主要风险与缓解

| 风险 | 影响 | 缓解措施 |
| --- | --- | --- |
| 页面结构差异导致定位失效 | 提示/注入失败 | XPath + CSS 双策略，建立页面定位体检 |
| 上游接口不稳定 | 注入成功率下降 | 超时/重试平台统一治理 + 降级提示 |
| 脚本节点误用 | 页面副作用与合规风险 | 高风险发布审批 + 全量审计 + 灰度 |
| 配置复杂度高 | 业务人员上手慢 | 向导 + 模板 + 字段级校验与示例 |
| 日志成本高 | 存储与查询压力 | 热冷分层 + 生命周期归档策略 |

## 9. 执行节奏建议

1. 每周固定两次跨团队站会：一次进度同步，一次风险清理。
2. 每阶段结束输出阶段报告：完成项、阻塞项、质量状态、下一阶段入口条件。
3. 灰度期间每日输出运行快报：命中率、成功率、异常TOP、用户反馈。

## 10. 启动清单（可直接执行）

1. 冻结 P0 范围与统一口径，完成评审签字。
2. 指定各 WP owner，并建立依赖关系看板。
3. 先并行启动 `WP-A（易用性）`、`WP-B（提示引擎）`、`WP-C（接口治理）`。
4. 在 Phase 1 完成后立即启动端到端联调样板菜单。
5. 以样板菜单结果驱动 Phase 2/3 的优化和模板沉淀。
