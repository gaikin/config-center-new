## 0. 执行摘要

### 0.1 本次交付目标
基于已确认 spec，落地营小助配置中心 P0 能力：在开阳业务系统菜单级启用“智能提示 + 智能作业（含场景编排）”，满足可配置、可审计、可验证上线门禁。

### 0.2 最小闭环
最小闭环定义为：  
业务页面预注入 JSSDK -> 外部 SDK 监听 DOM 并通知 -> 智能提示规则计算（固定值/页面元素/接口值 + 预处理链）-> 提示合窗展示 -> 命中后触发单作业单编排 -> 节点顺序执行（`page_get`/`api_call`/`js_script`/`page_set`）-> 编排完成后统一预览 -> 一次性写入 -> 全量日志留痕与脱敏查询。

### 0.3 任务分块
1. 运行时链路：JSSDK、DOM 触发、规则引擎、提示渲染。
2. 配置与治理：菜单级启停、接口子项、权限与发布流转。
3. 智能作业：编排执行器、内置节点、预览与悬浮按钮。
4. 安全与审计：敏感脱敏、全量日志、运行指标。
5. 验证与准入：验收用例、性能门禁、试点联调。

### 0.4 并行建议
1. 前端运行时与后端规则服务并行。
2. 接口子项治理与编排执行器并行。
3. 安全审计能力与业务功能联调并行推进。

### 0.5 主要风险
1. 业务页面 DOM 差异导致 XPath 适配成本升高。
2. 内部数据源 SLA 波动影响注入时效与成功率。
3. 编排脚本节点允许直接改页面，若缺少沙箱和审计约束，存在稳定性与追责风险。

## 3. 技术方案

### 3.1 整体思路
1. 采用“配置中心 + 决策服务 + 浏览器 JSSDK”三段式架构。
2. 提示与作业统一走提示命中链路，避免双触发口径分叉。
3. 编排执行器与规则引擎解耦：规则只决定“是否触发 + 触发哪个作业/编排”，编排负责执行步骤。

### 3.2 关键技术决策
1. 触发入口唯一：外部 SDK DOM 监听事件 -> 智能提示 SDK 规则计算。
2. 提示窗口策略：同轮命中合窗展示，联动作业按触发顺序串行执行。
3. 编排约束：一作业一编排；节点串行；失败固定 `STOP`；悬浮按钮触发新实例。
4. 节点类型：首批内置 `page_get`、`page_set`、`api_call`、`js_script`，支持 `custom` 扩展。
5. 预览时机：编排节点全部完成后统一预览，确认后一次性写入。
6. 接口治理：接口子项先发布后引用，`timeout_ms/retry_count` 平台统一控制且前端不展示。
7. 日志策略：参数/调用/编排/脚本行为全量落库，查询默认脱敏。

### 3.3 依赖与约束
1. 依赖开阳业务系统稳定注入 JSSDK。
2. 依赖内部系统接口可用性与鉴权能力。
3. 受限于页面 DOM 差异，需建立菜单级 XPath 映射规范与校验机制。

## 8. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| DOM 结构变更频繁 | XPath 失效导致触发/注入失败 | 建立菜单级 XPath 体检任务 + 回归样本库 |
| 内部接口 SLA 抖动 | 注入超时、体验下降 | 接口健康监控 + 失败降级提示 + 重试策略平台统一治理 |
| 脚本节点可改页面 | 页面副作用与审计争议 | 强制脚本行为留痕、发布前自测门禁、按菜单灰度 |
| 多规则并发命中 | 提示疲劳、重复联动 | 合窗展示 + 防抖频控 + 顺序执行 |
| 合规审计压力 | 上线受阻 | 全量日志、脱敏查询、审计字段前置验收 |
