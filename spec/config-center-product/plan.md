# 营小助配置中心实施计划（Plan）

> 基线文档：
> [prd-analysis.md](./prd-analysis.md)（总纲）  
> [sentinel-prd.md](./sentinel-prd.md)（智能提示子文档）  
> [job-prd.md](./job-prd.md)（智能作业子文档）  
> 决策台账：[decision-ledger.md](./decision-ledger.md)  
> 目标：基于最新产品文档形成可执行、可校验、可治理的实施路径  
> 版本：V2.0-P0  
> 更新日期：2026-03-12

## 1. 实施目标

1. 以统一平台思路建设营小助配置中心，打通“页面资源 -> 规则触发 -> 提示引导 -> 作业执行 -> 审计治理”的完整闭环。
2. 坚持“总纲定义边界、子文档承接细节”的文档分工，避免跨文档口径冲突。
3. 以治理、校验、留痕为上线底线，优先保障稳定可控，再逐步扩展复杂能力。
4. 形成可复制的试点实施方法，支撑后续按业务菜单、组织范围和场景模板扩面。

## 2. 范围

### 2.1 本次范围（P0）

1. 公共支撑层：页面资源、API注册、预处理器、角色与权限基础能力。
2. 智能提示层：规则建模、条件判断、提示展示、关闭策略、提示入口联动。
3. 智能作业层：场景建模、节点编排、执行方式、预览确认、悬浮按钮二次触发。
4. 生命周期治理：状态流转、版本替换、发布前强校验、自动执行风险确认。
5. 审计与运营：治理日志、触发日志、执行日志、治理视图、核心指标。
6. 试点交付：样板页面接入、外部环境验证、灰度发布、回滚预案。

### 2.2 不在本次范围（P1+）

1. 平台内置完整联调测试能力或真实链路模拟引擎。
2. 高自由度低代码平台化扩展与复杂脚本生态市场化。
3. 提示多按钮分发中心与复杂提示类型体系。
4. 跨域复杂系统兼容的全面覆盖与深水区专项治理。

### 2.3 文档读取与分工约定

1. `plan.md` 固定输出结构：实施目标 -> 范围 -> 阶段计划 -> WBS -> 验收 -> 风险 -> 启动清单。
2. `plan.md` 只保留总体设计、实施边界与交付要求，不展开智能提示/智能作业实现细节。
3. 智能提示细节统一以 `sentinel-prd.md` 为准；智能作业细节统一以 `job-prd.md` 为准。
4. 计划只引用“已确认决策”；未确认事项统一留在 `decision-ledger.md` 管理。

## 3. 阶段计划（按顺序推进）

### 3.1 Phase 0：基线冻结与对象模型确认

目标：冻结统一产品口径，确认跨模块对象与治理基线。

交付物：
1. 对象模型基线：页面、规则、提示、作业场景、API注册、预处理器、角色。
2. 生命周期与发布治理基线：主状态、版本替换、强校验、风险确认。
3. 试点范围清单：样板菜单、页面、组织范围、责任人、样板场景。

退出条件（Exit Criteria）：
1. 关键对象模型无阻塞分歧。
2. 关键依赖关系和交付边界明确。
3. 试点业务责任人确认完成。

### 3.2 Phase 1：公共支撑层落地

目标：搭建可复用的统一底座，支撑提示与作业共同实施。

交付物：
1. 页面资源管理主链路（建档、识别、元素映射、版本化）。
2. API注册主链路（参数映射、出参映射、运行策略、调试入口）。
3. 预处理器中心（内置能力、受控扩展、跨模块复用）。
4. 角色与权限基础能力（角色管理、组织范围、操作类型边界）。

退出条件（Exit Criteria）：
1. 提示与作业均可复用同一公共支撑能力。
2. 样板页面可跑通“建档 -> 元素复用 -> API引用”链路。

### 3.3 Phase 2：智能提示主链路

目标：完成智能提示从规则配置到运行命中的稳定闭环。

交付物：
1. 规则配置与执行主链路（单层条件、全局 AND/OR、优先级）。
2. 提示展示与关闭策略（静默/浮窗、关闭行为、串行策略）。
3. 提示入口联动作业口径（单入口、确认触发、关闭不触发）。
4. 触发日志与异常定位能力。

引用：`sentinel-prd.md` 第 4、5、6、8 章。

退出条件（Exit Criteria）：
1. 样板页面可稳定命中并按策略展示提示。
2. 提示关闭、确认触发、串行展示行为符合口径。
3. 触发日志可支持业务与技术双向排查。

### 3.4 Phase 3：智能作业主链路

目标：完成智能作业从场景编排到执行注入的稳定闭环。

交付物：
1. 场景模型与对象关系落地。
2. 编排模型与基础节点能力落地。
3. 执行方式、预览确认、悬浮按钮二次触发落地。
4. 失败兜底、回退与执行留痕落地。

引用：`job-prd.md` 第 4、5、6、7、8、9 章。

退出条件（Exit Criteria）：
1. 样板场景可跑通“查数 -> 加工 -> 预览/确认 -> 注入 -> 留痕”。
2. 自动执行、预览确认、手动触发三类主模式可用。
3. 失败可回退手工流程，且日志可定位到节点级原因。

### 3.5 Phase 4：治理、审计与指标收口

目标：将平台从“功能可用”提升到“治理可控、审计可追溯、指标可运营”。

交付物：
1. 主状态流转与版本替换机制。
2. 发布前强校验体系与自动执行风险确认机制。
3. 治理视图（待处理、即将到期、校验失败、冲突对象）。
4. 日志查询与核心指标看板。

退出条件（Exit Criteria）：
1. 生效中对象无法被直接修改，版本替换闭环稳定。
2. 不合规配置可被强校验阻断。
3. 治理角色可在待处理视图完成主要治理动作。

### 3.6 Phase 5：试点灰度与扩面准备

目标：在真实业务环境验证可用性并沉淀扩面能力。

交付物：
1. 外部环境验证报告与问题闭环。
2. 灰度方案、回滚预案、运行值守机制。
3. 试点运行快报与扩面建议（模板、复用、组织复制）。

退出条件（Exit Criteria）：
1. 试点场景真实环境稳定运行。
2. 关键质量问题闭环，具备扩面条件。
3. Go/No-Go 评审通过。

## 5. 验收口径

### 5.1 功能验收

1. 提示链路可验收：规则配置、条件命中、提示展示、关闭/确认联动。
2. 作业链路可验收：场景编排、执行方式、预览确认、注入与失败回退。
3. 治理链路可验收：状态流转、发布校验、版本替换、治理视图。

### 5.2 质量验收

1. 发布前强校验可阻断不合规配置。
2. 失败场景可回退手工路径，不阻断业务办理。
3. 三类日志可追溯到对象、动作、时间与结果。

### 5.3 边界验收

1. 平台仅承诺基础校验能力。
2. 真实联调与最终业务结果验证在外部环境完成。
3. 运行时不承诺业务冲突仲裁，冲突治理以前置校验为主。

## 6. 主要风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 页面资源模型不稳定 | 同时影响提示命中和作业执行 | 先做样板页面建档，冻结识别与元素基线 |
| 自动执行风险失控 | 误填、误触发、责任不清 | 发布前风险确认、责任绑定、可回退 |
| 场景复杂度过快上升 | 业务配置门槛提高，交付变慢 | 先做高频稳定场景，复杂能力分阶段引入 |
| 脚本能力失控 | 合规与稳定性风险上升 | 内置能力优先，脚本受控发布与审计 |
| 指标口径不一致 | 无法评估真实业务价值 | 统一指标定义与计算口径，按台账维护 |
| 文档漂移 | 实施口径前后不一致 | 总纲管边界，子文档管细节，台账管决策 |

## 7. 启动清单（可直接执行）

1. 冻结试点业务菜单、页面与责任人清单。
2. 完成对象模型评审（页面、规则、提示、作业、API注册、预处理器、角色）。
3. 对齐 `permission-model.md` 的实施映射并确认治理边界。
4. 并行启动 WP-A、WP-B、WP-C 的方案与原型。
5. 固化发布前强校验清单与自动执行风险确认流程。
6. 建立外部环境验证计划与灰度回滚预案。
7. 约定阶段性评审机制：每个 Phase 输出“交付状态 + 风险状态 + 退出结论”。
