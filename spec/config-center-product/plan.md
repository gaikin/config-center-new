# 营小助配置中心实施计划（Plan）

> 来源文档：[prd-analysis.md](./prd-analysis.md)、[permission-model.md](./permission-model.md)  
> 目标：基于新版产品分析结论，形成统一的实施基线与分阶段落地路径  
> 版本：V1.1-P0  
> 更新日期：2026-03-12

## 1. 实施目标

1. 以统一平台思路建设营小助配置中心，打通 `页面资源 -> 规则触发 -> 提示引导 -> 作业执行 -> 审计治理` 的完整闭环。
2. 让业务角色成为主要配置者和治理者，技术角色聚焦平台支撑、通用能力维护与异常排查。
3. 以发布前强校验、失败可降级、全链路留痕为 P0 上线底线，先确保稳定可控，再逐步扩展复杂能力。
4. 形成可复制的试点实施方法，支撑后续按业务菜单和组织范围扩面。

## 2. P0 范围与非范围

### 2.1 本次范围（P0）
1. 公共支撑层：页面资源管理、接口定义中心、预处理器中心、角色管理基础能力。
2. 智能提示层：页面识别、规则模型、单层条件判断（全局 AND/OR）、优先级、提示展示与关闭策略、提示确认入口。
3. 智能作业层：独立作业场景模块、基础节点编排、执行方式、预览确认、悬浮按钮二次触发、失败降级。
4. 生命周期治理：待发布/生效中/已停用/已失效状态流转、版本替换、发布校验、停用与回滚。
5. 审计与运营：治理日志、触发日志、执行日志、待处理视图、核心指标看板。
6. 试点交付：试点页面接入、外部环境联调、灰度发布、回滚预案。


### 2.2 不在本次范围（P1+）
1. 平台内置完整联调测试能力或真实链路模拟引擎。
2. 高自由度低代码画布、复杂脚本生态和大规模自定义组件市场。
3. 提示多按钮分发中心、复杂提示类型体系和疲劳治理实验平台。
4. 以单页面、单规则、单场景为常规模式的细粒度 ACL 授权。
5. 跨域复杂页面、极端特殊系统兼容等深水区能力的全面覆盖。
### 2.3 当前前端交互口径（2026-03-12）
1. 规则条件编辑采用单层结构，统一全局 AND/OR，不做多层条件组嵌套。
2. 预处理器在条件行内使用下拉多选，不再通过弹窗选择。
3. 作业场景中心采用左（节点库）-中（画布）-右（属性）布局，移除执行实例与手工预览模块。
4. 作业编排支持节点选中属性编辑和一键自动排版。
## 3. 统一实施基线（必须统一）

1. 先建设完整平台骨架，不把页面资源、提示、作业、治理拆成孤立项目。
2. 页面资源是所有能力起点；页面识别优先稳定业务标识，URL 作为兜底，元素选择器统一在页面资源中维护。
3. 规则和作业场景分开建模、分开版本化、分开治理，通过明确关联关系协同工作。
4. 规则仅保留名称作为标识，名称需全局唯一；作业场景同样仅保留名称，名称需全局唯一。
5. 提示当前只支持静默提示和浮窗提示两种方式；提示中的作业入口统一为单一“确认”按钮，不扩展多按钮。
6. 执行方式统一支持无提示自动执行、提示后自动执行、预览确认后执行；悬浮按钮作为补充触发入口，每次触发必须新建执行实例。
7. 同页同字段写入冲突必须在发布前校验并消解；运行时只保留最小化技术防护，不再做业务仲裁。
8. 所有生效中对象的内容修改都必须生成新的待发布版本，不能直接改动线上生效版本。
9. 预处理器优先提供内置能力，自定义脚本和 `js_script` 必须纳入受控发布和审计，不作为默认主路径。
10. 平台只承诺基础校验能力；真实联调、页面注入验证和最终结果确认仍在外部系统或实际业务环境完成。
11. 治理首页默认进入“待处理”视图，优先帮助业务管理角色处理待发布、即将到期、待风险确认等对象。
12. 核心运营指标统一收口为作业执行成功率、平均节省时长、失败原因分布、已失效数量、即将到期数量。

## 4. 阶段计划（按里程碑）

### 4.1 Phase 0：基线冻结与对象建模

目标：冻结新版产品口径，明确跨模块对象模型、状态模型和试点范围。

交付物：
1. 统一对象模型：页面、元素、规则、提示、作业场景、接口、预处理器、角色。
2. 生命周期与发布治理口径：四类主状态、版本替换方式、强校验清单、自动执行风险确认要求。
3. 试点范围清单：首批业务菜单、页面、组织范围、责任人和样板场景。
4. 角色与权限边界对齐稿：以 [permission-model.md](./permission-model.md) 为准完成实施映射。

退出条件：
1. 不再保留阻塞实施的口径分歧。
2. 各工作包 owner、依赖关系和样板页面明确。
3. 试点业务责任人和业务管理责任人确认到位。

### 4.2 Phase 1：公共支撑层先行

目标：先搭好可复用的页面、接口、预处理和角色治理底座。

交付物：
1. 页面资源管理：站点/专区/菜单/页面建档、页面编码、识别规则、元素映射、版本能力。
2. 接口定义中心：参数来源、出参路径、统一超时/重试、敏感信息保护、复用机制。
3. 预处理器中心：内置常用处理能力、受控扩展机制、提示/作业共用链路。
4. 角色管理基础版：角色定义、复制、停用/恢复、人员绑定、批量分配、组织范围与操作类型绑定。
5. 基础治理日志模型：资源变更、状态变更、发布记录、责任人记录。

退出条件：
1. 规则和作业都可以复用统一页面资源、接口和预处理能力。
2. 角色与组织范围控制可支撑后续规则、作业、页面、接口治理。
3. 页面建档到元素复用的主链路在样板页面可跑通。

### 4.3 Phase 2：智能提示主链路

目标：完成从页面识别到提示展示的稳定闭环。

交付物：
1. 页面运行时识别与上下文采集。
2. 规则配置与执行能力：单层条件编辑、全局 AND/OR、多来源取值、预处理链、优先级 1-999。
3. 规则配置校验能力：行内编辑校验、保存前完整性校验与引用有效性校验。
4. 提示渲染能力：静默提示/浮窗提示、三类关闭行为、同规则稳定状态下不重复打扰。
5. 单一确认入口：提示中“确认”按钮触发关联作业，关闭提示则不触发作业。
6. 触发日志与异常定位：命中、未命中、字段缺失、预处理失败、接口异常等原因可追踪。

退出条件：
1. 试点页面可稳定识别并按规则优先级正确展示提示。
2. 提示关闭、确认触发、不重复打扰等关键行为符合产品口径。
3. 触发日志可支持业务和技术定位主要问题。

### 4.4 Phase 3：智能作业主链路

目标：完成独立作业场景编排与执行闭环。

交付物：
1. 独立作业模块：场景基础信息、关联信息、节点编排区、执行方式配置、基础校验区。
2. 编排交互：流程图式主视图 + 结构化表单编辑，不引入完全自由画布。
3. 基础节点能力：`page_get`、`api_call`、`js_script`、`page_set`。
4. 执行方式落地：无提示自动执行、提示后自动执行、预览确认后执行、悬浮按钮手动触发。
5. 预览确认页：字段名称、原值、拟写入值、数据来源、异常标识、按字段勾选写入。
6. 执行日志与失败处理：节点输入输出、耗时、失败原因、手工兜底路径。
7. 悬浮按钮二次触发：新建执行实例，不复用旧上下文。

退出条件：
1. 样板场景可实现“查数 -> 加工 -> 预览/确认 -> 注入 -> 留痕”全链路。
2. 自动执行、预览确认、手动触发三类主要模式均可落地。
3. 失败后可回退手工流程，且日志能定位到节点级原因。

### 4.5 Phase 4：治理与运营收口

目标：把配置平台从“可用”提升到“可治理、可审计、可运营”。

交付物：
1. 四类主状态流转与版本替换机制。
2. 发布前强校验：责任人、适用范围、生效失效时间、冲突检查、资源有效性、人工作业时长、自动执行风险确认。
3. 冲突前置治理：同页同字段冲突扫描、阻断发布、例外确认留痕。
4. 治理视图：默认待处理视图、即将到期、校验不通过、存在冲突、待风险确认等快捷筛选。
5. 审计体系：治理日志、触发日志、执行日志统一查询；敏感信息脱敏展示。
6. 运营指标看板：作业执行成功率、平均节省时长、失败原因分布、已失效数量、即将到期数量。

退出条件：
1. 生效中对象不能被直接修改，版本替换流程可闭环。
2. 发布校验和风险确认可阻断不合规配置上线。
3. 业务管理角色可以在待处理视图完成主要治理动作。

### 4.6 Phase 5：外部验证与灰度上线

目标：在真实业务环境中完成验证、灰度和扩面准备。

交付物：
1. 外部环境联调方案与结果报告。
2. 样板菜单灰度方案、回滚预案和异常值守机制。
3. 试点业务运行日报：触发情况、执行成功率、主要失败原因、用户反馈。
4. 扩面建议：模板沉淀、页面资源复用策略、组织复制方案。

退出条件：
1. 试点场景在真实环境稳定运行，可回退、可追溯。
2. 主要质量问题已完成闭环，具备扩面条件。
3. Go/No-Go 评审通过。

## 6. 验收口径

### 6.1 功能验收（必须通过）
1. 页面可完成识别、建档、元素复用，规则与作业共享同一资源底座。
2. 规则可完成单层条件配置（全局 AND/OR）、优先级排序、提示展示、关闭与确认触发。
3. 作业场景可独立编排、执行、预览确认、注入页面，并支持悬浮按钮二次触发。
4. 规则与作业场景均能完成待发布、生效中、停用、失效和版本替换流程。
5. 角色管理、组织范围控制、操作类型控制和治理视图可支撑业务侧日常治理。

### 6.2 治理与质量验收（必须通过）
1. 发布前强校验可以阻断缺责任人、缺范围、缺时间、冲突写入、失效引用、缺人工作业时长、缺自动执行风险确认等问题。
2. 失败场景可回退到手工路径，不因平台异常阻断业务办理。
3. 三类日志完整可查，关键敏感信息按权限脱敏展示。
4. 运营看板可稳定输出核心指标，并支持定位主要失败原因。
5. 生效中对象不能被直接修改，所有关键变更均可追溯到版本和责任人。

### 6.3 验证边界（需明确）
1. 平台内仅提供基础校验，不以“完整测试平台”作为 P0 验收目标。
2. 真实页面注入效果、跨系统联调结果和最终业务正确性，须在外部系统或实际业务环境中验证。
3. 运行时不以“业务冲突仲裁”作为兜底能力；业务冲突必须在发布前完成消解。

## 7. 主要风险与缓解

| 风险 | 影响 | 缓解措施 |
| --- | --- | --- |
| 页面资源模型不稳 | 同时影响提示命中和作业执行 | 页面资源先行建设，试点先做样板页面建档与复用验证 |
| 业务配置门槛仍偏高 | 业务侧无法真正独立使用 | 规则与作业分模块建设，主流程向导化，提供示例与默认值 |
| 自动执行带来业务风险 | 造成误填或不可控操作 | 自动执行前置风险确认，发布留痕，失败可回退手工路径 |
| 规则和场景长期堆积 | 平台治理失控 | 强制责任人、有效期、待处理视图和即将到期治理机制 |
| 自定义脚本失控 | 引入合规和稳定性风险 | 内置预处理优先，`js_script` 受控发布、审计和灰度使用 |
| 指标口径不一致 | 无法评估平台真实价值 | 统一核心指标定义，按成功执行和人工作业时长计算收益 |

## 8. 执行节奏建议

1. 以样板菜单为牵引推进，先跑通一条完整闭环，再复制到更多页面和组织。
2. Phase 1 与 Phase 2 可以并行准备，但必须以前置对象模型和页面资源基线为准。
3. 每个 Phase 结束都输出阶段报告：已完成项、阻塞项、质量状态、进入下一阶段条件。
4. 灰度阶段按日输出运行快报，重点关注失败原因、即将到期对象和手工兜底占比。

## 9. 启动清单（可直接执行）

1. 冻结首批试点业务菜单、页面和责任人名单。
2. 完成页面、规则、作业、接口、预处理器、角色六类对象模型评审。
3. 以 [permission-model.md](./permission-model.md) 为基线补齐角色管理与组织范围实施映射。
4. 并行启动 `WP-A（页面资源与公共支撑）`、`WP-B（规则与提示）`、`WP-C（作业场景）` 的方案与原型设计。
5. 明确发布前强校验清单和自动执行风险确认流程，作为后续开发的前置约束。
6. 准备外部环境联调计划和样板菜单灰度方案，避免把真实验证滞后到开发尾声。
