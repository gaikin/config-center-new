## 1. 实施目标

1. 建设统一配置平台，打通“页面资源 -> 页面启用策略 -> 规则触发 -> 提示引导 -> 作业执行 -> 治理审计”的完整闭环。
2. 将业务系统页面建模固定为 `site -> region -> menu -> page` 四层结构，菜单父级统一使用 `regionId` 表达。
3. 将 JSSDK 发布模式固化为“稳定入口 `loader` + 菜单级版本控制 + 页面级能力启用 + CDN 分包加载”。
4. 明确职责边界：技术人员发布 SDK 制品与槽位，业务人员配置菜单灰度范围与页面启用策略。
5. 形成与文档一致的 Admin Web 原型，覆盖页面资源、SDK 版本中心、页面启用策略、规则、作业、治理与审计关键场景。

## 2. 范围

### 2.1 本次范围（P0）

1. 公共支撑层：页面资源、接口注册、预处理器、角色与权限基础能力。
2. 页面资源层级：站点、专区、菜单、页面、页面字段与 iframe 页面建档。
3. SDK 发布控制：SDK 制品版本、版本槽位、菜单级版本控制、机构灰度范围、版本回退。
4. 页面启用控制：页面是否启用智能提示、是否带作业、作业预热策略。
5. 智能提示层：规则建模、条件判断、提示展示、关闭策略、提示入口联动。
6. 智能作业层：场景建模、节点编排、执行方式、预览确认、悬浮按钮二次触发。
7. 生命周期治理：状态流转、版本替换、发布前强校验、自动执行风险确认、回滚。
8. 审计与运营：治理日志、触发日志、执行日志、治理视图、核心指标。
9. 试点交付：样板菜单接入、真实环境验证、机构灰度、回滚预案。

### 2.2 不在本次范围（P1+）

1. 平台内置完整联调测试能力或真实链路模拟引擎。
2. 人员级 SDK 灰度、复杂白名单运营平台和实验平台化能力。
3. 跨域复杂 iframe 的全面兼容与深水区专项治理。
4. 高自由度低代码平台化扩展与复杂脚本生态市场化。
5. 提示多按钮分发中心与复杂提示类型体系。

### 2.3 文档分工约定

1. `plan.md` 只保留实施目标、阶段计划、WBS、验收、风险与启动清单。
2. 总体架构、JSSDK 分包、菜单级版本控制、页面启用策略等设计细节，以 `architecture.md` 与 `jssdk-design.md` 为准。
3. 智能提示细节以 `sentinel-prd.md` 为准；智能作业细节以 `job-prd.md` 为准。
4. 未冻结事项继续记录在 `decision-ledger.md`，不在 `plan.md` 中混入未确认方案。

## 3. 当前冻结决策

1. 业务系统统一注入稳定入口 `cc-sdk-loader.js`，由 `loader` 基于 `appId + env + regionId + menuCode + orgId` 解析实际 SDK 版本。
2. SDK 版本控制绑定到菜单层级，菜单下页面与 iframe 默认继承同一 SDK 版本。
3. 菜单的父级不是 `parentMenuCode`，而是专区主键 `regionId`。
4. 技术人员负责发布 SDK 制品版本与维护 `stable / gray-a / gray-b` 等版本槽位。
5. 业务人员负责配置菜单灰度范围，灰度范围在 P0 仅到机构维度，不开放人员级配置。
6. 页面是否启用智能提示放在配置端由业务人员管理，但页面识别方式、字段模型、iframe 定位由技术人员维护。
7. JSSDK 必须采用“统一宿主 + 分包加载 + 下载与实例化分离”，不做提示与作业全量大包。
8. 运行面发布产物拆分为 `manifest + sdk-release-index + page-index + page-config + scene-config`。

## 4. 阶段计划（按顺序推进）

### 4.1 Phase 0：基线冻结与对象模型确认

目标：冻结跨模块对象模型、角色边界与运行时控制口径。

交付物：
1. 页面资源层级基线：`site -> region -> menu -> page`。
2. SDK 发布控制基线：制品版本、槽位、菜单策略、机构灰度。
3. 页面启用控制基线：页面启用策略、提示规则集、作业预热策略。
4. 试点范围清单：站点、专区、菜单、页面、机构、责任人。

退出条件：
1. 关键对象模型无阻塞分歧。
2. 技术角色与业务角色职责边界明确。
3. 试点业务责任人确认完成。

### 4.2 Phase 1：公共支撑层与页面资源建模

目标：完成公共底座和页面资源主链路，支撑菜单级版本控制与页面级启用控制。

交付物：
1. 站点、专区、菜单、页面资源、元素映射主链路。
2. iframe 页面建档与 `frameCode` 管理口径。
3. API 注册、预处理器、角色权限基础能力。
4. 页面识别依赖的稳定业务标识口径。

退出条件：
1. 页面资源可按专区/菜单/页面维度稳定建档。
2. 样板菜单可跑通“建档 -> 元素复用 -> 页面识别”链路。

### 4.3 Phase 2：SDK 发布控制与运行时分发

目标：完成技术发布 SDK、业务配置灰度范围、运行时解析版本的闭环。

交付物：
1. SDK 制品版本登记与槽位维护。
2. 菜单级 SDK 版本控制与机构灰度配置。
3. `loader -> sdk-release-index -> core` 的运行时解析链路。
4. `manifest + sdk-release-index + page-index + page-config + scene-config` 发布产物。

退出条件：
1. 样板菜单可按机构命中不同 SDK 槽位。
2. 未命中灰度策略时可稳定回退到 `stable`。
3. 版本切换与回滚链路可验证。

### 4.4 Phase 3：页面启用控制与智能提示主链路

目标：完成页面级启用策略与智能提示从配置到运行命中的稳定闭环。

交付物：
1. 页面启用策略配置：启用开关、提示规则集、作业开关、预热策略。
2. `page-index` 生成与页面命中判定。
3. 智能提示规则配置、提示展示、关闭/确认策略。
4. 触发日志与异常定位能力。

退出条件：
1. 业务人员可自行控制菜单下哪些页面启用提示。
2. 页面未命中启用策略时，`prompt` 不加载。
3. 样板页面可稳定命中提示并按策略展示。

### 4.5 Phase 4：智能作业、治理与发布收口

目标：完成智能作业、预览确认、治理与发布闭环。

交付物：
1. 作业场景模型、编排与执行方式。
2. `job / preview` 按需加载、预热与确认链路。
3. 发布前强校验、风险确认、治理视图、版本回滚。
4. 审计日志、执行日志、指标看板。

退出条件：
1. 样板场景可跑通“查数 -> 加工 -> 预览/确认 -> 注入 -> 留痕”。
2. 菜单策略、页面启用策略、规则与作业均可经过治理发布。
3. 失败可回退人工流程，日志可定位原因。

### 4.6 Phase 5：试点灰度与扩面准备

目标：在真实业务环境完成试点、机构灰度与扩面准备。

交付物：
1. 外部环境验证报告与问题闭环。
2. 机构灰度方案、回滚预案、运行值守机制。
3. 试点运行快报与扩面建议（菜单模板、页面模板、组织复制）。

退出条件：
1. 试点菜单在真实环境稳定运行。
2. 关键质量问题闭环，具备扩面条件。
3. Go/No-Go 评审通过。

## 6. 原型实现清单（当前 Admin Web）

### 6.1 本轮原型必须体现

1. 页面资源中心使用 `site -> region -> menu -> page` 口径，并展示 `pageCode / frameCode`。
2. 新增 SDK 版本中心，体现制品版本、版本槽位、菜单级版本控制、机构灰度范围。
3. 新增页面启用策略页，体现页面启用开关、提示规则集、作业开关、预热策略。
4. 治理工作台可看到菜单策略与页面启用策略的待发布/校验/回滚事项。
5. 文案、字段命名、对象关系与 `architecture.md`、`jssdk-design.md` 对齐。

### 6.2 原型允许简化

1. 版本制品与槽位可使用 Mock 数据，不接真实制品仓库。
2. 机构灰度范围可先使用多选列表表达，不实现复杂树形机构选择器。
3. `loader`、`manifest`、`sdk-release-index` 仅展示为配置对象与说明，不实现真实下载链路。
4. 规则集与作业场景绑定可用引用字段表达，不实现真实编译器。

## 7. 验收口径

### 7.1 功能验收

1. 页面资源链路可验收：站点、专区、菜单、页面、元素映射建模完整。
2. SDK 版本链路可验收：技术发布版本、业务配置菜单灰度、运行时版本解析口径清晰。
3. 页面启用链路可验收：业务可配置菜单下哪些页面启用提示与作业。
4. 提示链路可验收：规则配置、条件命中、提示展示、关闭/确认联动。
5. 作业链路可验收：场景编排、执行方式、预览确认、注入与失败回退。
6. 治理链路可验收：状态流转、发布校验、版本替换、治理视图、回滚。

### 7.2 质量验收

1. 发布前强校验可阻断不合规菜单策略、页面启用策略、规则和作业。
2. 未命中页面启用策略时，运行时不加载 `prompt`；未声明作业时，不预热 `job`。
3. 同一菜单会话内主页面与 iframe 使用同一 SDK 版本。
4. 失败场景可回退手工路径，不阻断业务办理。
5. 三类日志可追溯到对象、动作、时间、结果和命中的菜单/机构范围。

### 7.3 边界验收

1. P0 仅支持机构级 SDK 灰度，不支持人员级灰度。
2. 平台仅承诺基础校验能力，真实联调与最终业务验证在外部环境完成。
3. 运行时不承担业务冲突仲裁，冲突治理以前置校验和发布控制为主。

## 8. 主要风险与缓解

| 风险 | 影响 | 缓解措施 |
| --- | --- | --- |
| 页面资源模型不稳定 | 同时影响提示命中、作业执行与页面启用策略 | 先冻结样板菜单的页面识别与字段基线 |
| 菜单与专区标识不稳定 | 导致 SDK 版本解析与灰度命中不准 | 业务系统接入前强制确认 `regionId/menuCode` 提供方式 |
| 灰度范围配置失控 | 版本回退与排障复杂度上升 | P0 仅开放机构级灰度，并保留 `stable` 强制回退 |
| iframe 结构复杂 | 影响页面识别和注入稳定性 | 先做同源 iframe 样板，跨域 iframe 走降级策略 |
| 自动执行风险失控 | 误填、误触发、责任不清 | 风险确认、版本发布治理、失败回退并行落地 |
| 文档与原型漂移 | 实施口径前后不一致 | 以 `architecture.md`、`jssdk-design.md` 为唯一设计基线，原型迭代同步校验 |

## 9. 启动清单（可直接执行）

1. 冻结试点站点、专区、菜单、页面、机构与责任人清单。
2. 完成对象模型评审：`page_region`、`page_menu`、`page_resource`、`sdk_release_lane`、`menu_sdk_policy`、`page_activation_policy`。
3. 对齐业务系统接入方需要提供的 `appId / env / regionId / menuCode / pageCode / orgId` 口径。
4. 并行启动页面资源中心、SDK 版本中心、页面启用策略页原型。
5. 固化菜单级版本控制、页面启用策略、规则与作业的发布前强校验清单。
6. 建立试点机构灰度方案、回滚预案与运行值守机制。
7. 约定阶段评审机制：每个 Phase 输出“交付状态 + 风险状态 + 退出结论”。
