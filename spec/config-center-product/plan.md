# 营小助配置中心实施计划（Plan）

> 基线文档：

> 目标：将配置中心从“按系统对象拆分的配置后台”重构为“按业务任务组织、可独立完成配置闭环”的业务工作平台  
> 版本：V3.0-Biz  
> 更新日期：2026-03-14

## 1. 实施目标

1. 建立以业务任务为主线的产品路径，让业务人员进入系统后知道“先去哪、下一步做什么、完成后怎么看结果”。
2. 将 `菜单管理` 固化为业务人员主入口，围绕菜单承接页面、提示、作业、API、发布与运行结果的聚合视图。
3. 将 `智能提示`、`智能作业`、`API注册`、`发布与灰度` 四类高频流程改造成可预览、可校验、可确认影响范围的向导式流程。
4. 将首页改造为 `我的工作台`，聚焦待处理、最近修改、运行提醒、常用入口和业务看板，而不是平台建设状态。
5. 将 `运行统计` 收敛为业务视角的 `触发统计` 与 `下降提醒`，不把技术排障面板直接暴露给业务人员。
6. 将低频复杂能力统一收进 `高级配置`，按角色开放，避免干扰业务人员的高频日常操作。
7. 保持既有技术基线稳定：菜单级 SDK 控制、机构级灰度、页面启用策略、控制与审计口径以及运行时分发口径继续有效，不在本计划中推翻。

## 2. 范围

### 2.1 本次范围（P0-P1）

1. 一级菜单重组，统一为：`我的工作台`、`菜单管理`、`智能提示`、`智能作业`、`API注册`、`发布与灰度`、`运行统计`、`高级配置`。
2. 工作台重构：待发布、待确认、待补充配置、最近编辑、下降提醒、常用动作、业务看板。
3. 菜单管理重构：菜单筛选、菜单详情聚合、当前页面快捷动作、启用范围、已开通能力、运行趋势。
4. 智能提示重构：规则列表前置、模板作为快捷复用来源、四步向导、预览闭环、高频项前置、复杂条件后置。
5. 智能作业重构：从页面进入场景配置、场景卡片化、自动触发与预览确认显式表达、复杂编排下沉高级区。
6. API注册重构：保留名称、改为五步向导、在线测试、模板复用、引用关系回显、基础信息与高级配置分层。
7. 发布与灰度重构：待发布内容汇总、生效范围选择、机构级试点、立即/定时生效、影响确认、发布记录。
8. 运行统计重构：只保留触发统计与下降提醒两类业务视图，保留发布前后对比和页面/规则/机构维度分析。
9. 高级配置收口：预处理器、高级识别参数、接口高级配置、底层运行参数、权限管理统一后置，并按角色开放。
10. 原型导航、页面标题、说明文案、角色可见性和旧入口迁移策略同步调整。

### 2.2 不在本次范围（P2+）

1. 人员级灰度与复杂实验平台化能力。
2. 直接面向业务人员的技术故障告警台或运行明细大盘。
3. 推翻既有 `loader + 菜单级版本控制 + 页面级启用策略` 的运行时架构。
4. 深水区跨域 iframe 的专项兼容控制。
5. 全量重做后端数据模型或真实联调平台。
6. 将全部高级配置能力前置为一级高频入口。

### 2.3 文档分工约定

1. `plan.md` 只保留实施目标、阶段计划、WBS、原型落地要求、验收、风险和启动清单。
2. 业务侧体验、菜单结构和交互原则以 `business-user-optimization.md` 为主基线。
3. 总体架构、JSSDK 分包、菜单级版本控制、页面启用策略等技术细节以 `architecture.md` 与 `jssdk-design.md` 为准。
4. 智能提示与智能作业详细规则分别以 `sentinel-prd.md` 与 `job-prd.md` 为准。
5. 未冻结事项继续记录在 `decision-ledger.md`，不在 `plan.md` 混入待定方案。

## 3. 当前冻结决策

1. 业务侧一级菜单固定为：`我的工作台`、`菜单管理`、`智能提示`、`智能作业`、`API注册`、`发布与灰度`、`运行统计`、`高级配置`。
2. `菜单管理` 是业务人员主入口，业务侧高频动作优先从“先找到菜单，再定位页面”展开。
3. 行业内已形成共识的术语继续保留：`API注册`、`页面`、`规则`、`模板`、`发布`、`灰度`、`机构`、`专区`。
4. `规则集`、`版本槽位`、`frameCode`、`loader/core` 等内部实现词默认不出现在业务主路径，只在高级区、详情或技术视图中出现。
5. `API注册` 名称保持不变，但新建与编辑流程必须改造成业务可理解的向导式流程。
6. `运行统计` 仅保留 `触发统计` 与 `下降提醒` 两类业务核心视图，`匹配失败` 不进入业务提醒。
7. 灰度范围在业务侧仍只开放到 `机构` 维度，不开放人员级灰度。
8. 高频流程必须具备结果闭环：规则有预览、API有在线测试、发布有影响确认、统计有发布前后对比。
9. 低频复杂能力默认后置到 `高级配置` 或各页面的高级区，不再与高频任务同层暴露。
10. 旧原型页面和路由允许迁移期保留兼容别名，但不得继续在主导航中暴露低层技术菜单。
11. 技术人员继续负责 SDK 制品版本和槽位维护；菜单能力配置（菜单是否启用智能提示 / 智能作业、菜单槽位、机构范围、IP 试点）属于特殊权限，默认仅业务超管或被单独授权的菜单开通管理员可操作，普通业务配置人员无权修改。
12. 稳定入口 `cc-sdk-loader.js`、菜单级版本控制、页面启用控制和审计口径维持现状，不在本计划中改变。

## 4. 阶段计划（按顺序推进）

### 4.1 Phase 0：信息架构、术语与角色分层冻结

目标：冻结业务侧新导航、页面承载关系、术语口径和角色可见性，避免后续实现反复返工。

交付物：
1. 新旧导航映射清单：
   - `总览 -> 我的工作台`
   - `页面资源中心 + 页面启用策略 -> 菜单管理`
   - `规则模板中心 -> 智能提示` 内部复用入口
   - 旧发布入口 + `SDK版本中心` 的业务侧可见部分 -> `发布与灰度`
   - `审计与指标中心 -> 运行统计`
   - `预处理器中心 + 角色管理 + 平台级参数 -> 高级配置`
2. 新一级菜单、页面标题、页面描述、帮助文案统一稿。
3. 平台术语与业务展示术语对照表，作为页面命名、按钮文案、卡片标题、步骤标题的前置检查项。
4. 角色可见性矩阵：业务人员、高级运营/配置人员、平台管理员。
5. 工作台卡片口径和菜单详情聚合字段基线。

退出条件：
1. 菜单结构、页面承载关系与术语说明评审通过。
2. 业务主路径术语已与平台术语分层，不再在业务主路径混入历史旧词或平台内部口径。
3. 角色分层与高级配置收口边界明确。
3. 旧入口迁移和兼容策略确认完成。

### 4.2 Phase 1：我的工作台与菜单管理落地

目标：先解决“从哪里开始”和“如何围绕页面做事”两大核心问题。

交付物：
1. `我的工作台`：待处理、最近修改、运行提醒、常用入口、业务看板。
2. `菜单管理` 列表：支持按 `专区 / 菜单名称 / 页面名称 / 机构 / 启用状态` 快速筛选。
3. 菜单详情聚合视图：`菜单上下文`、`当前页面信息`、`运行情况`。
4. 菜单详情内为当前页面直接暴露 `新增提示规则`、`新增作业配置`、查看关联 API、查看最近触发趋势等动作。

退出条件：
1. 业务人员可从工作台进入菜单管理并快速定位目标菜单与页面。
2. 业务人员在菜单详情中可判断当前页面已开通哪些能力、影响哪些机构、近期运行是否正常。
3. 高频动作不再要求跨多个一级菜单来回切换。

### 4.3 Phase 2：高频配置流程向导化

目标：完成智能提示、智能作业和 API注册 三类高频流程的业务化重构。

交付物：
1. 智能提示四步流程：`选页面 -> 选场景模板 -> 改内容 -> 预览并保存`。
2. 智能提示列表重构：突出适用页面、提示场景、启用状态、最近触发、最近修改人。
3. 智能作业页面化入口：从页面进入场景配置，区分自动触发、预览确认、悬浮再次触发。
4. 智能作业场景卡片化表达，复杂编排与节点细节进入高级区。
5. API注册五步流程：`选用途 -> 填基础信息 -> 填参数示例 -> 在线测试 -> 保存`。
6. API注册支持复制已有 API、从常用模板创建、保存后展示引用关系。

退出条件：
1. 业务人员可在不理解底层实现细节的情况下完成规则、作业、API 的新建和编辑。
2. 保存前均有可理解的预览、测试或校验反馈。
3. 高频配置项前置，复杂项后置，不影响主路径效率。

### 4.4 Phase 3：发布与灰度、运行统计闭环

目标：让业务人员明确“改了什么、影响哪里、何时生效、上线后结果如何”。

交付物：
1. `发布与灰度` 四块视图：`待发布内容`、`生效范围`、`试点发布`、`发布记录`。
2. 发布流程：选 `专区/菜单`、自动汇总待发布内容、选机构范围、选立即/定时生效、发布前影响确认。
3. 生效结果表达统一为 `线上版本`、`试点版本`、`生效范围`，弱化内部技术实现词。
4. `运行统计` 的 `触发统计` 视图：总触发次数、页面排行、规则排行、机构分布、近 7/30 天趋势、发布前后对比。
5. `运行统计` 的 `下降提醒` 视图：页面、规则、机构范围、当前周期、对比周期、下降比例、首次发现时间。

退出条件：
1. 业务人员在发布前可清楚看到影响页面数、影响机构数和生效时间。
2. 业务人员在发布后可用业务视角验证触发趋势是否正常。
3. 下降提醒能够聚焦真正需要关注的问题，不把技术噪音推给业务。

### 4.5 Phase 4：高级配置收口与权限管理

目标：完成低频复杂能力后置、按角色开放和旧入口主导航下线。

交付物：
1. `高级配置` 入口与信息架构，收纳预处理器、高级识别参数、接口高级配置、底层运行参数、权限管理。
2. 高级项的场景说明文案：什么场景才需要改、谁可以改、改动风险是什么。
3. 角色可见性裁剪：业务人员默认不见高风险配置，高级运营仅见必要项，平台管理员可见全部。
4. 旧一级菜单从主导航下线，保留兼容跳转或页面内深链。

退出条件：
1. 业务人员默认只接触高频主路径。
2. 高级能力在需要时“找得到、看得懂、改得少”。
3. 角色权限与导航可见性一致，不出现误入高风险模块。

### 4.6 Phase 5：试点灰度、培训与扩面准备

目标：在真实业务路径中验证新信息架构和新流程是否提升自助配置效率。

交付物：
1. 试点菜单和试点机构清单。
2. 业务操作手册与角色培训材料。
3. 旧入口迁移提醒、FAQ 与发布沟通稿。
4. 试点运行快报：任务完成效率、配置漏项、发布风险、统计可用性、用户反馈。

退出条件：
1. 试点用户可独立完成“找页面、配规则、接接口、做发布、看结果”全流程。
2. 主路径关键问题闭环，具备扩面条件。
3. Go/No-Go 评审通过。

## 6. 原型实现清单（当前 Admin Web）

### 6.1 本轮原型必须体现

1. 主导航切换为新的八个一级菜单，不再直接暴露 `页面资源中心`、`页面启用策略`、`SDK版本中心`、`规则模板中心`、旧发布入口、`审计与指标中心` 等低层技术入口。
2. `DashboardPage` 改造成 `我的工作台`，展示业务待办、最近修改、下降提醒、常用入口和业务看板。
3. `PageResourcesPage` 与 `PageActivationPoliciesPage` 共同承载 `菜单管理` 能力，菜单详情要能聚合启用范围、已开通能力、关联 API 与运行情况。
4. `RulesPage` 与 `RuleTemplateCenterPage` 共同承载 `智能提示`，模板中心只作为复用入口，不再与规则列表同级争抢主入口。
5. `JobScenesPage` 承载 `智能作业`，必须支持从页面上下文进入，而不是只从底层场景清单进入。
6. `InterfacesPage` 承载 `API注册`，必须体现向导、在线测试、模板复用和引用关系。
7. `PublishPage` 与 `SdkVersionCenterPage` 的业务侧可见能力共同承载 `发布与灰度`，突出待发布内容、生效范围和试点机构。
8. `AuditMetricsPage` 承载 `运行统计`，视图必须收敛成 `触发统计 + 下降提醒`。
9. `PreprocessorsPage`、`RolesPage` 及平台级参数类能力统一纳入 `高级配置`，并按角色做可见性控制。
10. 旧路由允许保留兼容访问，但只能作为迁移期别名，不再在主导航中出现。

### 6.2 原型允许简化

1. 各页面仍可使用 Mock 数据，不强依赖真实后端。
2. 菜单详情中的“引用关系”“影响范围”“最近触发趋势”允许先基于聚合 Mock 数据实现。
3. 高级配置可先使用单页分区或二级 Tab 承载，不要求一次性拆成完整独立子系统。
4. 发布影响确认可先以汇总面板表达，不要求一开始就具备真实差异比对引擎。
5. 运行统计的趋势与提醒口径可先由前端演示数据驱动，后续再接真实统计源。

## 7. 验收口径

### 7.1 功能验收

1. 业务人员进入系统后，首页能够直接看到待处理事项、常用入口和运行提醒。
2. 业务人员可通过 `菜单管理` 快速找到目标菜单与页面，并在菜单详情中判断当前启用范围、已开通能力、关联 API 和运行情况。
3. 智能提示支持按页面新建、模板复用、预览保存，并将复杂条件放到高级区。
4. 智能作业支持按页面进入、选择场景卡片、明确触发方式和预览确认方式。
5. API注册支持向导式新建、在线测试、复制创建和引用关系查看。
6. 发布与灰度支持按专区/菜单汇总待发布内容、选择机构范围、选择生效时间并确认影响范围。
7. 运行统计只展示触发统计和下降提醒，不把匹配失败作为业务提醒项。
8. 高级配置默认不打扰业务人员，且按角色控制可见性。

### 7.2 业务体验验收

1. 用户知道从哪里开始：业务主入口明确，首页和菜单管理承担导流职责。
2. 用户知道下一步做什么：高频流程均采用向导或页面内明显动作入口。
3. 用户知道有没有配对、有没有生效：规则预览、API测试、发布影响确认、运行统计形成闭环。
4. 低频复杂能力不打扰高频操作：高级配置后置、术语分层、技术词弱化。
5. 业务术语保持稳定，不额外制造新的学习成本。

### 7.3 边界验收

1. P0 仍只支持机构级灰度，不支持人员级灰度。
2. `匹配失败` 不进入业务提醒，只保留在内部明细或高级视图。
3. 技术底层架构、JSSDK 分包策略、页面识别与运行时解析口径不因本轮业务侧重构而改变。
4. 旧入口如保留兼容访问，必须明确标识迁移路径，且不再暴露在主导航。
5. 高级配置只能按角色开放，不允许所有业务角色默认看见全部平台能力。

## 8. 主要风险与缓解

| 风险 | 影响 | 缓解措施 |
| --- | --- | --- |
| 新旧导航映射不清 | 用户找不到原有功能入口 | 先冻结映射清单，迁移期保留兼容跳转和页面内提示 |
| 菜单详情聚合字段不完整 | 菜单管理无法真正成为主入口 | Phase 0 先冻结聚合字段基线，优先补齐关联 API、能力状态、运行趋势 |
| 结果闭环数据来源不足 | 规则预览、API测试、发布影响确认体验不完整 | 原型阶段允许 Mock 聚合，先固定展示口径和页面结构 |
| 高级配置收口过度 | 高级运营无法完成复杂配置 | 先按角色分层而不是一刀切下线，保留必要深链入口 |
| 运行统计口径不一致 | 发布效果无法判断，提醒失真 | 先固定统计指标定义和提醒门槛，再接数据源 |
| 术语与旧文案混用 | 业务人员理解成本上升 | 页面标题、帮助文案、导航标签统一收口到业务侧口径 |
| 旧入口长期保留 | 新信息架构无法真正被采用 | 在试点通过后逐步下线旧导航，仅保留必要兼容深链 |

## 9. 启动清单（可直接执行）

1. 冻结一级菜单、页面标题、页面描述和新旧入口映射清单。
2. 冻结 `我的工作台` 五类卡片的数据口径和优先级。
3. 冻结 `菜单管理` 的筛选条件、菜单详情结构和页面内快捷动作。
4. 冻结智能提示四步向导、智能作业页面化入口、API注册五步向导的步骤定义与阻断校验点。
5. 冻结发布与灰度的汇总字段、影响确认字段、机构级试点口径和立即/定时生效表达。
6. 冻结运行统计的指标字典、趋势窗口、下降提醒口径和展示字段。
7. 冻结高级配置的收口范围、角色可见性和场景说明模板。
8. 对齐原型页面承载分工、路由兼容方案和迁移期提示策略。
9. 建立阶段评审机制：每个 Phase 输出“交付状态 + 风险状态 + 退出结论”。
