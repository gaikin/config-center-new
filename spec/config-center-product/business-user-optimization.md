# 面向业务人员的产品优化分析与交互重构建议

> 所属产品：营小助配置中心  
> 文档定位：业务侧体验优化专题  
> 更新日期：2026-03-14  
> 适用对象：产品、业务配置人员、前端设计与实现团队

## 1. 背景与目标

营小助配置中心的主要使用者不是纯技术人员，而是**有少量技术背景的业务人员**。  
这类用户通常理解页面、规则、接口、发布、灰度等概念，但不适合直接面对过多底层实现词、分散入口和复杂配置流程。

因此，本轮优化的目标不是简单隐藏技术能力，而是做到以下四点：

1. 让业务人员知道**从哪里开始**。
2. 让业务人员知道**下一步该做什么**。
3. 让业务人员在配置后能快速确认**有没有配对、有没有生效**。
4. 让低频复杂能力不打扰高频日常操作。

## 2. 已确认的优化原则

### 2.1 术语原则

1. 保留行内已经形成共识的术语，避免为了“更业务化”而制造新学习成本。
2. `API注册` 保留原名称，不做替换。
3. `页面`、`规则`、`模板`、`发布`、`灰度`、`机构`、`专区` 等词继续保留。
4. `规则集`、`版本槽位`、`frameCode`、`loader/core` 等内部实现词应弱化，只在高级配置或详情中出现。

### 2.2 交互原则

1. 主路径按“业务任务”组织，不按“系统对象”拆散。
2. 高频任务前置，低频复杂能力后置。
3. 每个核心流程都要有示例、校验、预览或测试闭环。
4. 规则建立强调“快速复用”，而不是让用户先学习模板体系。
5. 发布与灰度重点表达“改了什么、影响哪里、何时生效”。

### 2.3 统计与提醒原则

1. 运行类菜单统一命名为 `运行统计`。
2. `匹配失败` 不进入提醒，只保留在明细或内部统计中。
3. 业务侧提醒仅保留“**触发量明显下降**”这类真正值得关注的问题。
4. `运行统计` 重点展示 `触发统计` 与 `下降提醒`，不做技术告警台。

## 3. 当前核心问题清单

| 优先级 | 问题 | 当前表现 | 对业务人员的影响 | 优化方向 |
|------|------|------|------|------|
| P1 | 入口不按业务任务组织 | 页面资源、启用策略、智能提示、作业、发布等分散在不同菜单 | 用户知道要做事，但不知道先点哪里 | 用“菜单管理”为主入口，围绕业务动作重组菜单 |
| P1 | 同一件事被拆成多步多页面 | 配一个页面通常要跨多个模块反复切换 | 学习成本高，容易漏配置 | 把“按页面开通能力”做成主路径 |
| P1 | 用户会配，但不敢配 | 缺少测试、预览、发布影响确认 | 反复试错，不敢独立操作 | 在规则、API、发布环节补齐校验闭环 |
| P1 | API 注册虽高频，但流程偏技术化 | 字段多、概念杂、缺少模板和在线验证 | 依赖经验，复制错误概率高 | 保留名称，改为向导式流程 |
| P1 | 日常能力和平台能力混在一起 | 业务人员和管理员看到相似菜单层级 | 误入低频复杂模块，增加心理负担 | 高频入口前置，高级能力统一收口 |
| P2 | 规则建立偏“从零配置” | 用户需要理解较多条件和配置项 | 规则搭建速度慢，不易复用 | 以场景模板为起点，改少量内容即可保存 |
| P2 | 首页不是业务视角 | 首页更像平台建设状态，不像业务工作台 | 进入系统后不知道今天该处理什么 | 首页改成待办、最近修改、运行提醒、常用入口 |
| P2 | 术语层次不清 | 页面、菜单、专区、规则、模板、版本控制混在一起 | 不容易理解配置边界 | 保留通用术语，弱化内部术语，增强说明文案 |
| P2 | 发布与灰度表达偏底层 | 用户更容易看到“版本”而不是“影响范围” | 发布前难以判断风险 | 突出改动内容、生效范围、试点机构、回退记录 |
| P3 | 缺少新手引导和推荐路径 | 首次进入没有清晰动作指引 | 上手慢，容易跳错模块 | 在工作台和关键页面增加快捷任务入口 |

## 4. 优化后的一级菜单建议

建议将业务侧主菜单收敛为以下结构：

1. `我的工作台`
2. `菜单管理`
3. `智能提示`
4. `智能作业`
5. `API注册`
6. `发布与灰度`
7. `运行统计`
8. `高级配置`

### 4.1 菜单定位说明

| 菜单 | 主要作用 | 主要用户 |
|------|------|------|
| 我的工作台 | 看待办、最近修改、运行提醒、常用入口 | 所有业务角色 |
| 菜单管理 | 选菜单、定位页面、看已配置能力 | 业务配置人员 |
| 智能提示 | 建规则、套模板、看命中情况 | 业务配置人员 |
| 智能作业 | 配作业场景、触发方式、执行控制 | 业务配置人员/高级运营 |
| API注册 | 维护业务接口、做在线测试、查看引用关系 | 业务配置人员 |
| 发布与灰度 | 看待发布内容、选生效范围、做机构试点 | 业务配置人员/管理人员 |
| 运行统计 | 看触发趋势和下降提醒 | 业务配置人员/管理人员 |
| 高级配置 | 放低频复杂能力，默认不打扰 | 高级运营/平台管理员 |

## 5. 核心页面优化建议

### 5.1 我的工作台

工作台的目标不是展示平台建设状态，而是回答“**今天要处理什么**”。

建议包含以下内容：

1. `我的待处理`：待发布、待确认、待补充配置。
2. `我最近改过`：最近编辑的页面、规则、API。
3. `运行提醒`：仅展示触发量明显下降的页面/规则。
4. `常用入口`：新建提示规则、复制已有规则、注册 API、查看发布结果。
5. `业务看板`：已启用页面数、提示触发量、作业执行量、下降提醒数。

### 5.2 菜单管理

`菜单管理` 应作为业务人员的主入口，很多配置动作都从“先找到菜单，再定位页面”开始。

建议：

1. 菜单列表支持按 `专区 / 菜单名称 / 页面名称 / 机构 / 启用状态` 快速筛选。
2. 菜单详情收敛为“菜单上下文 + 当前页面信息 + 页面运行情况”。
3. 在菜单详情中承接的当前页面信息块直接展示：
   - 是否启用智能提示
   - 是否启用智能作业
   - 当前绑定的提示规则
   - 当前关联的 API
   - 最近触发趋势
4. 在菜单详情中为当前页面提供明显动作入口：`新增提示规则`、`新增作业配置`。

### 5.3 智能提示

业务人员关注的是“这个页面怎么提示、什么时候提示”，而不是先理解模板体系。

建议：

1. 首屏先展示规则列表，不先强调模板中心。
2. 列表重点展示：适用页面、提示场景、启用状态、最近触发情况、最近修改人。
3. 新建流程改成四步：
   - 选页面
   - 选场景模板
   - 改内容
   - 预览并保存
4. 模板中心定位为“快捷复用来源”，不是主学习入口。
5. 高频配置项放在第一屏，复杂条件放入 `高级条件`。

### 5.4 智能作业

智能作业应延续提示的页面化思路，避免和提示割裂。

建议：

1. 作业场景优先从页面进入，不建议业务人员先从底层节点编排开始。
2. 用场景卡片表达“查询并回填”“多字段辅助录入”“预览确认后执行”等常见作业类型。
3. 复杂编排、节点细节、风险参数统一收进高级区。
4. 页面上明确区分：
   - 是否自动触发
   - 是否需要预览确认
   - 是否支持悬浮入口再次触发

### 5.5 API注册

`API注册` 保留原名称，但交互应更贴近业务配置习惯。

建议：

1. 列表页重点展示：接口用途、调用方式、状态、引用关系。
2. 新建流程改成五步：
   - 选用途
   - 填基础信息
   - 填参数示例
   - 在线测试
   - 保存
3. 基础信息与高级配置分层展示，避免第一次就暴露全部字段。
4. 支持 `复制已有 API`、`从常用模板创建`。
5. 保存后直接展示“已被哪些页面/规则/作业引用”。

### 5.6 发布与灰度

业务人员最关心的是“这次改了什么，会影响哪些页面、哪些机构，什么时候生效”。

建议：

1. 页面重点展示四块：`待发布内容`、`生效范围`、`试点发布`、`发布记录`。
2. 发布流程建议：
   - 选 `专区 / 菜单`
   - 系统自动汇总待发布内容
   - 选范围：全部机构或试点机构
   - 选时间：立即生效或定时生效
   - 发布前确认影响页面数和机构数
3. 灰度范围在业务侧只控制到 `机构`，不下钻到人员。
4. 页面表达以 `线上版本`、`试点版本`、`生效范围` 为主，弱化内部实现词。

### 5.7 运行统计

`运行统计` 应该是业务分析页面，而不是技术故障面板。

建议保留两个核心视图：

1. `触发统计`
   - 总触发次数
   - 按页面触发排行
   - 按规则触发排行
   - 按机构触发分布
   - 近 7 天 / 近 30 天趋势
   - 发布前后触发对比
2. `下降提醒`
   - 页面名称
   - 规则名称
   - 机构范围
   - 最近周期触发次数
   - 对比周期触发次数
   - 下降比例
   - 首次发现时间

说明：

1. `匹配失败` 不做提醒。
2. 只有“触发量明显下降”进入业务侧提醒。
3. 技术排查明细可以保留在内部日志或高级视图中，不进入业务主视角。

### 5.8 高级配置

高级配置的目标不是提高曝光度，而是在需要时“找得到、看得懂、改得少”。

建议放入：

1. 预处理器
2. 高级识别参数
3. 接口高级配置
4. 底层运行参数
5. 权限管理与平台级配置

使用原则：

1. 默认不放在主业务路径。
2. 按角色开放。
3. 每个高级项都写清楚“什么场景才需要改”。
4. 可以放在普通页面高级区的，不单独做一级菜单。

## 6. 角色分层建议

| 角色 | 建议可见菜单 | 说明 |
|------|------|------|
| 业务人员 | 我的工作台、菜单管理、智能提示、智能作业、API注册、发布与灰度、运行统计 | 满足日常配置与发布 |
| 高级运营/配置人员 | 在业务人员基础上增加部分高级配置权限 | 处理复杂条件和特殊接入 |
| 平台管理员 | 可见全部菜单 | 负责平台控制、底层参数和权限能力 |

## 7. 建议优先级

### 7.1 第一优先级

1. 菜单重组，建立业务任务主路径。
2. 将 `菜单管理` 做成业务人员主入口。
3. 将 `智能提示`、`API注册`、`发布与灰度` 改成向导式流程。

### 7.2 第二优先级

1. 首页改成真正的业务工作台。
2. 建立规则预览、API 在线测试、发布影响确认等结果闭环。
3. 将 `运行统计` 收敛成 `触发统计 + 下降提醒`。

### 7.3 第三优先级

1. 统一说明文案和术语口径。
2. 按角色收口高级配置。
3. 持续补充常见模板和快捷复用入口。

## 8. 推荐落地顺序

建议按以下顺序推进：

1. **先改入口**：先让用户知道该去哪里做事。
2. **再改主流程**：把规则、API、发布三个高频流程改顺。
3. **再补结果闭环**：把预览、测试、发布确认、统计补齐。
4. **最后做角色收口**：把高级能力逐步挪出主视角。

## 9. 结论

营小助配置中心下一阶段的重点，不是继续叠加功能，而是把现有能力重组成一套**业务人员更容易理解、更容易操作、更容易确认结果**的产品路径。

核心判断如下：

1. 主入口应是 `菜单管理`，不是技术对象清单。
2. `API注册` 名称应保留，但流程必须更顺手。
3. `运行统计` 只保留业务有感知的统计和提醒。
4. 高频任务必须向导化，低频复杂能力必须角色化、后置化。

当业务人员能够在平台中快速完成“找页面、配规则、接接口、做发布、看结果”这五件事，平台的自助能力和推广效率才会真正提升。
