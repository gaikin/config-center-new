# JSSDK 缓存方案评审稿（开阳系统）

> 版本：v1.4  
> 日期：2026-03-17  
> 目的：用于领导评审，围绕两种缓存方案进行对比，并明确评审关注点与待拍板事项

## 1. 背景

当前链路为：

`打开菜单 -> 注入 JSSDK -> iframe 内 JSSDK 初始化 -> 从服务端加载页面运行时配置 -> JSSDK ready`

当前问题：

1. 每次打开菜单都要重新加载页面运行时配置。
2. 重复进入同一菜单时，等待时间偏长。
3. 在网络波动场景下，打开耗时不稳定。

本次评审的前提是：

1. 开阳系统是重要生产系统。
2. 页面白屏、卡死、长时间无响应均不可接受。
3. 方案优先目标不是“性能最大化”，而是“在保证稳定的前提下提升打开菜单速度”。

## 2. 方案解决目标

本次方案希望解决以下问题：

1. 减少重复进入同一菜单时对服务端的重复加载。
2. 缩短重复打开菜单时的等待时间。
3. 降低网络波动对菜单打开耗时的影响。
4. 在提升性能的同时，不引入新的生产稳定性风险。
5. 保证缓存异常时不影响业务主流程。

## 3. 术语口径

为避免概念混乱，统一采用以下术语：

1. `开阳系统`
2. `JSSDK`
3. `运行时配置`
4. `页面运行时配置`
5. `开阳系统共享缓存`
6. `配置变更校验`
7. `ETag 校验`
8. `主流程`
9. `降级处理`
10. `缓存回退`
11. `快速切换开关`

## 4. 方案 A：开阳系统共享缓存 + 配置变更校验

### 4.1 方案说明

方案 A 由开阳系统提供共享缓存能力，JSSDK 在进入菜单时优先读取开阳系统共享缓存，并结合配置变更校验决定是否继续复用。

基本思路：

1. 服务端提供版本化的页面运行时配置。
2. JSSDK 优先读取开阳系统共享缓存。
3. 命中缓存后先做配置变更校验。
4. 若缓存不可用或校验失败，则回源服务端重新加载。
5. 缓存或网络异常时，进入降级处理，不阻塞主流程。

### 4.2 方案优点

1. 更符合生产系统统一治理要求。
2. 更利于统一清理、统一开关、统一安全控制。
3. 更适合作为生产级共享缓存方案。
4. 与当前开阳系统已落地的持久化 API 能力方向一致。

### 4.3 方案风险

1. 需要确认读取耗时和稳定性是否满足生产要求。
2. 需要确认异常时是否可以快速降级。
3. 需要确认是否支持快速切换开关。
4. 需要确认对多页面、多 iframe 场景的共享边界。

## 5. 方案 B：IndexedDB + 配置变更校验

### 5.1 方案说明

方案 B 由 iframe 内 JSSDK 直接使用浏览器 `IndexedDB` 保存页面运行时配置，并结合配置变更校验决定是否继续复用。

基本思路：

1. JSSDK 在本地保存页面运行时配置。
2. 重复进入菜单时优先读取本地缓存。
3. 命中缓存后先做配置变更校验。
4. 若缓存不可用或校验失败，则回源服务端重新加载。

### 5.2 方案优点

1. 接入相对简单。
2. 理论本地读取速度可能较快。

### 5.3 方案风险

1. 浏览器本地数据库能力在不同容器中的稳定性边界不够清晰。
2. 即使问题只发生在当前页面，也可能构成生产事故。
3. 不符合当前统一收敛到开阳系统能力的方向。
4. 对关键生产系统而言，整体风险偏高。

## 6. 方案对比

### 6.1 稳定性

1. 方案 A 更符合关键生产系统的稳定性要求。
2. 方案 B 在理论上可行，但页面级异常风险更难接受。

### 6.2 治理能力

1. 方案 A 更利于统一治理、统一开关和统一控制。
2. 方案 B 更偏浏览器本地能力，治理边界较弱。

### 6.3 实施复杂度

1. 方案 A 依赖开阳系统持久化 API 的能力边界和稳定性确认。
2. 方案 B 前端接入更直接，但不符合当前生产风险偏好。

### 6.4 与当前方向一致性

1. 方案 A 与开阳系统现有能力和生产治理方向一致。
2. 方案 B 与当前方向一致性较弱。

## 7. 评审关注点

本次评审建议重点关注以下问题：

1. 开阳系统共享缓存的读取耗时是否满足生产要求。
2. 开阳系统共享缓存异常时，是否能够立即进入降级处理。
3. 是否具备快速切换开关。
4. 服务端是否已具备配置变更校验能力。
5. 服务端是否已具备页面运行时配置版本化能力。
6. 缓存方案是否会对主流程造成新的稳定性风险。

## 8. 预估性能提升

本次性能预估主要针对“重复进入同一菜单”的场景，不针对首次进入菜单。

### 8.1 预估前提

按当前链路判断，菜单打开耗时主要包含：

1. JSSDK 初始化
2. 页面运行时配置请求
3. 页面运行时配置解析与处理

其中最适合通过缓存优化的是“页面运行时配置请求”这一段。

### 8.2 预估收益

在采用“开阳系统共享缓存 + 配置变更校验”后，建议按以下区间理解性能收益：

1. 重复进入同一菜单时，页面运行时配置加载阶段预计可缩短 `60% ~ 85%`
2. 折算到整体菜单打开耗时，预计可缩短 `20% ~ 50%`
3. 常见办公网络下，体感上预计可缩短约 `200ms ~ 800ms`
4. 在网络延迟较高场景下，收益会更明显

### 8.3 预估说明

需要强调：

1. 上述收益主要来自减少重复加载，而不是优化首次加载。
2. 若服务端当前返回耗时本身较低，则收益会偏保守。
3. 若页面运行时配置体积较大或网络波动明显，则收益会更明显。
4. 最终收益仍需以开阳系统真实环境压测和灰度数据为准。

## 9. 服务端配合要求

为了让缓存方案真正可控，服务端需要配合以下能力：

1. 页面运行时配置版本化。
2. 支持配置变更校验。
3. 配置变更后能够及时让旧缓存失效。
4. 支持关闭某类缓存或某类回退策略。

## 10. 实施路径建议

建议分两步实施：

### 10.1 第一步

先落地：

1. 配置变更校验
2. 服务端版本化页面运行时配置
3. 快速切换开关

### 10.2 第二步

在第一步稳定后，再评估是否接入：

1. 开阳系统共享缓存
2. 开阳系统共享缓存灰度验证
3. 开阳系统共享缓存监控与降级处理

## 11. 待拍板事项

本次评审建议领导重点拍板以下事项：

1. 是否将方案 A 作为生产主方向。
2. 是否明确不将方案 B 作为生产主方案。
3. 是否同意先完成服务端能力补齐，再接入开阳系统共享缓存。
4. 是否要求共享缓存能力必须具备快速切换开关和降级处理能力。

## 12. 当前建议

当前建议如下：

1. 优先将方案 A 作为生产主方向。
2. 不建议将方案 B 作为生产主方案。
3. 先补齐服务端版本化和配置变更校验能力，再推进共享缓存接入。
