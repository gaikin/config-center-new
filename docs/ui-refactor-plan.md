# 配置中心前端 UI 重构计划（业务主路径版）

> 更新日期：2026-03-14  
> 文档版本：v2.0  
> 适用范围：`config-center` 当前 React Admin Web 原型  
> 对齐基线：`spec/config-center-product/plan.md`、`spec/config-center-product/business-user-optimization.md`  
> 技术栈：React 18 + Ant Design 5 + TypeScript + Vite

---

## 1. 文档目标

本文件用于将前端原型的 UI 改造计划同步到最新业务侧实施计划，重点回答三个问题：

1. 现有原型的导航和页面承载与新 `plan.md` 差在哪里。
2. 前端应该如何把“按系统对象组织”的原型，改造成“按业务任务组织”的工作平台。
3. 具体到导航、页面、组件、路由和验收，前端改造应如何分阶段推进。

本文件不重复定义后端数据模型、JSSDK 运行时、发布产物与治理底层规则；相关内容仍以 `spec/config-center-product/*.md` 为准。

## 2. 当前实现基线

### 2.1 当前主导航

当前 `AppShell` 中主导航仍是技术对象型结构：

```typescript
[
  { key: "/", label: "总览" },
  { key: "/page-resources", label: "页面资源中心" },
  { key: "/sdk-version-center", label: "SDK版本中心" },
  { key: "/page-activation", label: "页面启用策略" },
  { key: "/interfaces", label: "API注册" },
  { key: "/preprocessors", label: "预处理器中心" },
  { key: "/rules", label: "智能提示" },
  { key: "/rule-templates", label: "规则模板中心" },
  { key: "/job-scenes", label: "智能作业" },
  { key: "/governance", label: "治理工作台" },
  { key: "/audit-metrics", label: "审计与指标中心" },
  { key: "/roles", label: "角色管理" }
]
```

### 2.2 当前实现特征

1. 首页还是“总览”，强调阶段计划和平台建设状态，不是业务工作台。
2. 页面资源、页面启用策略、规则模板、治理、指标等能力被拆散到多个一级入口。
3. 高频业务动作缺少统一主路径，用户需要在多个技术菜单间来回切换。
4. 页面间交互模式不统一：有的用 Modal，有的用 Drawer，有的强调表格 CRUD，有的强调节点编排，但缺少一致的任务流程框架。
5. 低频复杂能力与高频配置能力处于同一层级，容易增加业务人员心理负担。

## 3. 目标信息架构

### 3.1 目标一级菜单

前端主导航统一切换为以下八个一级菜单：

1. `我的工作台`
2. `菜单管理`
3. `智能提示`
4. `智能作业`
5. `API注册`
6. `发布与灰度`
7. `运行统计`
8. `高级配置`

### 3.2 新旧页面承载映射

| 目标菜单 | 前端承载方式 | 主要复用页面 |
| --- | --- | --- |
| 我的工作台 | 直接替换现有首页 | `DashboardPage` |
| 菜单管理 | 聚合页面资源与页面启用策略 | `PageResourcesPage` + `PageActivationPoliciesPage` |
| 智能提示 | 规则列表为主、模板内收 | `RulesPage` + `RuleTemplateCenterPage` |
| 智能作业 | 页面化入口 + 场景配置 | `JobScenesPage` |
| API注册 | 保留独立一级菜单 | `InterfacesPage` |
| 发布与灰度 | 聚合治理工作台与业务可见版本能力 | `GovernancePage` + `SdkVersionCenterPage` |
| 运行统计 | 收敛为业务视角统计 | `AuditMetricsPage` |
| 高级配置 | 收纳低频复杂能力 | `PreprocessorsPage` + `RolesPage` + 后续平台参数页 |

### 3.3 路由策略

1. 新主导航应切到业务菜单路由，不再直接暴露当前低层技术路由。
2. 旧路由在迁移期允许保留为兼容别名，用于深链和历史访问，但不出现在主导航中。
3. 页面标题、面包屑、帮助文案必须全部切换到业务侧口径，不能继续混用“治理工作台”“审计与指标中心”等旧名称。

建议目标路由：

```typescript
[
  { key: "/", label: "我的工作台" },
  { key: "/page-management", label: "菜单管理" },
  { key: "/prompts", label: "智能提示" },
  { key: "/jobs", label: "智能作业" },
  { key: "/interfaces", label: "API注册" },
  { key: "/publish", label: "发布与灰度" },
  { key: "/stats", label: "运行统计" },
  { key: "/advanced", label: "高级配置" }
]
```

## 4. 设计原则

### 4.1 任务优先

1. 用户先看到“我要做什么”，再看到“系统里有哪些对象”。
2. 菜单管理承担业务主入口职责，围绕菜单承接页面、提示、作业、API、运行情况。
3. 高频动作必须在页面内或列表页显式可达，不能要求用户先理解底层模型。

### 4.2 结果闭环

1. 智能提示必须有预览。
2. API注册必须有在线测试。
3. 发布与灰度必须有影响确认。
4. 运行统计必须能验证发布后是否生效、趋势是否下降。

### 4.3 复杂能力后置

1. 默认先展示高频配置项。
2. 高风险或低频配置进入 `高级条件`、`高级设置` 或 `高级配置` 一级菜单。
3. 高级区要说明“什么场景需要改”，不能只堆字段。

### 4.4 术语收口

1. 保留 `API注册`、`页面`、`规则`、`模板`、`发布`、`灰度`、`机构`、`专区`。
2. 弱化 `规则集`、`版本槽位`、`frameCode`、`loader/core` 等技术词。
3. 所有页面说明文案统一强调业务动作和影响结果。

## 5. 关键改造项

### 5.1 AppShell 与导航壳

**目标**

- 将 `AppShell` 从“实施导航”改为“业务工作导航”。
- 头部、副标题、侧边栏和移动 Drawer 全部同步新信息架构。

**改造点**

1. 替换 `navItems` 为新的八个一级菜单。
2. Header 副标题从 `Phase 0-5 | WP-A~WP-F 实施导航` 改为业务主路径说明。
3. 根据角色控制菜单可见性，业务人员默认不展示高风险配置入口。
4. 导航选中逻辑需兼容新主路由和旧兼容路由。

### 5.2 我的工作台

**目标**

- 将首页从“平台进度概览”改成“今天要处理什么”。

**必须包含**

1. `我的待处理`：待发布、待确认、待补充配置。
2. `我最近改过`：最近编辑的页面、规则、API。
3. `运行提醒`：仅展示触发量明显下降。
4. `常用入口`：新建提示规则、复制已有规则、注册 API、查看发布结果。
5. `业务看板`：已启用页面数、提示触发量、作业执行量、下降提醒数。

**当前页面改造基础**

- 基于 `DashboardPage` 改造，不再突出 Phase/WP 文案。

### 5.3 菜单管理

**目标**

- 让用户从“页面”出发理解启用范围、能力状态和下一步动作。

**必须包含**

1. 列表筛选：`专区 / 页面名称 / 机构 / 启用状态`。
2. 详情四块：`基本信息`、`启用范围`、`已开通能力`、`运行情况`。
3. 页面内直接动作：`新增提示规则`、`新增作业配置`、查看关联 API。
4. 明确展示：是否启用智能提示、是否启用智能作业、当前绑定规则、当前关联 API、最近触发趋势。

**当前页面改造基础**

- `PageResourcesPage` 承担页面列表与详情主体。
- `PageActivationPoliciesPage` 的启用策略能力需要并入菜单详情或页面内分区。

### 5.4 智能提示

**目标**

- 让用户先看到“哪些页面、哪些场景、最近效果如何”，而不是先学习模板体系。

**必须包含**

1. 规则列表前置，突出适用页面、提示场景、启用状态、最近触发、最近修改人。
2. 新建流程改为四步：`选页面 -> 选场景模板 -> 改内容 -> 预览并保存`。
3. 模板中心作为“快捷复用来源”，不再作为主一级入口。
4. 高频配置项放在第一屏，复杂条件进入 `高级条件`。

**当前页面改造基础**

- `RulesPage` 作为主页面。
- `RuleTemplateCenterPage` 并入 `RulesPage` 的二级入口、抽屉或 Tab。

### 5.5 智能作业

**目标**

- 延续页面化思路，先表达业务场景，再暴露节点编排。

**必须包含**

1. 从页面上下文进入作业配置，而不是只从全局场景清单进入。
2. 用场景卡片表达常见类型：查询并回填、多字段辅助录入、预览确认后执行等。
3. 明确区分自动触发、预览确认、悬浮再次触发。
4. 节点编排、风险参数、底层细节进入高级区。

**当前页面改造基础**

- `JobScenesPage` 继续承载，但默认视图从“节点编排中心”改为“场景配置中心”。

### 5.6 API注册

**目标**

- 保留名称，但交互语言和流程改为业务人员可理解的操作路径。

**必须包含**

1. 列表突出接口用途、调用方式、状态、引用关系。
2. 新建流程改为五步：`选用途 -> 填基础信息 -> 填参数示例 -> 在线测试 -> 保存`。
3. 基础信息与高级配置分层展示。
4. 支持复制已有 API、从常用模板创建。
5. 保存后直接展示页面/规则/作业引用关系。

**当前页面改造基础**

- `InterfacesPage` 保留为一级入口，但内部交互改造成向导式框架。

### 5.7 发布与灰度

**目标**

- 让业务人员看懂这次变更的影响范围和生效方式，而不是暴露底层版本词汇。

**必须包含**

1. 四块视图：`待发布内容`、`生效范围`、`试点发布`、`发布记录`。
2. 发布步骤：选 `专区/菜单`、自动汇总待发布内容、选机构范围、选生效时间、确认影响范围。
3. 生效结果表达优先使用 `线上版本`、`试点版本`、`生效范围`。
4. 灰度范围仅开放到机构，不下钻到人员。

**当前页面改造基础**

- `GovernancePage` 承担发布流程与待办。
- `SdkVersionCenterPage` 的业务可见内容并入发布页；技术细节移到高级视图或管理员视角。

### 5.8 运行统计

**目标**

- 只保留业务有感知的统计和提醒，不把技术排障台直接暴露给业务。

**必须包含**

1. `触发统计`：总触发次数、页面排行、规则排行、机构分布、近 7/30 天趋势、发布前后对比。
2. `下降提醒`：页面、规则、机构范围、当前周期、对比周期、下降比例、首次发现时间。
3. `匹配失败` 不进入业务提醒，只保留在内部明细或高级视图。

**当前页面改造基础**

- `AuditMetricsPage` 由日志/指标中心收敛成业务统计页。

### 5.9 高级配置

**目标**

- 收纳低频复杂能力，并对不同角色做可见性控制。

**必须包含**

1. 预处理器。
2. 高级识别参数。
3. 接口高级配置。
4. 底层运行参数。
5. 权限治理与平台级配置。

**当前页面改造基础**

- `PreprocessorsPage`、`RolesPage` 先合并到 `高级配置` 入口。
- 后续新增平台参数分区时，不再新开业务侧一级菜单。

## 6. 横切体验改造

这些内容不是本轮主导航重构的核心，但必须同步纳入 UI 标准，避免新导航下仍保留旧体验问题。

### 6.1 表单安全性

1. 系统生成字段统一只读，不允许手工维护。
2. 关键操作统一二次确认。
3. Drawer/Modal 关闭前统一未保存提醒。
4. 状态切换、删除、发布、停用、回滚都使用统一确认交互。

### 6.2 统一流程壳

建议抽象以下通用能力：

| 组件/Hook | 用途 |
| --- | --- |
| `PageHeader` | 页面标题、说明文案、右侧动作 |
| `TaskEntryCard` | 工作台常用入口与待办卡片 |
| `WizardLayout` | 四步/五步向导外壳 |
| `ContextPanel` | 菜单详情聚合面板 |
| `ImpactSummary` | 发布影响确认摘要 |
| `ResultPreviewPanel` | 规则预览 / API测试 / 发布前对比 |
| `RoleGate` | 按角色控制菜单和模块可见性 |
| `useUnsavedWarning` | 统一未保存提醒 |

### 6.3 页面一致性

1. 统一页头结构：标题、说明、状态、最近变更、右侧主按钮。
2. 统一列表结构：筛选区、摘要区、表格/卡片区、详情抽屉。
3. 统一高级区表达：`高级条件`、`高级设置`、`更多配置` 三类命名择一，避免混用。
4. 统一结果反馈：成功、失败、警告、影响确认、草稿保存提示。

## 7. 前端实施路线图

### Phase 0：信息架构冻结

**目标**

- 完成导航、路由、页面承载和角色可见性设计冻结。

**任务**

1. 冻结主导航八项菜单及其文案。
2. 冻结新旧路由映射和兼容策略。
3. 冻结每个一级菜单的前端承载页面与二级结构。
4. 冻结角色可见性矩阵。

### Phase 1：工作台与菜单管理

**目标**

- 先打通“从哪里开始”和“围绕页面做事”的主路径。

**任务**

1. 改造 `AppShell` 导航。
2. 将 `DashboardPage` 改为 `我的工作台`。
3. 合并 `PageResourcesPage` 与 `PageActivationPoliciesPage` 的菜单管理能力。
4. 落地菜单详情结构和页面内快捷动作。

### Phase 2：高频流程向导化

**目标**

- 打通智能提示、智能作业、API注册三类高频流程。

**任务**

1. `RulesPage` 改为规则列表 + 模板复用入口 + 四步向导。
2. `JobScenesPage` 改为页面化入口 + 场景卡片 + 高级编排区。
3. `InterfacesPage` 改为五步向导 + 在线测试 + 引用关系回显。
4. 抽象通用 `WizardLayout` 和结果闭环组件。

### Phase 3：发布与灰度、运行统计

**目标**

- 打通业务发布闭环和业务统计闭环。

**任务**

1. 合并 `GovernancePage` 与 `SdkVersionCenterPage` 的业务侧承载。
2. 将发布流程改为影响可理解的业务流程。
3. 将 `AuditMetricsPage` 收敛为 `触发统计 + 下降提醒`。
4. 补齐发布前后对比和下降提醒卡片化展示。

### Phase 4：高级配置与角色治理

**目标**

- 让低频复杂能力后置并稳定收口。

**任务**

1. 合并 `PreprocessorsPage`、`RolesPage` 到 `高级配置`。
2. 接入 `RoleGate` 控制不同角色可见性。
3. 下线旧主导航中的低层技术菜单，仅保留兼容深链。
4. 统一高级项说明文案与风险提示。

### Phase 5：试点与收尾

**目标**

- 验证新信息架构是否真正提升业务效率。

**任务**

1. 按业务五件事做走查：找页面、配规则、接接口、做发布、看结果。
2. 记录试点用户问题并修正文案、入口和流程阻断点。
3. 收敛旧入口迁移提示和帮助说明。
4. 形成扩面所需的模板化页面与培训材料。

## 9. 验收标准

### 9.1 信息架构验收

1. 主导航已切换到八个业务菜单。
2. 旧技术菜单不再出现在主导航中。
3. 旧路由如保留，只作为兼容入口存在。

### 9.2 主路径验收

1. 用户可从工作台进入菜单管理并快速定位页面。
2. 用户可从菜单详情直接进入提示规则或作业配置。
3. 用户可在智能提示中通过模板快速完成规则新建并看到预览。
4. 用户可在 API注册中先测试再保存，并查看引用关系。
5. 用户可在发布与灰度中明确看到改动范围、影响页面数、影响机构数和生效时间。
6. 用户可在运行统计中通过趋势和下降提醒验证效果。

### 9.3 体验验收

1. 页面命名、文案、帮助说明与新 `plan.md` 完全一致。
2. 高频流程至少一屏内能看到主动作入口。
3. 复杂项默认折叠或后置，不干扰主路径。
4. 系统生成字段只读、关键动作有确认、关闭前有未保存提醒。

### 9.4 角色验收

1. 业务人员默认只看到业务主路径。
2. 高级运营能访问必要高级项，但不暴露全部平台级配置。
3. 平台管理员可见全部能力。

---

## 10. 风险与注意事项

1. 如果只改导航，不改页面内部结构，用户仍会在新菜单中看到旧体验，重构效果会打折。
2. 如果菜单管理没有真正聚合能力状态和运行结果，就无法成为业务主入口。
3. 如果模板中心、SDK 版本中心、治理页仍保持强曝光，会继续拉回技术对象型心智。
4. 如果高级配置收口过早但角色裁剪不到位，可能影响高级运营完成工作。
5. 如果运行统计继续混入技术失败明细，会破坏业务视角的一致性。

---

## 11. 建议落地顺序

1. 先改导航和首页。
2. 再改菜单管理聚合。
3. 再改智能提示、智能作业、API注册三个高频流程。
4. 再改发布与灰度、运行统计的结果闭环。
5. 最后做高级配置收口和旧入口迁移。

---

## 12. 结论

当前前端原型的核心问题不是“单个表单控件是否完善”，而是整体信息架构仍偏技术对象视角。  
本轮 UI 重构的第一目标，应是把原型从“能展示很多能力”改成“业务人员能顺着主路径完成任务”。

前端实现上，必须优先完成以下三件事：

1. 用业务菜单重构导航。
2. 用菜单管理承接主入口。
3. 用向导和结果闭环重构高频流程。

在这三件事完成后，再持续收敛高级配置和细节体验，才能和新版 `plan.md` 保持一致。
