# 配置中心“生效流程统一”产品方案（前台合并心智，后台保留语义）

> 版本：v1.0  
> 日期：2026-03-16  
> 目标：在不破坏权限与审计边界的前提下，让业务用户只理解一件事：`让配置生效`

## 1. 背景与目标

当前用户视角中，“发布”和“启用/停用”是两套操作语言，学习成本高、决策负担重。  
业务侧希望流程更简单，产品侧需要同时保证：

1. 门禁不丢失（最终校验、风险确认）。
2. 权限不混淆（发布与停用权限边界仍可控）。
3. 审计可追溯（能区分是“新版本生效”还是“已生效内容停用”）。

本方案采用：

1. 前台统一心智为“生效”。
2. 后台保持“发布/停用/回滚”等语义动作不变。
3. 系统根据对象状态自动路由到正确动作链路。

## 2. 核心方案

### 2.1 单入口模型

用户在对象页只看到一个主动作区：`生效操作`。

不同状态对应不同主按钮：

1. `DRAFT`：`立即生效`
2. `ACTIVE`：`暂停生效`
3. `DISABLED`：`恢复生效`
4. `EXPIRED`：`基于当前内容重新生效`（实际走“新草稿 -> 发布”）

### 2.2 自动动作路由

点击主按钮后，系统按状态自动执行底层动作：

1. `DRAFT -> ACTIVE`：走 `publish` 链路（含最终校验）。
2. `ACTIVE -> DISABLED`：走 `disable` 链路（快速止损）。
3. `DISABLED -> ACTIVE`：走“恢复生效”策略。
   - 默认策略：走 `publish` 门禁（更安全）。
   - 可选策略：直接 `enable`（仅低风险对象开启）。
4. `EXPIRED`：先生成 `DRAFT` 再 `publish`。

### 2.3 统一确认弹层（生效确认）

弹层固定三块信息：

1. 影响范围：对象、页面数、机构范围、生效时间。
2. 自动校验：阻断项、风险项、可修复建议。
3. 执行动作：当前会执行“发布/停用/恢复”的哪一种。

用户只确认“是否生效”，系统负责细节编排。

## 3. 权限与审计策略

### 3.1 权限策略（不合并）

权限仍沿用动作型控制：

1. `PUBLISH`：允许“立即生效/恢复生效（需发布）”
2. `DISABLE`：允许“暂停生效”
3. `VALIDATE`：允许查看门禁校验详情
4. `RISK_CONFIRM`：允许处理风险确认阻断

前台按钮是否可点，由“状态 + 动作权限”共同决定。

### 3.2 审计策略（增强）

前台统一文案，后台日志保留真实动作：

1. `EFFECTIVE_REQUEST`（统一入口行为）
2. `PUBLISH` 或 `DISABLE` 或 `ROLLBACK`（实际执行行为）
3. `VALIDATE`（门禁过程）

可同时满足“用户易懂”和“审计可追责”。

## 4. 交互文案规范

### 4.1 用户可见主文案

1. 页面主动作：`立即生效`、`暂停生效`、`恢复生效`
2. 统一提示：`系统会自动执行生效前检查`
3. 阻断反馈：`当前无法生效，请先处理以下问题`

### 4.2 禁用文案

1. 避免同时展示“发布”“启用”“上线”“开通”混用。
2. 避免要求用户理解“先校验再发布”的双页面跳转。

## 5. 与当前代码的落地映射

### 5.1 已具备基础

当前已有能力可复用：

1. `save*Draft` 产出待发布对象。
2. `publishPendingItem` 负责最终校验与生效。
3. 各页面已有“保存后提示条”承接发布。

核心文件：

1. `src/services/configCenterService.ts`
2. `src/pages/RulesPage/useRulesPageModel.tsx`
3. `src/pages/JobScenesPage/useJobScenesPageModel.ts`
4. `src/pages/InterfacesPage/useInterfacesPageModel.tsx`
5. `src/components/PublishContinuationAlert.tsx`

### 5.2 改造清单（按优先级）

P0（主流程统一）：

1. 新增统一 Hook：`useEffectiveAction`（输入对象类型+状态，输出主按钮动作、可执行性、确认文案）。
2. 在规则/作业/API 三页替换现有“发布当前对象”逻辑为统一 Hook。
3. 新增统一弹层组件：`EffectiveConfirmModal`，承载影响范围和校验结果。
4. 统一按钮文案：`立即生效/暂停生效/恢复生效`。

P1（扩展到高级对象）：

1. 预处理器、页面开通策略、版本灰度策略接入同一 `useEffectiveAction`。
2. 补齐“DISABLED -> ACTIVE”对象级策略配置（发布恢复或直接恢复）。

P2（治理与数据）：

1. 审计日志新增 `EFFECTIVE_REQUEST` 事件。
2. 统计看板新增“生效一次成功率、阻断率、平均生效耗时”。

## 6. 风险与规避

### 6.1 主要风险

1. 把“暂停生效”错误走成“发布”链路，导致应急变慢。
2. 把“恢复生效”错误放行，绕过最终门禁。
3. 前台统一后，后台日志丢失真实动作语义。

### 6.2 规避策略

1. 动作路由写成显式状态机，不用散落 if-else。
2. 恢复生效默认走发布门禁，敏感对象不允许直接恢复。
3. 统一入口只改文案，不改审计动作类型。

## 7. 验收标准

1. 用户在主流程页面只需要理解“生效”动作。
2. 用户不再跨页面执行“先校验再发布”。
3. 发布门禁、风险确认、权限边界不回退。
4. 日志中仍可区分发布与停用等底层动作。
5. 规则/作业/API 三页的生效操作交互一致。

## 8. 推荐实施顺序

1. 先做规则/作业/API 三页统一（高频路径）。
2. 再扩展到高级配置对象。
3. 最后补治理指标与审计增强。
