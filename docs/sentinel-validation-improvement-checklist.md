# 智能提示校验分层改进清单

## 1. 结论

基于当前产品与架构文档，智能提示相关能力应采用“三层校验”：

1. 前端实时校验：尽早发现表单级问题。
2. 保存校验：保证单对象结构完整、可保存、可回显。
3. 发布校验：作为最终生效门禁，负责跨对象一致性、治理完整性与风险确认。

不建议取消发布前校验，但建议弱化其“单独流程感”，改为在点击“发布”时自动触发。

## 2. 改进目标

1. 将可前移的问题尽量前移，减少发布阶段才暴露低级错误。
2. 保留发布前最终校验，防止跨规则、跨场景、跨版本问题直接上线。
3. 让用户感知为一条连续流程，而不是割裂的“配置完成后再走一次校验流程”。
4. 保证配置角色与治理角色职责边界不被破坏。

## 3. 分层原则

### 3.1 前端实时校验

适合前移到编辑态即时反馈的问题：

1. 必填项校验。
2. 值类型校验。
3. 基础格式校验。
4. 当前表单上下文中的引用合法性校验。
5. 轻量结构约束校验。

处理原则：

1. 以字段级、区域级提示为主，不阻断用户继续编辑。
2. 错误信息需要结构化、可定位、可执行。
3. 不依赖线上真实样本或外部联调数据。

### 3.2 保存校验

适合在“保存草稿”或“提交发布前预校验”执行的问题：

1. 单个规则对象结构是否完整。
2. 条件、提示、关联关系是否可序列化保存。
3. 保存结构是否支持完整回显。
4. 当前对象引用的页面资源、API、预处理器是否存在且状态可用。
5. 当前对象内是否存在明显互斥或脏配置残留。

处理原则：

1. 保存失败时应返回对象级错误清单。
2. 允许以草稿形式保留未达发布条件的对象，但不能写入非法结构。
3. 对“引用有效性”类规则执行首次强校验。

### 3.3 发布校验

必须保留在“正式生效前”执行的问题：

1. 同页同字段跨规则、跨场景写入冲突检查。
2. 覆盖策略确认与日志留痕配置检查。
3. 责任人、适用范围、生效时间、失效时间完整性检查。
4. 自动执行类场景的业务风险责任确认。
5. 作业场景人工作业时长等治理信息完整性检查。
6. 引用资源在发布瞬间是否仍然有效。
7. 待发布版本集合是否能形成一致版本视图。

处理原则：

1. 这是最终放行门禁，未通过不得生效。
2. 不要求用户先手工点“校验”按钮，但发布动作必须自动触发。
3. 对需要治理角色确认的事项，必须保留独立确认动作与审计记录。

## 4. 改进清单

| 编号 | 改进项 | 所属层级 | 建议动作 | 优先级 |
| --- | --- | --- | --- | --- |
| 1 | 将必填、类型、基础格式错误改为编辑态即时提示 | 前端实时校验 | 输入后即时校验，避免堆积到保存/发布阶段 | P0 |
| 2 | 将引用选择后的基础有效性检查前移 | 前端实时校验 | 选中 API / 预处理器 / 页面资源后立即提示失效或不存在 | P0 |
| 3 | 对条件链、提示配置、来源切换残留做局部结构校验 | 前端实时校验 | 在切换来源、切换值类型、删除节点时提示脏数据风险 | P0 |
| 4 | 保存接口返回结构化错误清单 | 保存校验 | 按字段、区域、对象三级返回错误，支持前端精确定位 | P0 |
| 5 | 保存前统一检查“可保存、可回显” | 保存校验 | 对条件、提示、关联作业场景做完整序列化校验 | P0 |
| 6 | 引用有效性改为“保存时检查 + 发布时复检” | 保存校验 / 发布校验 | 避免保存后到发布前资源状态变化导致脏上线 | P0 |
| 7 | 发布时自动触发最终校验，不再要求用户单独走校验页面 | 发布校验 | 保留接口和门禁，但弱化额外操作步骤 | P0 |
| 8 | 冲突校验只保留在发布前最终判断 | 发布校验 | 编辑态可提示风险，正式阻断放在发布时 | P0 |
| 9 | 增加覆盖策略确认与留痕校验 | 发布校验 | 冲突不默认阻断时，必须补齐确认与日志配置 | P0 |
| 10 | 将责任人、范围、有效期纳入统一发布门禁 | 发布校验 | 作为治理字段统一检查，不分散在多个页面口径里 | P0 |
| 11 | 自动执行场景增加风险确认缺失拦截 | 发布校验 | 发布时校验业务管理角色确认状态 | P0 |
| 12 | 人工作业时长缺失时禁止正式生效 | 发布校验 | 对作业场景类对象保留独立门禁 | P1 |
| 13 | 将“保存草稿”和“提交发布”统一放在同一主流程出口 | 流程体验 | 保持流程连续，但后端仍保留状态流转与发布门禁 | P1 |
| 14 | 在发布结果页展示“本次阻断原因 / 风险确认项 / 影响范围” | 流程体验 | 提升治理可读性与可执行性 | P1 |
| 15 | 审计日志区分“配置错误”“保存阻断”“发布阻断” | 审计治理 | 便于后续统计问题前移效果 | P1 |

## 5. 推荐交互

建议将用户侧流程收敛为：

1. 编辑配置。
2. 实时看到字段与结构问题。
3. 点击保存草稿，处理对象级错误。
4. 点击发布。
5. 系统自动执行发布前校验。
6. 若失败，返回结构化阻断项。
7. 若通过，进入发布任务并完成状态流转。

建议取消“必须先手工点一次校验，再点发布”的强制交互，但保留：

1. 发布前自动校验。
2. 风险确认动作。
3. 冲突确认动作。
4. 审计留痕。

## 6. 实施顺序建议

### P0

1. 梳理三层校验口径，形成统一规则表。
2. 前端补齐实时校验与结构化错误展示。
3. 保存接口补齐对象级错误清单。
4. 发布动作自动触发最终校验。
5. 发布门禁统一收口责任人、范围、有效期、冲突、风险确认、引用有效性。

### P1

1. 优化发布结果页与风险提示文案。
2. 完善审计分类与问题前移指标。
3. 增加“编辑态风险提示，但不阻断”的轻提醒能力。

## 7. 验收口径

完成改进后，至少应满足以下结果：

1. 用户在编辑态即可发现大部分低级配置错误。
2. 保存阶段可拦截非法结构，且错误可定位到具体区域。
3. 发布阶段只重点拦截跨对象一致性、治理完整性和风险问题。
4. 用户无需额外手工进入独立校验流程也能完成发布。
5. 所有正式生效对象都经过最终发布门禁与审计留痕。
