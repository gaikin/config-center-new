# 平台级 JSSDK 版本治理设计

> 日期：2026-03-18
> 适用范围：`config-center` 前端原型中的平台运行参数、SDK 版本中心、菜单管理联动

## 1. 背景

当前系统已存在两类相关承载：

1. `高级配置 -> 平台参数`：仍是占位状态，尚未承载实际平台级运行参数。
2. `SDK版本中心`：当前已支持菜单层级的版本灰度策略，但模型仍偏通用，尚未明确区分智能提示与智能作业两条版本线。

本次设计要解决的问题是：

1. 平台级需要维护两条独立 JSSDK 版本线。
2. 正式版本应全平台统一，不应下沉到菜单逐个维护。
3. 灰度应允许按部分菜单、部分机构进行能力级覆盖。
4. 菜单管理页需要能看到结果态，但不承担版本治理操作。

## 2. 目标

本次设计目标：

1. 建立平台级默认版本基线：
   - 智能提示 JSSDK 正式版本
   - 智能提示 JSSDK 默认灰度版本
   - 智能作业 JSSDK 正式版本
   - 智能作业 JSSDK 默认灰度版本
2. 建立菜单级灰度覆盖能力：
   - 智能提示按菜单 + 机构灰度
   - 智能作业按菜单 + 机构灰度
3. 明确运行时命中规则，避免出现多层优先级歧义。
4. 将页面职责固定：
   - 平台参数负责默认值
   - SDK版本中心负责菜单灰度
   - 菜单管理只负责结果态展示和跳转

## 3. 设计结论

### 3.1 总体原则

1. 智能提示 JSSDK 与智能作业 JSSDK 是两条独立版本线。
2. 正式版本为平台统一口径，不在菜单级覆盖。
3. 菜单级只允许灰度覆盖，不允许维护菜单级正式版本。
4. 灰度粒度为：
   - 部分菜单
   - 部分机构
   - 分能力（提示 / 作业）

### 3.2 页面职责

1. `高级配置 -> 平台参数`
   - 维护平台默认版本。
   - 不出现菜单、机构、生效时间窗。
2. `SDK版本中心`
   - 维护菜单灰度策略。
   - 支持分别配置智能提示灰度和智能作业灰度。
3. `菜单管理`
   - 只展示当前菜单的版本状态与是否存在灰度。
   - 通过快捷入口跳转到 `SDK版本中心`。

## 4. 数据模型

### 4.1 平台运行参数

建议新增平台级对象：`PlatformRuntimeConfig`

字段建议如下：

```ts
type PlatformRuntimeConfig = {
  promptStableVersion: string;
  promptGrayDefaultVersion?: string;
  jobStableVersion: string;
  jobGrayDefaultVersion?: string;
  updatedAt: string;
  updatedBy: string;
};
```

说明：

1. `promptStableVersion`：智能提示平台正式版本，必填。
2. `promptGrayDefaultVersion`：智能提示默认灰度版本，可选，仅作为新建菜单灰度时的默认带入值。
3. `jobStableVersion`：智能作业平台正式版本，必填。
4. `jobGrayDefaultVersion`：智能作业默认灰度版本，可选，仅作为新建菜单灰度时的默认带入值。

该对象不包含菜单、机构、生效时间窗。

### 4.2 菜单灰度策略

建议将当前 `MenuSdkPolicy` 收口为“菜单能力灰度策略”，面向菜单级覆盖。

字段建议如下：

```ts
type MenuSdkPolicy = {
  id: number;
  menuId: number;
  menuCode: string;

  promptGrayEnabled: boolean;
  promptGrayVersion?: string;
  promptGrayOrgIds: string[];

  jobGrayEnabled: boolean;
  jobGrayVersion?: string;
  jobGrayOrgIds: string[];

  effectiveStart: string;
  effectiveEnd: string;
  status: "DRAFT" | "ACTIVE" | "DISABLED" | "EXPIRED";
  ownerOrgId: string;
  updatedAt: string;
};
```

说明：

1. 该对象不再维护菜单级正式版本。
2. 提示和作业两组字段保持结构对称，便于页面配置和运行时处理。
3. 若某能力不开灰度：
   - `xxxGrayEnabled = false`
   - `xxxGrayVersion = 空`
   - `xxxGrayOrgIds = []`

## 5. 运行时决策规则

运行时命中规则固定为单一优先级链路：

1. 按能力分流。
   - 智能提示只看提示版本配置。
   - 智能作业只看作业版本配置。
2. 判断菜单级灰度是否命中。
   - 当前菜单存在该能力灰度配置。
   - 当前机构命中灰度机构范围。
   - 当前时间命中灰度时间窗。
3. 若命中灰度，则使用菜单灰度版本。
4. 若未命中灰度，则回退到平台正式版本。

可表达为：

```text
菜单灰度命中 -> 灰度版本
否则 -> 平台正式版本
```

### 5.1 默认灰度版本的定位

平台默认灰度版本不直接参与运行时命中判断。

它只承担以下作用：

1. 新建菜单灰度策略时的默认带入值。
2. 运营侧统一维护常用灰度版本，减少重复选择。

不建议将其纳入运行时优先级，否则会形成“平台灰度”和“菜单灰度”双层命中，增加解释成本与排障复杂度。

## 6. 页面交互设计

### 6.1 平台参数页

位置：`高级配置 -> 平台参数`

建议交互：

1. 使用单张平台配置卡片或简洁表单。
2. 分为两组：
   - 智能提示
   - 智能作业
3. 每组字段为：
   - 正式版本
   - 默认灰度版本

页面职责：

1. 展示和维护平台默认版本基线。
2. 不承载菜单灰度配置。

### 6.2 SDK版本中心

建议拆为三块：

1. 平台默认版本摘要
   - 只读展示当前 4 个平台默认版本。
   - 方便用户在配置菜单灰度时对照。
2. 菜单灰度策略列表
   - 菜单
   - 提示灰度版本 / 灰度机构
   - 作业灰度版本 / 灰度机构
   - 生效时间
   - 状态
   - 操作
3. 新建 / 编辑灰度策略弹窗
   - 同一弹窗中配置提示灰度和作业灰度。
   - 两组字段结构一致。

建议交互规则：

1. 打开“提示灰度”时，自动带入 `promptGrayDefaultVersion`。
2. 打开“作业灰度”时，自动带入 `jobGrayDefaultVersion`。
3. 若关闭某项灰度开关，则自动清空对应版本和机构数据。

### 6.3 菜单管理页

菜单管理页不承担版本配置操作，仅承载结果态展示：

1. 当前菜单提示能力状态。
2. 当前菜单作业能力状态。
3. 当前菜单是否存在提示灰度。
4. 当前菜单是否存在作业灰度。
5. 提供“去 SDK版本中心配置”的快捷入口。

## 7. 约束与校验

### 7.1 平台参数约束

1. `promptStableVersion` 必填。
2. `jobStableVersion` 必填。
3. 默认灰度版本可为空。

### 7.2 菜单灰度约束

对每个能力分别校验：

1. 若开启灰度，则灰度版本必填。
2. 若开启灰度，则灰度机构至少一个。
3. 灰度版本不能等于平台正式版本。
4. 灰度时间窗必须合法：`effectiveStart <= effectiveEnd`。
5. 菜单能力未开通时，不允许配置对应灰度能力。

### 7.3 策略冲突约束

建议同一菜单、同一能力、同一时间窗内只保留一条生效灰度策略。

这样可以避免：

1. 同一菜单同一能力存在多条重叠策略。
2. 运行时命中结果不确定。
3. 后续发布与排障无法解释。

## 8. 运行结果展示联动

为避免平台治理与业务页面脱节，建议增加以下结果态联动：

1. `菜单管理`
   - 展示当前菜单是否存在提示灰度 / 作业灰度。
2. `SDK版本中心`
   - 在列表中直观看到菜单灰度覆盖情况。
3. `高级配置 -> 平台参数`
   - 修改正式版本后，未命中灰度的菜单将默认走新正式版本。

## 9. 非目标

本次设计不包含以下范围：

1. JSSDK Loader / Manifest 协议改造细节。
2. 实际运行时接口字段设计。
3. JSSDK 本地缓存实现细节。
4. 后端发布流水线与制品仓库接入细节。
5. 多层灰度叠加策略（如平台灰度 + 菜单灰度同时命中）。

## 10. 验收标准

### 10.1 平台参数

1. 能正确保存并回显：
   - 智能提示正式版本
   - 智能提示默认灰度版本
   - 智能作业正式版本
   - 智能作业默认灰度版本

### 10.2 菜单灰度配置

1. 能为某菜单分别配置：
   - 提示灰度版本 + 灰度机构
   - 作业灰度版本 + 灰度机构
2. 能正确编辑、停用、回显。

### 10.3 运行时命中

应能验证以下 4 类场景：

1. 命中提示灰度 -> 走提示灰度版本。
2. 未命中提示灰度 -> 走提示正式版本。
3. 命中作业灰度 -> 走作业灰度版本。
4. 未命中作业灰度 -> 走作业正式版本。

### 10.4 约束阻断

以下场景应阻断保存：

1. 正式版本为空。
2. 灰度开启但未选版本。
3. 灰度开启但未选机构。
4. 灰度版本等于正式版本。
5. 时间窗非法。
6. 菜单能力未开通却配置对应灰度。

### 10.5 页面联动

1. `菜单管理` 能看到菜单当前是否存在提示 / 作业灰度。
2. `平台参数` 修改正式版本后，未命中灰度的菜单默认走新正式版本。
3. `SDK版本中心` 新建菜单灰度时，能自动带出平台默认灰度版本。

## 11. 推荐实施顺序

1. 增加平台运行参数模型与 mock/service 存储。
2. 调整 `SDK版本中心` 的列表与编辑模型，改为按能力配置灰度。
3. 在 `高级配置 -> 平台参数` 落地平台默认版本编辑。
4. 在 `菜单管理` 展示菜单结果态与跳转入口。
5. 最后补运行时命中模拟与联动校验。
